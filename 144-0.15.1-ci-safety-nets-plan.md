# Plan 144 — 0.15.1 CI safety nets

## 1. Summary

The 0.15.0 cycle had two embarrassing pre-publish near-misses
that all share a root cause: **CI didn't enforce what the project
actually expected**.

1. The `add_class_config` rename / stringly-typed
   `Connection::add_class("htb", &[..])` survived three earlier
   audits and was caught only by a fourth pass. **Mitigation**:
   `cargo-semver-checks` + `cargo public-api` baseline + diff
   would have flagged the rename + the asymmetric API in 0.15.0
   (or pre-cut, if run on a release candidate branch).

2. The `xfrm_ipsec_monitor` example was promoted to use
   `nlink::lab::with_namespace` but its `[[example]]` entry never
   got `required-features = ["lab"]`. CI runs
   `cargo test --verbose` from the workspace root, which compiles
   examples — so CI is currently red, but only because no PR was
   opened on top of broken master to surface it. **Mitigation**:
   make `cargo test --workspace` (default features) an explicit,
   named CI step that's hard to ignore, AND audit every existing
   example for similar gating issues.

This plan adds the safety nets. Pure CI + Cargo metadata work; no
library code changes.

## 2. Goals

By the end of this plan:

- CI runs `cargo test --workspace` (default features) as a
  named, fail-loud step. A future example with missing
  `required-features` cannot ship.
- CI runs `cargo-semver-checks` against the latest published
  release on every PR. A future accidental `pub fn` rename or
  trait-bound change cannot ship.
- CI runs `cargo public-api --diff` against the previous tag and
  fails on any unexpected delta. The diff is committed as a
  snapshot at release time so the next release's baseline is
  unambiguous.
- `cargo machete` runs without `|| true`. Either the two
  pre-existing entries (`bins/ss`, `bins/bridge`) are fixed in
  this plan, or they're moved to `[workspace.exclude]` with a
  written deferral explanation.
- Every existing example with `nlink::lab::*` or other
  feature-gated imports is audited and gated correctly.

## 3. Scope

### In

- `.github/workflows/rust.yml` rewrite (or split into multiple
  named jobs) to make each gate distinct.
- `crates/nlink/Cargo.toml` `[[example]]` audit (Phase 0 of
  Plan 143 covers `xfrm_ipsec_monitor`; this plan covers the
  others).
- Resolution of `bins/ss` and `bins/bridge` machete warnings
  (audit + fix, OR archive — see §5.4).
- One-time creation of the `cargo public-api` baseline file
  under `crates/nlink/public-api.txt` (or wherever the convention
  lands).

### Out (defer to 0.16.0+)

- MSRV declaration (`rust-version` in Cargo.toml). The moment
  it's declared, downstream pinning becomes load-bearing —
  better to declare in 0.16 with a written cadence policy. Plan
  144 only adds the *plumbing* (no `rust-version` field yet),
  so adding it later in 0.16 is one-line.
- Feature-permutation matrix (test 2^N feature combinations).
  Right call for 0.16; out of scope for a patch release.
- Kernel version matrix (test against multiple kernel
  versions). Right call eventually; out of scope.
- Distro matrix.
- `cargo-mutants` runs.

## 4. Approach

### 4.1 The CI workflow split

Today, `.github/workflows/rust.yml` is one job with two steps
(`cargo build`, `cargo test`). That's not enough granularity to
distinguish what failed.

Proposed split (named jobs, all parallel-able):

```yaml
jobs:
  build-and-test-default-features:
    # cargo test --workspace (default features) — catches example-gating regressions
  build-and-test-all-features:
    # cargo test --workspace --all-features — catches feature-flag-cfg regressions
  clippy:
    # cargo clippy --workspace --all-targets --all-features -- --deny warnings
  semver-checks:
    # cargo-semver-checks against latest published release on crates.io
  public-api-diff:
    # cargo public-api --diff vs. last tag; fail on unexpected delta
  machete:
    # cargo machete (NO || true)
  doc:
    # cargo doc -p nlink --no-deps; fail on warnings (depends on Plan 145 landing)
```

Six small jobs replace one monolithic job. Each is named so a
red CI line in the GitHub UI immediately tells you what broke.

### 4.2 Example-feature audit

Per-example check: "does this example import any item from a
feature-gated module without declaring `required-features`?"

Reproducible audit script:

```bash
# For every [[example]] entry in crates/nlink/Cargo.toml:
#   1. extract `path`
#   2. grep the source for `use nlink::(lab|sockdiag|tuntap|output|namespace_watcher)`
#   3. cross-reference against the entry's required-features
#   4. report mismatches
#
# One-shot tool (bash): scripts/audit-example-features.sh
# (commit alongside the workflow change so future contributors
#  can re-run it manually before opening a PR.)
```

Known mismatch from Plan 143's diagnosis: `xfrm_ipsec_monitor`
(fixed by Plan 143 Phase 0). Confirmed via grep at audit time
that the only other lab-using example is `lab/three_namespace.rs`,
which is correctly gated.

Future-proofing: the `cargo test --workspace` (default features)
CI job from §4.1 is the *enforcement* layer. The audit script is
the *diagnostic* layer for contributors.

### 4.3 cargo-semver-checks

```yaml
# Runs on every PR, against the latest published version on crates.io.
# Action handles the registry fetch + diff automatically.
- name: cargo semver-checks
  uses: obi1kenobi/cargo-semver-checks-action@v2
  with:
    package: nlink
```

**On the 0.15.0 → 0.15.1 candidate diff**: should be empty (patch
releases don't change public API). If the tool reports anything
non-trivial, it's a bug in either the tool or our patch.

**Initial run risk**: the tool may flag something in 0.15.0 that
was *already* published as a semver-major change retroactively
(e.g. a `non_exhaustive` enum we didn't think of, or a removed
public item). Treat the initial run as informational; if it
flags real issues in already-published 0.15.0, document and
move on (we can't un-publish).

### 4.4 cargo public-api snapshot + diff

```yaml
- name: Install cargo-public-api
  run: cargo install --locked cargo-public-api

- name: Public API diff vs last tag
  run: |
    LATEST_TAG=$(git describe --tags --abbrev=0)
    cargo public-api --diff-git-checkouts $LATEST_TAG HEAD --deny=all
```

Two outcomes:

- **Empty diff** (expected for 0.15.1): pass. Patch releases
  must not change the public API.
- **Non-empty diff**: fail. Either the change was intentional
  (and shouldn't be in a patch) or accidental. Either way, the
  PR doesn't merge until a maintainer adjudicates.

For 0.16+ minor releases the diff will be non-empty and that's
fine; the gate catches *unintended* changes, not all changes.
For 0.16, we'll either commit a snapshot to the repo (`cargo
public-api > crates/nlink/public-api.txt`) or rely on the
git-checkouts diff. **Pick one in 0.16's cycle; stub the snapshot
file in 0.15.1 with a one-line comment**.

### 4.5 machete enforcement (`bins/ss` + `bins/bridge` resolution)

Today, `.github/workflows/integration-tests.yml` line 73 (and
proposed `rust.yml` machete step) suppresses warnings with
`|| true` because of two pre-existing dead deps in
`bins/ss/Cargo.toml` (`atty`, `libc`) and
`bins/bridge/Cargo.toml` (`tokio-stream`, `tracing`,
`tracing-subscriber`).

**Two options to resolve, pick one in this plan:**

#### Option A — Clean up the deps (preferred if quick)

```bash
# bins/ss/Cargo.toml: remove `atty`, remove `libc` (verify nothing in src/ uses them)
# bins/bridge/Cargo.toml: remove tokio-stream, tracing, tracing-subscriber (verify)
```

Risk: low — these are POC bins with low traffic. If a `use`
statement breaks, the build catches it immediately.

Effort: 30 minutes if the deps really are dead. Worth doing.

#### Option B — Move bins to `archive/` directory

```toml
# Cargo.toml (workspace)
members = [
  "crates/nlink",
  "bins/ip", "bins/tc", "bins/wg", "bins/diag", "bins/config",
  "bins/ethtool", "bins/devlink", "bins/nft", "bins/wifi",
  # Removed: bins/ss, bins/bridge — moved to archive/, see archive/README.md
]
exclude = [
  "archive/ss",
  "archive/bridge",
]
```

Risk: cosmetic only — these bins aren't published anyway.

Effort: 30 minutes (mv + Cargo.toml edit + archive/README.md).

**Recommendation**: try Option A first. If `cargo build -p
nlink-ss` still passes after dep removal, Option A wins. If
removal breaks the bin, fall back to Option B (archive).

Either way, drop `|| true` from the machete CI step.

## 5. Implementation plan

### Phase 1: example-feature audit (writes the script + uses it)

1. Add `scripts/audit-example-features.sh` (~50 LOC bash).
2. Run it; verify only `xfrm_ipsec_monitor` was the problem
   (Plan 143 Phase 0 already fixed it).
3. Commit the script. Reference from `CONTRIBUTING.md` (or
   create one) as a pre-PR check.

### Phase 2: rust.yml CI workflow split

4. Rewrite `.github/workflows/rust.yml` per §4.1 — six small
   named jobs.
5. Verify each job runs correctly on a test PR before merging.

### Phase 3: semver-checks + public-api wiring

6. Add `cargo-semver-checks` GHA step.
7. Add `cargo public-api` GHA step (install + diff).
8. Run on the 0.15.0 baseline to seed expectations; document any
   flagged issues in PR description.

### Phase 4: machete resolution

9. Try Option A (clean up `bins/ss` + `bins/bridge` deps).
10. If that succeeds, drop `|| true` from machete step in both
    `rust.yml` (new step) and `integration-tests.yml`.
11. If Option A fails, switch to Option B (archive).

### Phase 5: ship

12. Single PR or 4 small PRs (one per phase) — maintainer's
    call. Lower-risk to do as 4 PRs since each is independently
    revertable.

## 6. Acceptance criteria

- [ ] `.github/workflows/rust.yml` has at least 6 named jobs
      covering: default-features build/test, all-features
      build/test, clippy, semver-checks, public-api-diff,
      machete.
- [ ] `cargo machete` step does **not** end with `|| true`
      anywhere in the repo.
- [ ] `cargo-semver-checks` passes on a 0.15.1 candidate (empty
      diff against published 0.15.0).
- [ ] `cargo public-api --diff` is empty between the tag for
      0.15.0 and the tip for 0.15.1.
- [ ] `scripts/audit-example-features.sh` exists, executable,
      reports zero mismatches.
- [ ] All `[[example]]` entries that import `nlink::lab` (or any
      other feature-gated module) declare the corresponding
      `required-features`.
- [ ] CI on master after merge is green.

## 8. Risks

- **Risk**: `cargo public-api` install adds 1-2 minutes to CI
  on every PR. **Mitigation**: cache the install with
  `actions/cache` keyed on a hash of `~/.cargo/bin/` — first run
  takes 2 min, subsequent runs use the cache.
- **Risk**: `cargo-semver-checks` produces false positives on
  edge-case generic-bound changes. **Mitigation**: the tool
  supports per-lint allow/deny config; document any waivers in a
  `cargo-semver-checks.toml` file at repo root with a comment
  explaining each waiver.
- **Risk**: Splitting CI into 6 jobs increases GHA spend.
  **Mitigation**: each job is fast (\<2 min); GHA's free tier
  for public repos is generous; no real risk.
- **Risk**: Option A (clean up deps) reveals that `bins/ss` or
  `bins/bridge` actually depend on those crates transitively in
  some build mode. **Mitigation**: fallback to Option B
  (archive); reverts the metadata change cleanly.
