### 4.1 `wait_link_up(iface, timeout)` helper

#### Design

```rust
impl Connection<Route> {
    /// Wait for an interface to reach the `IFF_UP` state.
    ///
    /// Polls the interface state with exponential backoff (starting
    /// at 10ms, capped at 100ms) until `IFF_UP` is observed or
    /// `timeout` elapses.
    ///
    /// Returns `Error::Timeout` on deadline. Returns
    /// `Error::InterfaceNotFound` if the interface is removed during
    /// the wait.
    ///
    /// # Namespace safety
    ///
    /// Takes `impl Into<InterfaceRef>`. With a `Name` variant, the
    /// initial lookup reads `/sys/class/net/` from the calling
    /// process's mount namespace (caveat: not necessarily the
    /// connection's netns). Pre-resolve with [`get_link_by_name`]
    /// and pass `Index(idx)` for namespace-correct behavior.
    pub async fn wait_link_up(
        &self,
        iface: impl Into<InterfaceRef>,
        timeout: Duration,
    ) -> Result<()> {
        let ifindex = self.resolve_interface(&iface.into()).await?;
        let deadline = std::time::Instant::now() + timeout;
        let mut backoff = Duration::from_millis(10);
        loop {
            let link = self
                .get_link_by_index(ifindex)
                .await?
                .ok_or(Error::InterfaceNotFound { interface: ... })?;
            if link.flags() & IFF_UP != 0 {
                return Ok(());
            }
            if std::time::Instant::now() >= deadline {
                return Err(Error::Timeout);
            }
            tokio::time::sleep(backoff).await;
            backoff = (backoff * 2).min(Duration::from_millis(100));
        }
    }
}
```

Polling rather than subscribe-based for simplicity. The exponential
backoff caps quickly so we don't burn cycles on a slow interface.

Subscribe-based would be more efficient under load but adds
multicast-socket lifetime concerns; the polling version is good
enough for the 80% case (tests + container init).

#### Tests

In `tests/integration/link.rs`:

```rust
#[tokio::test]
async fn wait_link_up_returns_when_link_comes_up() -> Result<()> {
    require_root!();
    nlink::require_module!("dummy");
    let (_ns, conn, ifindex) = setup_dummy("waitup").await?;
    // Set down explicitly.
    conn.set_link_down_by_index(ifindex).await?;
    // Spawn a task that brings it up after 50ms.
    let conn2 = conn.clone(); // or open second connection
    tokio::spawn(async move {
        tokio::time::sleep(Duration::from_millis(50)).await;
        let _ = conn2.set_link_up_by_index(ifindex).await;
    });
    // Wait with generous timeout.
    conn.wait_link_up(ifindex, Duration::from_secs(2)).await?;
    Ok(())
}

#[tokio::test]
async fn wait_link_up_times_out_when_link_stays_down() -> Result<()> {
    require_root!();
    nlink::require_module!("dummy");
    let (_ns, conn, ifindex) = setup_dummy("waitdown").await?;
    conn.set_link_down_by_index(ifindex).await?;
    let err = conn
        .wait_link_up(ifindex, Duration::from_millis(100))
        .await
        .unwrap_err();
    assert!(err.is_timeout(), "expected timeout, got {err}");
    Ok(())
}
```

Note: `Connection<P>` isn't `Clone` today. The test may need to open
two connections in the same namespace via `namespace::connection_for`
rather than cloning.

### 4.2 `get_link_stats(iface)` helper

#### Design

`LinkMessage` already carries an `rtnl_link_stats64` blob; we just
expose it cleanly:

```rust
/// Convenience: get the kernel-reported per-link statistics for the
/// named or indexed interface.
///
/// Returns `Err(Error::InterfaceNotFound)` if no interface matches.
pub async fn get_link_stats(
    &self,
    iface: impl Into<InterfaceRef>,
) -> Result<LinkStats> {
    let link = self
        .get_link_by_index(self.resolve_interface(&iface.into()).await?)
        .await?
        .ok_or(Error::InterfaceNotFound { ... })?;
    link.stats().cloned().ok_or_else(|| Error::InvalidMessage(
        "interface message did not include stats attribute".into()
    ))
}
```

Where `LinkStats` is the existing typed struct from `link.rs`. Confirm
the path — there's `LinkMessage::stats()` (or similar) already, just
unwrapping into a Result.

#### Test

```rust
#[tokio::test]
async fn get_link_stats_returns_zero_on_unused_dummy() -> Result<()> {
    require_root!();
    nlink::require_module!("dummy");
    let (_ns, conn, _ifindex) = setup_dummy("statszero").await?;
    let stats = conn.get_link_stats("dummy0").await?;
    assert_eq!(stats.rx_bytes, 0);
    assert_eq!(stats.tx_bytes, 0);
    Ok(())
}
```

### 4.4 Namespace-safety docs on `*_by_index` / `*_by_name`

#### Diagnosis

CLAUDE.md says (in CC's previous review): "for namespace-aware code,
prefer `*_by_index` Connection methods over `*_by_name` — the name
variants read `/sys/class/net/` from the host namespace, which is
wrong inside a foreign netns."

The actual method docstrings don't repeat this. For CNI plugins and
similar multi-namespace consumers, this is a footgun.

#### Fix

Add a copy-paste-ready doc block to every `*_by_index` method:

```rust
/// # Namespace safety
///
/// Takes a kernel-assigned interface index directly. The index is
/// always valid in the namespace the connection is bound to, so this
/// method is safe to call from any process mount-namespace. Compare
/// with [`get_link_by_name`] / [`set_link_mtu_by_name`] / etc. which
/// read `/sys/class/net/` from the calling process's mount namespace
/// to resolve names; those variants surprise inside foreign netns.
```

And the inverse for every `*_by_name` method:

```rust
/// # Namespace safety
///
/// Resolves `iface` by reading `/sys/class/net/` from the calling
/// process's mount namespace, which is typically the host even when
/// the connection is bound to a different netns. For namespace-correct
/// code, pre-resolve via [`get_link_by_index`] or use the
/// `*_by_index` variant of this method.
```

Targets: every `*_by_index` / `*_by_name` method in `connection.rs`
and the per-subsystem facades. Count: probably ~20 methods total
(rough count from `grep "_by_index\|_by_name" -c`).

#### Effort

1 hour mechanical paste-and-adapt. Do it in one PR; mostly diff noise
but high-value for downstream consumers.

### 4.5 Sealed `Connection::<P>::new()` for GENL types

#### Diagnosis

`Connection::<Wireguard>::new()` compiles but returns a connection
with family ID 0; the first operation fails with a confusing error.
Real fix: sealed marker trait that excludes GENL families from
sync construction.

#### Design

```rust
// In crates/nlink/src/netlink/protocol.rs or connection.rs
mod sealed {
    pub trait SyncConstructible {}
    pub trait AsyncConstructible {}
}

// Implementations for non-GENL families:
impl sealed::SyncConstructible for Route {}
impl sealed::SyncConstructible for Netfilter {}
impl sealed::SyncConstructible for Xfrm {}
// ... and all the other non-GENL protocols

// GENL families:
impl sealed::AsyncConstructible for Wireguard {}
impl sealed::AsyncConstructible for Macsec {}
impl sealed::AsyncConstructible for Mptcp {}
impl sealed::AsyncConstructible for Ethtool {}
impl sealed::AsyncConstructible for Nl80211 {}
impl sealed::AsyncConstructible for Devlink {}

// The constructor:
impl<P: ProtocolState + sealed::SyncConstructible> Connection<P> {
    pub fn new() -> Result<Self> { ... }
}

impl<P: ProtocolState + sealed::AsyncConstructible> Connection<P> {
    pub async fn new_async() -> Result<Self> { ... }
}
```

Now `Connection::<Wireguard>::new()` fails to compile with an
error pointing at the missing trait bound. The compiler error is
better than a runtime trap.

#### Public-API impact

This is shaped like a breaking change — anyone today writing
`Connection::<Wireguard>::new()` will get a compile error. But:

1. The runtime behavior was always broken (family ID 0 → confused
   error on first call). Anyone relying on it had broken code.
2. Per `cargo-semver-checks`, adding a trait bound to an existing
   method is technically a breaking change. So this needs to ship as
   part of a minor version cut (0.16 is fine), not a patch.

CHANGELOG note this prominently: "compile-time prevention of a class
of bug that previously failed silently at runtime."

#### Downstream consumers in 0.16

Two later plans depend on the sealed traits introduced here:

- **Plan 154** (`nlink-macros`): the `#[genl_family]` attribute
  macro generates `impl __sealed::AsyncConstructible for MyFamily {}`
  so user-defined GENL families inherit the same constructor
  discipline as the in-tree ones.
- **Plan 159** (`ConnectionPool<P>`): `ConnectionPoolBuilder<P>::build`
  is bounded `where P: SyncConstructible` for the sync overload
  and `P: AsyncConstructible` for the async overload — same split
  as `Connection::<P>::new()` / `new_async()`.

If this plan slips, both 154 and 159 can ship without the
sealed-trait impls and gain them in a 0.16 patch. The user-visible
behavior stays correct either way; the only loss is the compile-time
error for `Connection::<Wireguard>::new()`.

#### Tests

Add a `trybuild`-style test if the project uses one (it doesn't today
— skip). Instead, verify by manual repro: write a snippet in
`crates/nlink/examples/genl/wireguard.rs` that called `new()`, change
it to `new_async().await?`, and confirm it compiles.

#### Effort

2 hours.

#### Files

- `crates/nlink/src/netlink/protocol.rs` (or connection.rs) — sealed
  trait + impls
- `crates/nlink/src/netlink/connection.rs` — bound `new()` /
  `new_async()`
- All affected examples — already use `new_async()` for GENL, verify

### 4.8 nftables `NFT_TABLE_F_PERSIST`

#### Diagnosis

Kernel 6.9 added `NFT_TABLE_F_PERSIST` — a flag bit on the table
that causes it to survive ruleset flushes. Useful for "the table
itself should outlive a `nft flush ruleset`". Currently nlink's
nftables `Table` builder doesn't expose this.

#### Fix

```rust
// crates/nlink/src/netlink/nftables/types.rs
impl Table {
    /// Mark the table as **persistent** — it survives `nft flush
    /// ruleset` operations issued against the same family. Kernel
    /// 6.9+.
    ///
    /// Internally sets `NFT_TABLE_F_PERSIST` (= 0x4) on `NFTA_TABLE_FLAGS`.
    pub fn persist(mut self, on: bool) -> Self {
        if on {
            self.flags |= NFT_TABLE_F_PERSIST;
        } else {
            self.flags &= !NFT_TABLE_F_PERSIST;
        }
        self
    }
}

pub const NFT_TABLE_F_PERSIST: u32 = 0x4;
```

#### Test

A unit test in the nftables module verifying the flag bit gets set
correctly in the serialized message.

#### Effort

1 hour including the unit test.

## 8. Risks

- **§4.1 `wait_link_up` interface-deletion race**: if the interface
  is deleted mid-wait, our `get_link_by_index` returns `Ok(None)` and
  we error with `InterfaceNotFound`. Is that the right behavior, or
  should we report a separate error to distinguish "removed during
  wait" from "never existed"? Default to InterfaceNotFound — matches
  the rest of the API.
- **§4.4 doc audit completeness**: easy to miss methods. Mitigation:
  grep for all `_by_index` / `_by_name` first, generate a checklist,
  tick off as you go. Ship "good enough" — even doc on half the
  methods improves over the current zero-doc state.
- **§4.5 GENL constructor breakage**: if anyone in tree is calling
  `Connection::<Wireguard>::new()` without realizing it doesn't work,
  the change will surface as a compile error. Mitigation: grep first
  (`grep -rn 'Connection::<\(Wireguard\|Macsec\|Mptcp\|Ethtool\|Nl80211\|Devlink\)>::new\b'`)
  and verify all callsites use `new_async`.
- **§4.8 `NFT_TABLE_F_PERSIST` kernel ABI verification**: ensure
  the constant value (0x4) matches kernel headers (`include/uapi/linux/netfilter/nf_tables.h`).
  Don't guess; verify.
