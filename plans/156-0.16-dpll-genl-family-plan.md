# Plan 156 — DPLL netlink family for 0.16

The DPLL (Digital Phase-Locked Loop) generic-netlink family lets
userspace introspect and configure clock-synchronization hardware:
SyncE, PTP, GNSS-disciplined oscillators. Telco/RAN, time-sync
infrastructure, SmartNIC users (Intel `ice` / Mellanox `mlx5` /
NVIDIA BlueField) consume it daily.

DPLL stabilized in **kernel 6.7** (Nov 2023). The wire format is
small, frozen, and additions through 6.8–6.12 are strictly
appended attributes — perfect for a clean Rust binding.

This plan **builds DPLL using `nlink-macros`** (Plan 154). It is
the first non-trivial in-tree user of the proc-macro derives —
the existence proof that the macros work end-to-end on a real
modern kernel feature, not just on the synthetic
`define_taskstats.rs` showcase.

## 1. Summary

Three deliverables:

1. New `Dpll` protocol marker + GENL family resolution.
2. Typed command/attribute enums + message structs, **all derived
   via `nlink-macros`**. Lines of declaration target: ~80.
3. `Connection<Dpll>` methods for the canonical device/pin
   workflow + multicast monitor subscription + recipe.

The plan **must land after Plan 154** (the macros) since it consumes
them. Master plan §7 already sequences them in that order.

## 2. Goals

By end of plan:

- `Connection::<Dpll>::new_async()` resolves the `"dpll"` GENL
  family ID and is ready to use.
- A telco operator can list devices, list pins per device, change
  pin priority/state/frequency, and subscribe to change
  notifications — all through typed nlink APIs.
- The DPLL family definition fits in **≤120 lines** of nlink-macros
  declaration (commands + attribute enums + message structs +
  family marker). Validates the Plan 154 line-count success criterion
  on a real subsystem.
- Recipe `docs/recipes/dpll-monitor.md` shows the canonical
  "watch lock-status transitions" pattern.
- Example `crates/nlink/examples/genl/dpll.rs` dumps every DPLL
  device + its pins on the host.

## 3. Scope

### In

- `crates/nlink/src/netlink/genl/dpll/{mod,connection,types}.rs`
  module.
- Command enum (12 commands), attribute enums (14 device + 31 pin),
  value enums (mode, lock-status, type, pin-type, pin-direction,
  pin-state, pin-capabilities, lock-status-error,
  clock-quality-level, feature-state).
- Device-side message structs: `DpllDeviceGetRequest`,
  `DpllDeviceReply`, `DpllDeviceSetRequest`.
- Pin-side message structs: `DpllPinGetRequest`, `DpllPinReply`,
  `DpllPinSetRequest`.
- `Connection<Dpll>` helper methods: `get_device(id)`,
  `dump_devices()`, `set_device_mode(...)`, `get_pin(id)`,
  `dump_pins()`, `set_pin_priority(...)`, `set_pin_state(...)`,
  `set_pin_frequency(...)`.
- Multicast subscription on the `"monitor"` group; `events()`
  returns `Stream<Item = Result<DpllEvent>>` with typed variants
  per `*_NTF` command.
- Integration test gated on `require_root!` +
  `require_module!("dpll")` (loadable on some kernels;
  built-in elsewhere). All tests skip cleanly on hosts with no
  DPLL hardware.
- Recipe + runnable example.
- CHANGELOG entry.

### Out

- ESYNC-specific helpers (`esync-frequency`, `esync-pulse`) —
  modeled as `Option<…>` on the reply struct; no separate
  high-level helper. Demand-gated.
- `reference-sync` / `phase-adjust-gran` /
  `fractional-frequency-offset-ppt` / `measured-frequency` —
  same: present as `Option<…>` fields on `DpllPinReply` (kernel
  6.11+), not given dedicated setters.
- Linuxptp-style application logic (`ts2phc` orchestration). nlink
  exposes the kernel surface; userspace coordination is downstream.
- DPLL device *creation*. The kernel-side `*_CREATE_NTF` is a
  notification, not a userspace command — drivers register their
  DPLL devices.

The pin side adds another ~50. Total: ~130 lines for the full family.
Comparable to neli's `nl80211.rs` reference (~80 lines but smaller

`#[genl_enum]` is a new derive **added to Plan 154's scope** —
strictly weaker than `GenlAttribute`, used for value enums that
appear *inside* an attribute payload rather than as the attribute
kind itself. Plan 154 task list amended in §7 below.

### 4.3 Connection methods

```rust
// crates/nlink/src/netlink/genl/dpll/connection.rs

impl Connection<Dpll> {
    pub async fn get_device(&self, id: u32) -> Result<DpllDeviceReply> {
        self.send_typed(DpllDeviceGetRequest { id }).await
    }

    pub fn dump_devices(&self)
        -> impl Stream<Item = Result<DpllDeviceReply>> + Send + '_
    {
        self.dump_typed_stream(DpllDeviceGetRequest::dump())
    }

    pub async fn set_device_mode(&self, id: u32, mode: DpllMode) -> Result<()> {
        self.send_typed(DpllDeviceSetRequest::new(id).mode(mode)).await
    }

    pub async fn get_pin(&self, id: u32) -> Result<DpllPinReply> {
        self.send_typed(DpllPinGetRequest { id }).await
    }

    pub fn dump_pins(&self)
        -> impl Stream<Item = Result<DpllPinReply>> + Send + '_
    {
        self.dump_typed_stream(DpllPinGetRequest::dump())
    }

    pub async fn set_pin_priority(&self, id: u32, priority: u32) -> Result<()> {
        self.send_typed(DpllPinSetRequest::new(id).prio(priority)).await
    }

    pub async fn set_pin_state(&self, id: u32, state: DpllPinState) -> Result<()> {
        self.send_typed(DpllPinSetRequest::new(id).state(state)).await
    }

    pub async fn set_pin_frequency(&self, id: u32, hz: u64) -> Result<()> {
        self.send_typed(DpllPinSetRequest::new(id).frequency(hz)).await
    }

    /// Subscribe to DPLL monitor multicast (DPLL_MCGRP_MONITOR).
    /// Yields typed events for device/pin create/delete/change.
    pub async fn subscribe_monitor(&mut self) -> Result<()> {
        self.subscribe_group("monitor").await
    }

    pub fn events(&mut self)
        -> impl Stream<Item = Result<DpllEvent>> + Send + '_
    {
        self.event_stream()
    }
}

#[derive(Debug, Clone)]
#[non_exhaustive]
pub enum DpllEvent {
    DeviceCreated(DpllDeviceReply),
    DeviceDeleted { id: u32 },
    DeviceChanged(DpllDeviceReply),
    PinCreated(DpllPinReply),
    PinDeleted { id: u32 },
    PinChanged(DpllPinReply),
}
```

Every method takes typed inputs and returns typed outputs. No
`&[&str]` params; no `u16` raw attribute IDs leaking to the user.

### 4.4 Typed-units integration

Frequencies are `u64` Hz on the wire per the kernel UAPI. DPLL
does not use nlink's `Rate` type (which is bytes/sec for traffic
shaping) — that would be the wrong abstraction. Follow CLAUDE.md's
typed-units convention (`Rate`, `Bytes`, `Percent`) and introduce
a **newtype** `Hz` rather than a bare type alias. Newtype gives:

- Compile-time prevention of mixing Hz with byte-counts, durations,
  or other u64s.
- Unit-bearing constructors (`Hz::mhz(10)`) instead of magic
  literals (`10_000_000`).
- A natural place to add ordering / arithmetic impls if downstream
  use cases surface them.

```rust
/// DPLL pin frequency in Hz. Newtype to prevent accidental
/// mixing with other u64-typed quantities, matching the project's
/// `Rate` / `Bytes` / `Percent` typed-units convention.
#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, Hash)]
pub struct Hz(u64);

impl Hz {
    pub const fn new(hz: u64) -> Self { Self(hz) }
    pub const fn khz(khz: u64) -> Self { Self(khz.saturating_mul(1_000)) }
    pub const fn mhz(mhz: u64) -> Self { Self(mhz.saturating_mul(1_000_000)) }
    pub const fn ghz(ghz: u64) -> Self { Self(ghz.saturating_mul(1_000_000_000)) }
    pub const fn as_u64(self) -> u64 { self.0 }
}

impl std::fmt::Display for Hz {
    fn fmt(&self, f: &mut std::fmt::Formatter<'_>) -> std::fmt::Result {
        // Pretty-print at the natural unit (10000000 → "10 MHz").
        match self.0 {
            n if n >= 1_000_000_000 && n % 1_000_000_000 == 0 =>
                write!(f, "{} GHz", n / 1_000_000_000),
            n if n >= 1_000_000 && n % 1_000_000 == 0 =>
                write!(f, "{} MHz", n / 1_000_000),
            n if n >= 1_000 && n % 1_000 == 0 =>
                write!(f, "{} kHz", n / 1_000),
            n => write!(f, "{} Hz", n),
        }
    }
}

pub mod common_freq {
    use super::Hz;
    pub const ONE_HZ: Hz             = Hz::new(1);
    pub const TEN_KHZ: Hz            = Hz::khz(10);
    pub const SEVENTY_SEVEN_KHZ: Hz  = Hz::new(77_500);  // EBU 77.5 kHz
    pub const TEN_MHZ: Hz            = Hz::mhz(10);
}
```

`nlink-macros` (Plan 154 §4.3) treats `Hz` as a newtype wrapping
`u64` and emits via the u64 helper (auto-derive `From<Hz> for u64`
+ `From<u64> for Hz` via `#[genl_attr_transparent(u64)]` on the
field, or simply teach the macros to look through `#[repr(transparent)]`
single-field tuple structs — pick whichever proves cleaner during
Plan 154's implementation).

### 4.5 Scaling helpers

Three kernel-side dividers must be applied client-side per the
UAPI (research §4 — `DPLL_TEMP_DIVIDER`,
`DPLL_PHASE_OFFSET_DIVIDER`,
`DPLL_PIN_MEASURED_FREQUENCY_DIVIDER`, all = 1000). Expose them
as named consts + as `*_celsius()` / `*_ns()` / `*_hz()` helpers
on the reply structs:

```rust
impl DpllDeviceReply {
    /// Device temperature in degrees Celsius, if reported.
    pub fn temp_celsius(&self) -> Option<f32> {
        self.temp_mdeg.map(|m| m as f32 / 1000.0)
    }
}

impl DpllPinReply {
    /// Pin phase offset in nanoseconds, if reported. Kernel reports
    /// in attoseconds × 1000; divide by `DPLL_PHASE_OFFSET_DIVIDER`.
    pub fn phase_offset_ns(&self) -> Option<i64> {
        self.phase_offset.map(|p| p / DPLL_PHASE_OFFSET_DIVIDER)
    }

    /// Measured pin frequency in Hz, if reported.
    pub fn measured_frequency_hz(&self) -> Option<u64> {
        self.measured_frequency_mhz.map(|m| m / DPLL_PIN_MEASURED_FREQUENCY_DIVIDER)
    }
}

pub const DPLL_TEMP_DIVIDER: i32 = 1000;
pub const DPLL_PHASE_OFFSET_DIVIDER: i64 = 1000;
pub const DPLL_PIN_MEASURED_FREQUENCY_DIVIDER: u64 = 1000;
```

The raw fields (`temp_mdeg`, `phase_offset`) remain accessible for
users who want the unscaled wire value (e.g., for high-resolution
calculations).

## 7b. Macro dependencies on Plan 154

This plan consumes two Plan 154 derives in particular:

- **`#[derive(GenlEnum)]`** — for the value enums encoded inside
  attribute payloads (`DpllMode`, `DpllLockStatus`, `DpllType`,
  `DpllPinType`, `DpllPinDirection`, `DpllPinState`,
  `DpllLockStatusError`, `DpllClockQualityLevel`, the 0-based
  `DpllFeatureState`). Plan 154 §1 derives table covers this;
  effort was already folded into Plan 154's 7.5-day estimate.
- **Transparent newtype handling** — `Hz` is a single-field tuple
  struct wrapping `u64`. Plan 154 §4.3 supported-types table
  needs to either explicitly enumerate "`#[repr(transparent)]`
  newtype around a supported primitive" or accept a
  `#[genl_attr_transparent]` field attribute. Pick during Plan
  154 implementation; both shapes serve the same goal.

If Plan 154 ships without transparent-newtype support, this plan
falls back to `Hz` being a type alias for `u64` (single-line
fallback noted in §4.4). The newtype is preferred for type safety
but isn't a hard blocker.

## 8. Risks

- **Hardware-test scarcity**: as with Plan 153's
  XFRM-offload/devlink-rate testing, CI does not have DPLL
  hardware. All wire-format correctness is unit-tested; the
  integration test is permissive (empty-dump-is-success). Real
  validation is manual on dev hardware before the release cut.
- **Kernel 6.11+ attribute optionality**: `esync-frequency`,
  `reference-sync`, `phase-adjust-gran`, `phase-offset-monitor`,
  `frequency-monitor`, `measured-frequency`,
  `fractional-frequency-offset-ppt` were added across 6.8–6.12.
  All modeled as `Option<…>`; absence on older kernels is normal
  parse outcome, not error. Document per-attribute the minimum
  kernel version in the rustdoc.
- **`sint` encoding for FFO attributes**: YNL's variable-length
  signed integer. `nlink-macros` runtime must support this type;
  add to Plan 154's supported-types table. Internal `read_sint`
  helper needed (see kernel `lib/nlattr.c` `nla_get_sint`).
- **GENL family ID resolution**: `Connection::<Dpll>::new_async()`
  returns `FamilyNotFound` cleanly on kernels < 6.7 or without
  `CONFIG_DPLL=y`. Connection construction is the right failure
  point — don't try to soft-degrade.
- **Macro design fallout**: this is the first time
  `nlink-macros` is exercised on a real family with nested
  attributes (`pin-parent-device`, `pin-parent-pin`,
  `frequency-range`, `reference-sync`). Expect 1–2 design
  iterations on the `#[genl_attr(...)]` syntax for nested groups.
  Mitigation: develop Plan 156 in parallel with Plan 154 Phase 6
  (the example) — DPLL is a more demanding showcase than
  `define_taskstats.rs` and a better design driver.

- **High-level "SyncE failover orchestrator"** — application
  logic on top of `Connection<Dpll>`. Downstream project.
