# Plan 151 — ENOBUFS resync helper for 0.16

When a multicast subscriber falls behind the kernel's event production
rate, the kernel returns `ENOBUFS` on the next `recvmsg()`. The
correct recovery is: (a) re-dump current state, (b) resume the
multicast stream. Currently nlink leaves this entirely to the user,
which the strategic analysis flagged as a pan-ecosystem pain point —
[Cilium issue #40280](https://github.com/cilium/cilium/issues/40280)
documents the same friction in Go.

This plan adds `events_with_resync()` — a stream that handles
`ENOBUFS` automatically by transparently re-syncing and resuming.

## 1. Summary

One new public method per multicast-supporting Connection type:

```rust
impl Connection<Route> {
    pub fn events_with_resync<F, T>(
        &mut self,
        groups: &[RtnetlinkGroup],
        resync: F,
    ) -> impl Stream<Item = ResyncedEvent<T>>
    where
        F: Fn(&Self) -> Pin<Box<dyn Future<Output = Result<Vec<T>>> + Send + '_>>;
}

// Same shape for Connection<Netfilter>, Connection<Nftables>, etc.
```

The user supplies the resync closure (typically a `get_links` /
`get_routes` / `get_conntrack` call). On `ENOBUFS`, the stream
invokes the closure, yields each item as a synthetic event, then
resumes the multicast stream. From the consumer's perspective the
stream never errors with overflow — it just yields a `Resynced`
marker followed by current-state replay.

Substantial design work because the resync dump and the multicast
stream are different code paths; combining them cleanly takes care.
Worth it: every downstream consumer of multicast events reinvents
this, often badly.

## 2. Goals

By end of plan:

- `Connection<P>::events_with_resync()` exists for every protocol with
  multicast support (Route, Netfilter, Nftables, possibly more).
- Stream yields a sum type: `Event(T) | Resynced(T) | Marker(Resync)`
  so consumers can distinguish multicast deltas from full-state
  replay.
- On `ENOBUFS`, the helper:
  1. Drains any remaining buffered events
  2. Invokes the user's resync closure
  3. Yields each resync item with `Resynced` variant
  4. Emits a `Marker(Resync)` boundary
  5. Resumes multicast stream
- A recipe `docs/recipes/events-with-resync.md` shows the canonical
  usage pattern.

## 3. Scope

### In

- `events_with_resync()` for `Connection<Route>`,
  `Connection<Netfilter>`, `Connection<Nftables>`.
- Sum-type stream item: `enum ResyncedEvent<T> { Event(T),
  Resynced(T), Marker(ResyncMarker) }`.
- Unit tests in `connection.rs::tests` for the state machine using a
  synthetic socket that yields ENOBUFS at a configurable point.
- Integration test under `tests/integration/events.rs` that
  intentionally overflows the socket buffer (small buffer + slow
  consumer + kernel-generated events) and verifies resync engages.
- Recipe.
- CHANGELOG entry.

### Out

- Automatic resync for GENL families (wireguard, macsec, etc.) —
  most GENL multicast volume is low enough that resync isn't useful.
  Wire if a user asks.
- `SO_RCVBUF` tuning helpers — that's a related but separate concern
  (increase the kernel-side buffer rather than recover from
  overflow). One-line helper if anyone asks; not in this plan.
- Cancellation safety beyond what the underlying socket provides.

## 4. Design

### 4.1 The wire-level signal

When a multicast subscriber falls behind, the kernel marks the socket
overflowed. The next `recvmsg()` returns `ENOBUFS` (errno 105). The
overflow is **consumed** by the read — subsequent reads succeed
normally, but the consumer has missed an unknown set of events
between the last successful read and the overflow point.

The correct recovery, per kernel maintainers' guidance:
1. On ENOBUFS, the subscriber assumes its view of state is stale.
2. Re-dump current state (the same dump that would have been done at
   subscription start).
3. Continue consuming multicast events from where the read left off.

### 4.2 API shape

```rust
// crates/nlink/src/netlink/connection.rs (or a new module)

/// A stream item from [`events_with_resync`].
pub enum ResyncedEvent<T> {
    /// A multicast event delivered normally.
    Event(T),
    /// An item replayed during resync after `ENOBUFS`.
    Resynced(T),
    /// A boundary marker. Emitted once when resync begins; once when
    /// resync finishes. Consumers tracking "I've seen the full state
    /// at least once" can use these to coordinate.
    Marker(ResyncMarker),
}

pub enum ResyncMarker {
    /// Resync started; the next items will be `Resynced(T)`.
    ResyncStart,
    /// Resync finished; subsequent items resume as `Event(T)`.
    ResyncEnd,
}

impl Connection<Route> {
    /// Subscribe to multicast events with automatic resync on
    /// `ENOBUFS`.
    ///
    /// The `resync` closure is called whenever the kernel reports
    /// the multicast socket has overflowed. Typically `resync`
    /// invokes the same `get_*` method used to initially seed the
    /// consumer's view (e.g. `get_routes` for route subscribers).
    ///
    /// # Example
    ///
    /// ```ignore
    /// let mut conn = Connection::<Route>::new()?;
    /// conn.subscribe(&[RtnetlinkGroup::Link])?;
    /// let stream = conn.events_with_resync(
    ///     &[RtnetlinkGroup::Link],
    ///     |c| Box::pin(c.get_links()),
    /// );
    /// pin_mut!(stream);
    /// while let Some(event) = stream.next().await {
    ///     match event {
    ///         ResyncedEvent::Event(link) => handle_delta(link),
    ///         ResyncedEvent::Resynced(link) => handle_replay(link),
    ///         ResyncedEvent::Marker(ResyncMarker::ResyncStart) => {
    ///             log::info!("resync starting; my state is stale");
    ///         }
    ///         ResyncedEvent::Marker(ResyncMarker::ResyncEnd) => {
    ///             log::info!("resync complete; state is fresh");
    ///         }
    ///     }
    /// }
    /// ```
    pub fn events_with_resync<F, T>(
        &mut self,
        resync: F,
    ) -> impl Stream<Item = Result<ResyncedEvent<T>>> + Send + '_
    where
        F: Fn(&Self) -> Pin<Box<dyn Future<Output = Result<Vec<T>>> + Send + '_>>
            + Send + Sync + 'static,
        T: FromNetlink + Send + 'static;
}
```

Note: the `resync` closure must work without `&mut self` (the
connection is already borrowed by the stream). It takes `&self` and
returns a boxed future. Slightly awkward; alternatives include:

- Take an owned `Connection` (and return a new stream/connection
  pair), letting the closure use `&mut`. Cleaner but changes the
  whole subscription model.
- Use a separate "dump" connection. The user provides two
  connections: one for events, one for resync dumps. Decouples the
  borrow.

The second option is what Cilium's issue suggests. **Adopt it:**

```rust
pub fn events_with_resync<T>(
    &mut self,
    dump_conn: Arc<Connection<Self::Protocol>>,  // separate dump conn
    resync: impl Fn(&Connection<Self::Protocol>) -> Pin<Box<dyn Future<Output = Result<Vec<T>>> + Send>> + ...,
) -> impl Stream<Item = Result<ResyncedEvent<T>>> + '_;
```

Trade-off: forces users to open two connections (one for
events/subscribe, one for dumps). Concrete cost: one extra socket per
subscriber. Not a big deal. Buys clean borrow semantics and the
ability to dump in parallel with consuming events.

**Coordination with Plan 159 (`ConnectionPool<P>`)**: the canonical
source for `dump_conn` is `pool.acquire().await?`. Document the
pattern in this plan's recipe (`docs/recipes/events-with-resync.md`)
once Plan 159 is in tree:

```rust
let pool = Arc::new(ConnectionPool::<Route>::builder().size(4).build().await?);
let events = events_conn.events_with_resync(
    Arc::clone(&pool),                            // dump_conn comes from the pool
    |c| Box::pin(c.get_links()),
);
```

The pool's RAII drop returns the connection on each resync — the
user gets clean parallelism without per-subscriber socket
bookkeeping.

### 4.3 State machine

```text
State::Normal
  poll_next: recv from events socket
    Ok(events) -> yield each as Event(t); stay in Normal
    Err(ENOBUFS) -> transition to State::ResyncStart

State::ResyncStart
  yield Marker(ResyncStart)
  transition to State::ResyncRunning { future: resync(dump_conn) }

State::ResyncRunning { future }
  poll the future
    Pending -> Pending
    Ready(Ok(items)) -> transition to State::DrainingResync { items: vec.into_iter() }
    Ready(Err(e)) -> yield Err; transition to State::Normal

State::DrainingResync { items }
  if let Some(item) = items.next() -> yield Resynced(item); stay
  else -> yield Marker(ResyncEnd); transition to State::Normal
```

Implementation: hand-roll as a `Stream` impl with state enum. Cleaner
than `async_stream::try_stream!` because the resync future ownership
gets messy through the macro.

### 4.4 Recipe outline

`docs/recipes/events-with-resync.md`:

1. The problem (kernel can overflow our socket; events lost; we
   don't know which).
2. The naive workaround (manually drop the stream, re-subscribe,
   re-dump; lossy and racy).
3. The `events_with_resync` shape (subscribe once, hand the stream a
   dump closure, get clean replay semantics for free).
4. How to track "have I seen full state at least once" using the
   `Marker` events.
5. Sizing the socket buffer (`SO_RCVBUF`) to reduce overflow
   frequency.
6. When NOT to use this (low-volume event consumers can just handle
   ENOBUFS as a hard error and bubble up).

## 8. Risks

- **State-machine complexity**: hand-rolled `Stream` with multiple
  states + an inner future is a common source of bugs. Mitigation:
  unit-test every state transition with a mock socket; don't ship
  until the unit tests cover at least normal/overflow/error paths.
- **Closure ergonomics**: the `resync` closure returning
  `Pin<Box<dyn Future>>` is awkward. Consider exposing a trait
  `ResyncFn` to give it a nicer name. Don't gold-plate; users will
  learn the pattern fast enough.
- **Integration-test flakiness**: deliberately overflowing the
  socket buffer in CI is timing-sensitive. The test may pass
  reliably on developer machines but flake on slower CI. Mitigation:
  use a tight loop with explicit `tokio::task::yield_now()` to
  schedule the producer ahead of the consumer; verify by adding a
  large delay in the consumer.
- **`Arc<Connection>` requirement for dump_conn**: forces the
  connection to be `Arc`-shareable, which has implications for tokio
  task spawning. Verify `Connection` is `Send + Sync` (or make it
  so).
