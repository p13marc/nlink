# Plan 152 — integration showcases for 0.16

Three end-to-end examples that demonstrate nlink working with the
adjacent ecosystem rather than in isolation. None of them changes the
library API; the value is **showing the library works in
production-shaped scenarios** that downstream users actually have.

## 1. Summary

| # | Showcase | Form | Effort |
|---|---|---|---|
| 1.1 | `aya` co-demo: load TC eBPF program with aya, attach via tcx (kernel 6.6+) or fall back to TC, monitor counters with nlink | `crates/nlink/examples/integrations/aya_tcx.rs` | 1 day |
| 1.2 | Prometheus exporter POC | `bins/exporter/` (new POC bin) | 1.5 days |
| 1.3 | OpenTelemetry tracing wire-up example | `crates/nlink/examples/observability/otel.rs` | 0.5 day |

Total: ~3 days. All three are additive — no impact on existing API,
no semver implications.

## 2. Goals

By end of plan:

- **`aya_tcx.rs`** demonstrates the canonical pattern for combining
  nlink (TC config / monitoring) with aya (eBPF program loading + tcx
  attachment). Closes the "tcx isn't in nlink" gap by showing the
  *right* integration: aya owns BPF, nlink owns netlink, they
  cooperate cleanly.
- **`bins/exporter/`** is a working `node_exporter`-style Prometheus
  exporter exposing nlink's coverage areas (qdisc stats, conntrack
  counts, xfrm SA counters, etc.) that `node_exporter` itself either
  doesn't cover or covers thinly. Adoption bait for ops teams.
- **`otel.rs`** is a 1-file example wiring nlink's existing `tracing`
  spans to OpenTelemetry / OTLP via `tracing-opentelemetry`. No nlink
  code changes — the spans are already there; this just shows users
  how to consume them.

## 3. Scope

### In

- All three examples / bins above.
- For `bins/exporter/`: `cargo build` clean, runs locally, exposes
  `/metrics` on a configurable port, includes a README.
- For `aya_tcx.rs`: documented as kernel-6.6+ only (tcx requirement);
  fallback to TC `add_filter` for older kernels.
- Recipes — at least one each linking the new bins/examples from
  `docs/recipes/`.
- CHANGELOG entry under Documentation section.

### Out

- Publishing `bins/exporter/` to crates.io (stays `publish = false`
  per the project's bin convention — POC, not maintained product).
- Production-hardening the exporter (rate limiting, auth, TLS) —
  leave that to a downstream "operator" project.
- Co-marketing post / coordinated release with the aya project
  (out of scope for the lib, worth doing as a follow-up).

## 4. Approach — per showcase

### 4.1 `aya_tcx.rs` co-demo

#### Design intent

This example does **triple duty**:

1. **aya + nlink integration**: the canonical pattern for using
   both libs in one program. Load an eBPF TC classifier with aya,
   attach via TCX (kernel 6.6+), use nlink for the netlink-side
   work (interface setup, TC tree inspection, monitoring).
2. **`nlink-macros` showcase** (added per Plan 154): the example
   uses `#[derive(GenlMessage)]` etc. to **define a small custom
   GENL family** that the BPF program exposes (e.g. a synthetic
   "counters" family for reading per-CPU counters from userspace).
   This proves the proc-macro derives work end-to-end in a real
   integration scenario.
3. **tcx vs TC fallback**: shows the right boundary between aya
   (BPF program loading + TCX attach) and nlink (TC qdisc/filter
   for older kernels, monitoring counters either way).

Concrete steps:

1. Load a minimal eBPF TC classifier with aya (counter on packet).
2. Attach via TCX (kernel 6.6+) or fall back to nlink's
   `add_filter(..., bpf)` on older kernels.
3. **Define a custom GENL family using `nlink-macros`** for the
   BPF→userspace counter readout path. ~20 lines using
   `#[derive(GenlMessage)]`, demonstrating that downstream crate
   authors can extend nlink without forking it.
4. Use nlink to create the dummy interface
   (`add_link(DummyLink::new("ebpf-test"))`), inspect the TC tree,
   subscribe to events.

The example is **opinionated**: it demonstrates one good pattern,
not a survey of every aya/nlink interop possibility. Real users
will copy-paste and adapt.

#### Files

```
crates/nlink/examples/integrations/
  aya_tcx.rs           # the example
  Cargo.toml           # aya + nlink + tokio = dev deps for this example
  bpf/                 # the BPF program source
    src/main.rs        # bumps a per-CPU counter
    Cargo.toml         # aya-ebpf crate
```

The BPF program is a few lines (counter on packet arrival). aya's
build infrastructure (`xtask` pattern, `aya-build`) compiles it.

#### Cargo.toml integration

Add an `[[example]]` entry in `crates/nlink/Cargo.toml`:

```toml
[[example]]
name = "integrations_aya_tcx"
path = "examples/integrations/aya_tcx.rs"
required-features = ["lab"]   # uses LabNamespace for clean isolation
```

aya is a heavy dep (~30 transitive crates). Don't add it as a
direct dependency of the lib; put it in `[dev-dependencies]` so it
only compiles when building examples.

#### README

`crates/nlink/examples/integrations/README.md`:

- The integration pattern's intent
- Kernel version requirements (tcx = 6.6+, falls back to TC
  `add_filter` for older)
- How to run: `sudo cargo run --example integrations_aya_tcx --features lab`
- What's nlink's job, what's aya's job, why they're separate
- Pointer at aya's docs for "now I want to do more eBPF"

#### Effort

1.5 days (bumped from 1 day in the original draft to absorb the
`nlink-macros` integration). The BPF program is trivial; the
orchestration is mostly boilerplate; the macros showcase is
shared with Plan 154's `define_taskstats.rs` example.

#### Dependency on Plan 154

This sub-item depends on Plan 154 (`nlink-macros` crate) shipping
first. Per the master plan §7 sequencing, Plan 154 lands before
Plan 152 — the dependency is satisfied by the release order.

### 4.2 Prometheus exporter POC (`bins/exporter/`)

#### What it exposes

```
nlink_qdisc_bytes_total{interface, kind, handle} counter
nlink_qdisc_packets_total{interface, kind, handle} counter
nlink_qdisc_drops_total{interface, kind, handle} counter

nlink_link_rx_bytes_total{interface} counter
nlink_link_tx_bytes_total{interface} counter
nlink_link_rx_errors_total{interface} counter
nlink_link_tx_errors_total{interface} counter

nlink_conntrack_entries{family, protocol} gauge
nlink_conntrack_inserts_total counter
nlink_conntrack_drops_total counter

nlink_xfrm_sa_count{family, mode} gauge
nlink_xfrm_sa_bytes_total{spi} counter (if per-SA stats available)

nlink_route_count{family, table} gauge
```

(All metric names follow the [Prometheus naming conventions](https://prometheus.io/docs/practices/naming/).)

Coverage rationale: `node_exporter` covers interface stats well but
skips qdisc/conntrack/xfrm. The nlink-based exporter complements
`node_exporter` rather than replacing it — same Prometheus scrape
endpoint, distinct metric prefixes.

#### Architecture

```
bins/exporter/
  src/
    main.rs           # axum or hyper-based /metrics endpoint
    collector.rs      # background tokio task that polls nlink every N seconds
    metrics.rs        # prometheus-client metric registry
  Cargo.toml
  README.md
```

Deps: `axum` or `hyper` for HTTP, `prometheus-client` for the
exposition format, `nlink` (workspace), `tokio`, `tracing` for logs.

A `--namespace <name>` flag scopes the exporter to a specific netns
(uses `nlink::lab` / `namespace::connection_for` for the dump path).

#### Cross-plan benefits

The exporter is a natural showcase for two parallel-plan
additions:

- **Plan 158 (`syscall_batch`)**: the exporter's poll loop is
  dump-heavy by definition (links + routes + qdiscs + conntrack
  per scrape). Build the exporter with `--features
  nlink/syscall_batch` enabled by default in its `Cargo.toml`;
  demonstrates the ≥1.5× speedup in a realistic scenario.
- **Plan 159 (`ConnectionPool<P>`)**: the per-namespace mode (one
  exporter watching N netns) wants one `ConnectionPool<Route>` per
  namespace, scrapes running concurrently. Show the
  `Arc<ConnectionPool>` + `tokio::spawn` per-namespace pattern in
  `bins/exporter/src/collector.rs` — the canonical real-world
  consumer.

The exporter's README calls out both features as "we use these
because we're the kind of workload they were designed for; if
you're polling at low frequency on a single namespace, you
probably don't need them."

#### Tests

Smoke test in CI: build the bin, start it, curl `/metrics`, grep
for at least one nlink_* metric. Functional, not exhaustive.

#### Effort

1.5 days. ~600 LOC including the HTTP server scaffolding.

### 4.3 OpenTelemetry tracing wire-up (`otel.rs`)

#### Intent

nlink's `tracing` instrumentation is already in place (every public
method has `#[instrument(...)]`). This example shows how to consume
those spans through `tracing-opentelemetry` → OTLP → Jaeger /
Tempo / etc.

#### Shape

```rust
// crates/nlink/examples/observability/otel.rs

#[tokio::main]
async fn main() -> nlink::Result<()> {
    // 1. Set up tracing-opentelemetry → OTLP exporter
    let tracer = opentelemetry_otlp::new_pipeline()
        .tracing()
        .with_exporter(opentelemetry_otlp::new_exporter().tonic())
        .install_batch(opentelemetry::runtime::Tokio)?;
    let otel_layer = tracing_opentelemetry::layer().with_tracer(tracer);
    tracing_subscriber::registry()
        .with(otel_layer)
        .with(tracing_subscriber::EnvFilter::from_default_env())
        .init();

    // 2. Do some nlink work — the spans flow to OTLP automatically.
    let conn = Connection::<Route>::new()?;
    let links = conn.get_links().await?;
    println!("{} links observed via OTel-instrumented spans", links.len());

    // 3. Shut down cleanly so the OTel batch exporter flushes.
    opentelemetry::global::shutdown_tracer_provider();
    Ok(())
}
```

That's the whole example. Maybe 50 LOC including imports + comments.

Adjacent README:

```
crates/nlink/examples/observability/
  otel.rs
  README.md       # "how to run with a local jaeger via docker"
```

#### Dev-deps

`opentelemetry`, `opentelemetry_otlp`, `opentelemetry-stdout`,
`tracing-opentelemetry`, `tracing-subscriber`. All in
`[dev-dependencies]`.

#### Effort

0.5 day including the README + verifying the example renders sensible
spans in Jaeger.

## 8. Risks

- **aya build complexity**: aya programs require nightly Rust + the
  `bpf-linker` toolchain. The example's README needs clear setup
  steps; otherwise users hit walls before running. Mitigation:
  start with aya's published examples and follow their patterns
  rather than rolling our own build infra.
- **Prometheus exporter scope creep**: it's tempting to add more
  metrics. Resist; ship the core 5-6 metric families as MVP. More
  can come in follow-up PRs based on user feedback.
- **OTel example dep churn**: `opentelemetry_otlp` API has changed
  multiple times. Pin to a specific version and document it in the
  README. Update when nudged by users, not proactively.
- **CI cost**: building aya examples in CI adds substantial compile
  time (nightly toolchain + LLVM). Mitigation: gate the aya example
  build behind a feature or run it only in the integration-tests
  workflow (which already uses a heavier container).

## 9. No out-of-scope follow-ups

This plan ships standalone examples; everything else is downstream
adopter territory (production-hardened exporter, full eBPF program
zoo, OTel dashboards). Those are good projects but not nlink's job.

End of plan.
