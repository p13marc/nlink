# Plan 161 — examples coverage for 0.16 features

The pre-release audit's example-coverage pass (`0.16-release-readiness-report.md` §6) found that three of four
**headline** 0.16 features shipped with recipes pointing at companion
examples that didn't exist. Plan 161 closes that gap: four
runnable examples, one per headline feature, plus a forward-
looking list of lower-priority extensions for 0.17.

## 1. Summary

| # | Example | Headline feature | LOC | Status |
|---|---|---|---|---|
| 1.1 | `examples/route/stream_dump.rs` | Plan 149 streaming dump API | 65 | ✓ landed (commit 325d77c) |
| 1.2 | `examples/nftables/declarative.rs` | Plans 157 + 157b — declarative `NftablesConfig` + per-rule USERDATA | 101 | ✓ landed (commit 325d77c) |
| 1.3 | `examples/pool/parallel_dump.rs` | Plan 159 `ConnectionPool<P>` | 67 | ✓ landed (commit 325d77c) |
| 1.4 | `examples/events/resync_loop.rs` | Plan 151 ResyncedEvent + ResyncMarker | 97 | ✓ landed (commit 325d77c) |

Plus 4 Cargo.toml `[[example]]` registrations (route_stream_dump,
events_resync_loop, nftables_declarative, pool_parallel_dump).

## 2. Why this is a 0.16 must-ship

Per the audit's example-agent verdict:

> "The four other headline 0.16 features — **streaming dumps,
> declarative NftablesConfig, ConnectionPool, ENOBUFS resync** —
> have zero example coverage, and three of them have recipes
> that reference companions that don't exist. Given that the
> maintainer flagged 'recipe-without-example is a real gap' and
> the cycle's tag line is 'polish,' I'd block the cut until at
> least the four declarative-Nftables / pool / resync /
> streaming-dump examples land."

Recipe-without-runnable-example is a doc-credibility regression:
downstream users follow recipes; if the code in the recipe
references "see the companion example for the full pattern"
and the companion file doesn't exist, the doc fails.

## 4. Per-example design

### 4.1 `examples/route/stream_dump.rs`

**Demonstrates**: `Connection::<Route>::stream_links()` +
`stream_routes()` returning `DumpStream<'_, P, T>` (Plan 149
foundation). Prints first 5 items of each stream then drains the
rest without storing. Closing message documents the
O(1)-memory invariant ("peak buffer use is one socket batch
(~8KiB), not the full dump").

**Key APIs exercised**: `Connection::stream_links`,
`Connection::stream_routes`, `tokio_stream::StreamExt::next`.

**Run**: `cargo run --example route_stream_dump`

**No special privileges**: read-only RTM_GETLINK / RTM_GETROUTE
on any host.

### 4.2 `examples/nftables/declarative.rs`

**Demonstrates** the full Plan 157+157b reconcile cycle:

1. Declare two keyed rules (`ssh-allow`, `icmp-allow`).
2. Compute first diff against empty kernel state → all-adds.
3. `apply()` atomically.
4. Re-compute diff → empty (idempotent reapply).
5. Mutate one rule's port + re-diff → single
   `rules_to_replace` op (in-place NLM_F_REPLACE via the
   `nlink:<key>` USERDATA-tagged handle lookup).
6. `apply()` the single replace.
7. Teardown with `NftablesConfig::new()` + apply (empty
   declared → cascade delete).

**Asserts**: idempotent re-diff produces empty diff; mutating
diff has exactly one `rules_to_replace` entry.

**Key APIs exercised**: `NftablesConfig::new`, `.table(...)`,
`.chain(...)`, `.rule_keyed(...)`, `cfg.diff(conn).await?`,
`diff.apply(conn).await?`, `NftablesDiff::rules_to_replace`.

**Run**: `sudo -E cargo run --example nftables_declarative`
(needs CAP_NET_ADMIN).

**Graceful EPERM handling**: prints a friendly message + exits
0 when run without root, so `cargo run` works for the
build-validation case.

### 4.3 `examples/pool/parallel_dump.rs`

**Demonstrates**: 16 tokio tasks contending on a 4-slot
`ConnectionPool<Route>`. Each task acquires, dumps links,
releases on drop. Bounded-mpsc backpressure visibly serializes
the 12 extra tasks behind the 4 pooled connections; per-task
latency is printed for the avg-fanout metric.

**Key APIs exercised**: `ConnectionPoolBuilder::<Route>::new()`,
`.size(4)`, `.build().await?`, `pool.acquire().await?`,
`Arc::new` wrapping (since `ConnectionPool` is not `Clone`).

**Run**: `cargo run --example pool_parallel_dump`

**Tokio runtime**: explicit `#[tokio::main(flavor =
"multi_thread", worker_threads = 4)]` so the fanout is real
concurrent execution, not single-threaded interleaving.

### 4.4 `examples/events/resync_loop.rs`

**Demonstrates** the ENOBUFS-recovery pattern from Plan 151
+ the `docs/recipes/events-with-resync.md` recipe. Subscribes to
RTNLGRP_LINK events; on ENOBUFS (event-channel overflow), prints
a `ResyncMarker::ResyncStart` → re-snapshot stub →
`ResyncMarker::ResyncEnd` triplet to show the documented
recovery boundary.

**Key APIs exercised**: `Connection::subscribe(&[RtnetlinkGroup::Link])`,
`conn.events()`, `Error::errno()` for ENOBUFS detection,
`ResyncedEvent::Marker(ResyncMarker::ResyncStart|ResyncEnd)`,
`ResyncedEvent::Resynced(T)`.

**Run**: `cargo run --example events_resync_loop`
(10-second bounded loop so the example terminates cleanly).

**Trigger ENOBUFS manually** (in another terminal as root):
`while true; do ip link add d0 type dummy; ip link del d0; done`

## 9. Risks

- **Examples shipping broken APIs**: the recipe-currency audit
  found 3 broken code samples (`match_proto(Proto::Icmp)` —
  non-existent API). All 4 Phase 1 examples were spot-checked
  against the real public API; the `events_resync_loop.rs`
  example had two compile errors in the first draft
  (`ResyncedEvent::Resynced` with no payload;
  `ResyncMarker` used as unit-like instead of an enum variant)
  that were fixed before commit. Mitigation: every example
  must build clean under `cargo build --examples` before
  registration; `audit-example-registration.sh` is a CI gate.
- **Example bit-rot**: same risk as the catalogued Plan 160
  orphans. Mitigation: the CI gate prevents new orphans;
  Phase 1 examples are all registered + actively built.
- **Hardware-gated examples (§7.3, §7.4)**: pure code-validation
  only; runtime correctness depends on a real NIC that the
  maintainer / CI doesn't have. Mitigation: document the
  `--apply` runner pattern + manual verification expectations
  in each example's top-of-file rustdoc.

End of plan.
