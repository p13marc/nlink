# Plan 220 — 0.20 cycle master plan

## 1. Why this plan exists

Three audits in parallel on the post-0.19 tree surfaced 60 findings.
Two clusters dominate scope:

- **6 CRITICAL wire-format defects in XFRM** —
  `flush_policy()` actually flushes all SAs, `flush_sa()` sends
  `UPDPOLICY` with a too-small body, `update_sa`/`update_sp` always
  EEXIST when their target exists, `XFRMA_SRCADDR` /
  `XFRMA_OFFLOAD_DEV` are emitting under wrong attribute IDs. Plus a
  test that **encodes the bug** by asserting the wrong constant.
  Verified personally against kernel UAPI v6.13. Carved out into
  Plan 221 as a **0.19.1 hotfix** because waiting for the 0.20 cycle
  risks a user silently dropping their entire SA table via
  `flush_policy()` in the meantime.

- **3 HIGH wire-format defects scattered**: `CtKey::Expiration = 7`
  (kernel value 5; nlink emits L3PROTOCOL byte instead of expiration
  ms u32 — silent), DPLL fractional-frequency-offset typed as
  `Option<i32>` (kernel emits `sint`, often 8 bytes), `TCA_HTB_OFFLOAD`
  off by one (dead code today, blocks future HTB-offload work).

The second cluster is **systemic prevention**: the 0.19 Plan 213
sizeof CI gate verifies struct sizes but not constant values. All 4
XFRM defects are constants. Extending the gate to assert constant
values against kernel UAPI is the single highest-leverage durable
fix and is Plan 222.

The third cluster is **defensive correctness** the audit surfaced
beyond wire format:

- `from_le_bytes` for NLA headers in 3 files that 0.19 N3 missed
  (broken on big-endian).
- `recv_msg` silently truncates frames > 32 KiB (no `MSG_TRUNC`
  check); the sibling `recv_batch_inner` path handles it correctly.
- WireGuard `parse_timespec` panics on negative `secs` (verified by
  repro) — kills any long-lived subscriber.

These plus the typed-API tightening and doc-drift sweep round out
the cycle. Full prioritized list in §3.

## 2. Cycle theme

> *"Constants are part of the wire format too."*

0.19's Plan 213 closed the **struct-size** drift class. 0.20 closes
the **constant-value** drift class. Both are silent failures that
type-checked code, dummy tests, and even existing integration tests
can hide. The Plan 213 gate would have caught all 4 XFRM constants
at build time if its scope had included msg-type / attr-ID values.

Secondary theme: **defensive correctness sweep**. The 0.19 audit
found N1-N9 + Findings A-D; this audit found B1-B20 + W1-W18 + A1-A22.
The pattern is consistent — every audit surfaces a comparable
volume because the codebase keeps growing. The cycle includes one
broad-coverage CI gate (`audit-bytes-le.sh`) that closes the
endianness class for good, and a doc-test gate that catches the
0.19 async-ification drift.

Tertiary theme: **carry the queued items**. The seed list in the
prior `plans/INDEX.md` carries Plan 197 (ovpn), Plan 205 follow-on
(purge correctly), F1 follow-on (NlRouter dispatcher), Plan 208
Phase 3-4 (GENL command unification), Plan 198 (declarative
`DeclaredSet`), Plan 189 §8 (`schemars`), Plan 193 phase 2-3
(`cargo-fuzz` + `proptest`), Plan 195 (`StreamBackoff` + `Store<K>`),
Plan 196 (`WireguardConfig::client()`), Plan 201 (broader Display +
From sweep), audit follow-ups H7 / H8. Not all will fit; the cycle
ends when CI is green + the audit-derived plans (221-233) land.
The remainder slides to 0.21 or gets cut from the roadmap.

real ordering constraint — the doc sweep needs to know the
final builder shape. Everything else can land in any order.

Within the broad 222→{223..233} fan-out, the only inter-plan
dependency is **228 (Percent) before 229 (doc sweep)** — the doc
sweep needs to know the final builder shape to write the new
examples against.

## 6. Migration guide outline

The `0.19.0-to-0.20.0.md` will need entries for:

- **Plan 221 wire-format**: `flush_sa` and `flush_policy` actually
  did the wrong thing before this release. Users with running
  reconcile loops on 0.18 / 0.19 that called `flush_policy()` were
  silently flushing all SAs instead of all SPs. Document the
  behaviour fix + steer them to verify their state matches
  expectations.
- **Plan 222 sizeof gate**: if any downstream code depends on the
  exposed `XFRM_MSG_*` / `XFRMA_*` constants (none today AFAIK),
  the values 25, 28, 9, 26 → 28, 29, 13, 28 are observable breaking
  changes for anyone destructuring against them.
- **Plan 224 `Error::Truncated`**: new `Error` variant. The enum is
  `#[non_exhaustive]` so adding it is non-breaking for matches that
  don't exhaustively match every variant; downstream exhaustive
  matches need a new arm (typically `Error::Truncated { .. } => ...`).
  New `is_truncated()` predicate.
- **Plan 226 DPLL**: `DpllPin::fractional_frequency_offset` and
  `_ppt` widen from `Option<i32>` to `Option<i64>` (same shape as
  0.19's `phase_offset` widening — both are signed-but-stored-too-narrow).
- **Plan 227 Family**: `flush_rules(family: u8)` deprecated in
  favour of `flush_rules(family: Family)`. Same for 2 sibling
  methods.
- **Plan 228 Percent**: `QdiscBuilder::loss(f64)` deprecated in
  favour of `QdiscBuilder::loss(Percent)`. Mirrors the 0.13 typed-
  units rollout and the 0.13→0.14 migration guide's `tc::builders`
  deprecation.
- **Plan 230 Verdict**: `Verdict::Jump(String)` / `Verdict::Goto(String)`
  switch to `Verdict::Jump(ChainName)` / `Verdict::Goto(ChainName)`.
- **Plan 231 RuleMessage**: `pub` fields → `pub(crate)` + accessor
  methods. Breaking for any downstream code that destructured.

Plus the seed-list items (197, 234, 235) that land discretionarily
get their own migration entries.

## 7. Risks + mitigations

- **Hotfix splash damage** (221). Risk: the 4-constant flip in
  `xfrm.rs` is so small that landing it without the broader audit's
  integration-test coverage could miss a regression. Mitigation:
  Plan 221 §5 requires 6 root-gated integration tests gated
  `require_root!()` + `require_modules!("xfrm")` *before* the
  hotfix tags `v0.19.1`. The privileged-CI job runs them on every
  push to the hotfix branch.

- **Plan 222's gate could become noise**. Risk: the gate could
  fire on every kernel UAPI extension (new constants added
  upstream). Mitigation: the gate asserts each constant **equals**
  its expected value; new kernel-side constants the gate doesn't
  know about don't fail the build. Constants the gate DOES know
  about that change upstream produce a build error with the new
  value in the message — exactly what we want.

- **Plan 228 (Percent) breakage**. Risk: the `loss(f64)` deprecation
  affects downstream code that passes a literal `0.01` and depends
  on `Percent::from_fraction` semantics. Mitigation: keep `loss(f64)`
  as a deprecated alias for one release, document the equivalent in
  the migration guide, delete in 0.21 per the user's stated
  "aggressive deprecation cadence" preference.

- **Doc-test gate (Plan 229) is slow**. Risk: `cargo test --doc`
  on the workspace can take 5+ minutes. Mitigation: gate only
  `cargo test --doc -p nlink` in CI, not the full workspace; sample
  the example-driven docs in the integration-tests workflow as a
  second pass.

- **Cycle slippage**. Risk: 13 deterministic plans is a lot. If the
  cycle drags past 4 weeks, the discretionary items (234-235) drop
  silently. Mitigation: weekly check-in on the deterministic-plan
  progress against the seed-list; INDEX.md updated each Friday.

## 8. Acceptance test

A representative downstream user (controller-style: `apply_reconcile`
in a tight loop) upgrades from 0.19.0 → 0.20.0 and:

- The XFRM behaviour silently changes: `flush_policy()` actually
  flushes policies now. If the user was depending on the broken
  behaviour (very unlikely — the broken behaviour returns wrong
  data), they need to know.
- The TC `loss(f64)` deprecation produces a compile warning, not a
  break.
- The DPLL FFO field type widens — a `let foo: i32 = pin.fractional_*`
  destructuring breaks at compile time (intentional).
- The Family-typed `flush_rules` deprecation produces a compile
  warning, not a break.
- `cargo check --target s390x-unknown-linux-gnu` compiles.

If any of those don't behave as listed, the cycle hasn't shipped.

## 11. Frame for the maintainer

This cycle is about closing audit-class bug categories at the
build-time check level. 0.19 closed wire-struct-size drift; 0.20
closes wire-constant-value drift, big-endian wire parsing, recv
truncation, panic-on-malformed-event. Each is one CI gate or
audit script away from being unable to recur. The typed-API and
doc cleanup are the secondary themes.

The hotfix (Plan 221) is the immediate ask. The rest can sequence
naturally over the cycle. Nothing here requires architectural
debate — the audit is conclusive on every finding it surfaces.
