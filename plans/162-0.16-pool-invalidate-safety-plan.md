# Plan 162 — `PooledConnection::invalidate` consume-self safety

The pre-release audit's bug-hunt agent found a panic-on-misuse
in the Plan 159 `ConnectionPool<P>` guard type: `invalidate()`
takes `&mut self` and sets `self.conn = None`, but the
`&mut` borrow ends when the method returns — so a subsequent
`&*pool_guard` (Deref through the guard) panics on the
`expect("PooledConnection holds a Connection until drop")`
unwrap in `Deref::deref`.

The fix is API-level: change `invalidate(self)` to consume the
guard. The panic path becomes statically impossible.

## 1. The bug

`crates/nlink/src/netlink/pool/pooled.rs`:

```rust
impl<'p, P: ProtocolState> PooledConnection<'p, P> {
    pub fn invalidate(&mut self) {
        if let Some(conn) = self.conn.take() {
            tracing::warn!(/* ... */);
            drop(conn);
        }
    }
}

impl<P: ProtocolState> Deref for PooledConnection<'_, P> {
    type Target = Connection<P>;

    fn deref(&self) -> &Connection<P> {
        self.conn
            .as_ref()
            .expect("PooledConnection holds a Connection until drop")
    }
}
```

User code that reproduces the panic:

```rust
let mut p = pool.acquire().await?;
p.invalidate();   // &mut borrow released here
let _ = &*p;      // calls Deref::deref → panic
```

The inline comment on `deref` argues this is "impossible from
safe code because `invalidate` takes `&mut self` and deref needs
`&self`" — but the `&mut` borrow doesn't outlive `invalidate()`'s
return. The compiler is happy to let the user do this; the
panic only fires at runtime.

## 2. Goals

By end of plan:

- A user calling `invalidate()` cannot subsequently
  `&*pool_guard` to trigger a panic.
- The change is source-compatible for the documented
  use cases ("invalidate then drop").
- `PooledConnection::deref` no longer needs the
  `expect(...)` — the panic path is statically unreachable.

## 3. Scope

### In

- Change `pub fn invalidate(&mut self)` to
  `pub fn invalidate(self)` (consume).
- Remove the `Option<Connection<P>>` wrapping (the `None`
  state becomes unrepresentable).
- Update `Deref::deref` to return `&self.conn` directly
  (no `expect`).
- Update `Drop::drop` accordingly (no `take()` dance; the
  `Connection` always exists).
- Update the doc-strings on `invalidate`, `deref`, and the
  struct to reflect the new ownership shape.
- Migration-guide entry under §Breaking changes.
- CHANGELOG entry under Changed.

### Out

- Adding a fallible `try_deref` for the case of "invalidate
  then keep using" — that pattern is incorrect by design; the
  user wants `acquire()` again instead.
- Changing the bounded-mpsc pool refill semantics (Plan 159 §8
  trade-off: invalidate drops the connection without returning
  it; pool runs one slot short until the next `acquire()`
  blocks-or-creates).

## 4. Design

### 4.1 New struct shape

```rust
/// Guard borrowed from a `ConnectionPool`. Drops the inner
/// `Connection` back into the pool on `Drop`, or with
/// [`invalidate`](Self::invalidate) to discard a connection
/// the caller no longer trusts (e.g. after a kernel error
/// suggesting the socket is in a bad state).
pub struct PooledConnection<'p, P: ProtocolState> {
    /// The borrowed connection. Always `Some` for the lifetime
    /// of this guard — the `Option` exists only so `Drop` can
    /// `take()` the value to return it to the pool.
    conn: Option<Connection<P>>,
    pool: &'p PooledConnectionInner<P>,
}

impl<'p, P: ProtocolState> PooledConnection<'p, P> {
    /// Discard this connection (don't return it to the pool).
    /// Takes `self` by value so the guard is gone after the
    /// call — no further `Deref` is possible from safe code.
    ///
    /// Use this when an operation on the connection returned an
    /// error suggesting the socket is in a bad state (e.g.
    /// connection state out of sync with kernel state after an
    /// unexpected close). The pool will run one slot short
    /// until the next `acquire()` waiter unblocks naturally;
    /// callers that need eager refill should `drop(pool)` and
    /// rebuild via `ConnectionPoolBuilder`.
    pub fn invalidate(mut self) {
        // Take the connection out so Drop doesn't try to
        // return it to the pool.
        if let Some(conn) = self.conn.take() {
            tracing::warn!(/* ... */);
            drop(conn);
        }
        // `self` drops here; Drop sees conn == None and is a
        // no-op.
    }
}

impl<P: ProtocolState> Deref for PooledConnection<'_, P> {
    type Target = Connection<P>;

    fn deref(&self) -> &Connection<P> {
        // SAFETY: conn is set on construction and only cleared
        // by `invalidate(self)` which consumes the guard. A
        // `&self.conn` borrow proves the guard hasn't been
        // moved out of, so conn is still Some. The .as_ref()
        // pattern keeps the unwrap explicit but unreachable.
        debug_assert!(self.conn.is_some(),
            "PooledConnection::conn cleared without invalidate(self) — impossible bug");
        self.conn.as_ref().expect("see debug_assert above")
    }
}

impl<P: ProtocolState> Drop for PooledConnection<'_, P> {
    fn drop(&mut self) {
        if let Some(conn) = self.conn.take() {
            // try_send is non-blocking; can only fail if the
            // receiver was dropped (pool closed) — in which
            // case dropping `conn` here is the correct teardown.
            let _ = self.pool.available.try_send(conn);
        }
    }
}
```

The `Option<Connection<P>>` stays — `Drop` needs to `take()` to
move the connection out — but the only way it becomes `None`
before `Drop` runs is via `invalidate(self)` which consumes the
guard. The `debug_assert!` in `deref` documents the invariant
and panics in debug builds if a future refactor breaks it; in
release builds the `expect` is unreachable.

### 4.2 Why `invalidate(self)` is the right shape

This matches the Rust standard-library pattern for "I am done
with this guard, possibly in a non-default way":

- `mpsc::SyncSender::send` — consumes the sender.
- `Box::leak(b)` — consumes the box.
- `JoinHandle::join(self)` — consumes the handle.

All of them give up the guard's normal lifecycle in exchange
for a different outcome. `invalidate(self)` is "I'm done with
this guard, and I don't want the default drop-back-into-pool
behavior."

## 5. Breaking-change shape

This IS a breaking change. Callers using:

```rust
let mut p = pool.acquire().await?;
p.invalidate();
```

still compile — the existing `&mut` form is upgraded to a
move, and `p` is gone after the call (same semantic outcome).
Callers using the bug-shape:

```rust
let mut p = pool.acquire().await?;
p.invalidate();
let _ = &*p;      // <- this stops compiling (use-after-move)
```

now fail to compile with E0382 (use of moved value) instead of
panicking at runtime. **Compile error is strictly better than
runtime panic.**

Callers that called `invalidate()` through a `&mut` field
(rare, but possible) need to restructure to consume by value.

## 9. Risks

- **Compile-fail test machinery**: `trybuild` adds a dev-dep
  and 5-10 sec of CI time. Acceptable; we already use it for
  `nlink-macros`. Alternative: a `compile_fail` rustdoc example
  in the `invalidate` doc comment — zero new dev-dep, same
  effect. Recommended.

- **Breaking-change visibility**: 0.16 was supposed to be
  mostly-additive. Adding `invalidate(self)` to the migration
  guide widens the breaking-change list. Mitigation: this is
  a 0.16-internal API (Plan 159 ships in 0.16); no downstream
  user could be using the bug-shape today.

- **Maintainer-judgement deferral**: if the cut is imminent
  and Plan 162 doesn't land pre-cut, it ships in 0.16.1 as a
  bug-fix patch. The runtime-panic-on-misuse is acceptable for
  one release of soak; the fix is non-blocking.

## 10. Out-of-scope follow-ups

- **Pool refill on `invalidate`**: Plan 159 §8 documents the
  "one slot short until next acquire" trade-off. If users want
  eager refill, a `pool.refill(invalidated_count).await?`
  helper could land in 0.17. Demand-gated.

- **`PooledConnection::into_inner(self) -> Connection<P>`**:
  consumes the guard, returns the inner `Connection<P>` outside
  the pool's lifecycle (caller becomes the new owner; doesn't
  return on drop). Useful for handoff to a long-running task.
  Out of scope for 0.16; opens its own plan when a downstream
  use case asks.

End of plan.
