## 1. Why this plan exists

The post-audit verification pass (commit `ccbb661`) corrected a
factual error that had been propagating across docs: **Plan 140's
privileged-CI workflow shipped in 0.15.0**, not as a future-work
item. `.github/workflows/integration-tests.yml` is in tree and
runs the full integration-test suite on every push/PR to `master`
under a container with `CAP_NET_ADMIN` + `CAP_SYS_ADMIN` +
`seccomp=unconfined` + `apparmor=unconfined` + bind-mounted
`/lib/modules`.

This changes the meaning of the 20 new Plan 166 tests:

- **Old assumption**: "ship in 0.16, run under Plan 140 once that
  lands." → They were considered shelf-ware until a future CI
  upgrade.
- **Actual situation**: they activate the moment `0.16` is
  pushed — PR #3 (`0.16` → `master` draft) re-runs CI on every
  push to head, so the new tests get exercised against root in
  a privileged container immediately.

The 20 tests have only ever been verified locally as a non-root
user, where they all early-return. **The cut activation flow
must include surfacing their first real CI run + reacting to
anything it surfaces.** That's what this plan is for.

## 4. Phase 3 — failure triage (conditional)

Three failure modes are plausible. Each has a known shape and a
known fix.

### 3.1 Timing-dependent test flakes

Candidates: `pool_acquire_blocks_then_succeeds_after_release` (uses
a 100ms sleep before asserting `!waiter.is_finished()`),
`wait_link_up_returns_when_link_comes_up` (similar
spawn-then-sleep-then-act shape).

**Probability**: low. The CI workflow enforces
`--test-threads=1` (Plan 140 §"--test-threads=1 mandatory"),
which removes the parallel-test interference that's the
usual driver of these races. 100ms is several orders of magnitude
longer than a tokio scheduling tick on `rust:bookworm`. But OS
scheduling under load is non-deterministic.

**Fix shape if it flakes**: replace the real-time sleep with a
`tokio::sync::Notify` handshake so the test depends on logical
ordering, not wall-clock. ~15 min per test. Re-push.

### 3.2 Module-gate misses

The CI workflow modprobes ~50 modules at job start. If any test
gates on a module not in that list (`require_module!("foo")`
where "foo" isn't loaded + isn't built in), the test early-exits
as "skip", which logs `skip:` to stderr but passes. Not a
failure — but worth scanning the log for "skip:" lines that
identify a coverage gap. Add the missing modprobe to
`integration-tests.yml` for the next cycle.

Confirmed modprobe list (per the workflow): `dummy`, `veth`,
`bridge`, `bonding`, `ifb`, `vrf`, `macvlan`, `ipvlan`, `vlan`,
`vxlan`, `geneve`, `gre`, `gretap`, `ip6_gre`, `ipip`, `sit`,
`xfrm_user`, `xfrm4_tunnel`, `xfrm6_tunnel`, `macsec`, plus
TC qdiscs / classifiers / actions, plus `nf_tables`,
`nf_conntrack`, `nf_conntrack_netlink`.

Plan 166's new tests require: `dummy`, `nf_tables`,
`nf_flow_table` (Plan 150 flowtable tests). The first two are
present. **`nf_flow_table` is NOT in the modprobe list** — the
flowtable tests will skip with a `skip: nf_flow_table` line.
Add it to the next-cycle workflow update; not a regression for
this cut (test skips ≠ test fails).

### 3.3 Real bug

Possible if a Plan 166 test has a logical error that the local
non-root runs couldn't surface. Read the assertion that failed,
read the corresponding lib code, fix-forward. ~30 min – 1 h
per bug. Re-push.

## 6. Phase 5 — `cargo publish --dry-run`

Both crates must dry-run cleanly. Order matters: `nlink-macros`
first because `nlink`'s `Cargo.toml` has `version = "..."` on
the `nlink-macros` dep, so the dry-run of `nlink` resolves that
version off the local workspace.

```bash
cargo publish -p nlink-macros --dry-run
cargo publish -p nlink --dry-run
```

Both should succeed. Surface anything they flag (missing
metadata, README link rot, etc.) and fix-forward.

## 12. Risks

- **Timing flake on first CI run**: low-probability, low-cost
  fix-forward. Mitigation: §3.1 plan.
- **`cargo publish --dry-run` rejects nlink-macros for missing
  metadata**: known historical landmine. Mitigation: dry-run
  CI step in `rust.yml` would catch it; if not, fix the
  `Cargo.toml` field, push, re-run.
- **crates.io index propagation timing**: the 30s sleep
  between `nlink-macros` publish and `nlink` publish is the
  conservative wait per CLAUDE.md. If it fails with "no
  matching version found", sleep another 60s and retry.
- **Post-cut "we shipped breaking changes" surprise**: every
  documented breaking change is in the migration guide
  (`docs/migration_guide/0.15.1-to-0.16.0.md`). The
  `cargo public-api` review (Phase 4) is the catch-net for
  any undocumented one.

## 13. Why a plan for this and not just a checklist

The 0.16 cycle's audit-driven follow-ups touched 24 files +
shipped 20 new integration tests + added 2 breaking changes +
locked down 11 types with `#[non_exhaustive]`. The cut PR (#3)
is the single moment where everything lands at once, and the
new CI surface is the single moment where the new tests run
against root for the first time. A plan-shaped artifact keeps
the sequencing and the failure-mode triage in one place — both
for executing the cut and for any post-cut retrospective.

End of plan.
