# Plan 163 — `#[non_exhaustive]` lockdown for 0.16 new pub structs

The pre-release audit's bug-hunt agent found that `RuleInfo`
(nftables, gained 3 new fields in Plan 157b v2) is **missing
`#[non_exhaustive]`** — once 0.16 publishes, adding it later is
a breaking change (downstream code that destructures the struct
in a `match` arm without `..` will break). The right window to
lock down the surface is **before** the cut.

The right scope: every public struct that **shipped first in
0.16** and might gain fields in a future release should carry
`#[non_exhaustive]` proactively. This plan enumerates them,
adds the attribute where appropriate, and notes the (small)
migration-guide implications.

## 1. The semver mechanic

Per [the Rust API guidelines](https://rust-lang.github.io/api-guidelines/future-proofing.html#c-struct-private):

- A pub struct with all-pub fields is **destructure-stable** by
  default: any future field addition is a major-version break.
- Adding `#[non_exhaustive]` forces external destructuring to
  use `..` and forbids struct-literal construction. **Once
  added, downstream code that doesn't use `..` or `Default`
  fails to compile.**
- Adding it **before publish** costs nothing (no downstream user
  is using these types yet).
- Adding it **after publish** is itself a breaking change
  (requires a major-version bump).

So `#[non_exhaustive]` on every new 0.16 pub struct **before
the cut** locks down the forward-compatibility surface for
free. Skipping it means we're committing to the exact field
list shipped in 0.16 for the lifetime of every `[0.16.x]`
patch + every `[1.0.x]` post-1.0.

## 2. Goals

By end of plan:

- Every new-in-0.16 public struct that's likely to gain fields
  later carries `#[non_exhaustive]`.
- The migration guide documents the lockdown.
- 0.16.x patch + 0.17 + 1.0 cycles can freely add fields to
  these types without an SHV bump.

### In — types to deliberately NOT lock down

- `NftablesConfig` — opaque accessors only (no public fields);
  `#[non_exhaustive]` here is redundant. (Verify; if any fields
  are pub, add the attr.)
- `Flowtable`, `Rule`, `Chain`, `Table` (nftables) — these
  shipped pre-0.16 and `#[non_exhaustive]` on them now would
  be a breaking change to an already-stable 0.15.1 surface.
  Out of scope.
- `XfrmSaBuilder` / `XfrmSpBuilder` — shipped pre-0.16, same
  reason.
- `LinkMessage` / `RouteMessage` / `AddressMessage` /
  `NeighborMessage` / `TcMessage` — shipped pre-0.16.

### Out

- Adding `#[non_exhaustive]` to 0.15.1-shipped types (would be
  a 0.16 breaking change beyond what the migration guide
  covers).
- Internal-only `pub(crate)` types — they're not on the public
  surface.

## 4. Per-type design

### 4.1 `RuleInfo` (`nftables/types.rs:1193`)

```rust
/// Rule info parsed from a dump.
+#[non_exhaustive]
 #[derive(Debug, Clone)]
 pub struct RuleInfo {
     /// Table name.
     pub table: String,
     // ... existing fields ...
     pub expression_bytes: Vec<u8>,
 }
```

**Construction paths**: only the in-crate `parse_rule` constructs
`RuleInfo`. External code receives it via `list_rules` /
`stream_rules` outputs. No downstream user can be constructing
it without a field-add path on us.

**Destructuring paths**: downstream code matching on `RuleInfo`
likely binds individual fields. The `..` will be required on
fresh code; existing 0.16-window code (since 157b shipped
2026-05-24) is forced to add `..`.

### 4.2 `NftablesDiff` (`nftables/config/diff.rs:22`)

```rust
+#[non_exhaustive]
 #[derive(Debug, Clone, Default)]
 pub struct NftablesDiff {
     pub tables_to_add: Vec<DeclaredTable>,
     // ... 10 more fields ...
 }
```

**Construction paths**: only `NftablesConfig::diff(&conn)`
constructs it. Downstream user code reads + calls methods.

**Destructuring paths**: unlikely (11 fields). If anyone does
match exhaustively, they'd be brittle anyway.

### 4.3 `ReconcileOptions` (`nftables/config/apply.rs`)

```rust
+#[non_exhaustive]
 #[derive(Debug, Clone)]
 pub struct ReconcileOptions {
     pub max_retries: usize,
     pub backoff: Duration,
 }
```

**Catch**: `#[non_exhaustive]` forbids struct-literal
construction. Today's `ReconcileOptions { max_retries: 5,
backoff: Duration::from_millis(50) }` would break. Fix: add a
`Default` impl + builder-style setters, OR offer a `new(retries,
backoff)` constructor. Recommended: `Default` + setters so
existing usage code becomes:

```rust
// Was:
ReconcileOptions { max_retries: 5, backoff: ... }
// Becomes:
ReconcileOptions::default().max_retries(5).backoff(...)
```

The migration-guide entry documents the change. **This is the
one item in Plan 163 that has user-visible impact** — every
other type is parser-output and isn't constructed externally.

### 4.4 DPLL + net_shaper reply structs

All are `#[derive(Debug, Default, Clone)]` with public fields.
`#[non_exhaustive]` is purely additive — they're parser outputs.

```rust
+#[non_exhaustive]
 #[derive(GenlMessage, Debug, Default, Clone)]
 #[genl_message(cmd = DpllCmd::DeviceGet)]
 pub struct DpllDeviceReply { /* ... */ }
```

### 4.5 ResyncedEvent + ResyncMarker (`resync.rs`)

```rust
+#[non_exhaustive]
 #[derive(Debug, Clone, PartialEq, Eq)]
 pub enum ResyncedEvent<T> {
     Event(T),
     Resynced(T),
     Marker(ResyncMarker),
 }

+#[non_exhaustive]
 #[derive(Debug, Clone, Copy, PartialEq, Eq)]
 pub enum ResyncMarker {
     ResyncStart,
     ResyncEnd,
 }
```

`#[non_exhaustive]` on an enum means `match` arms must include
a `_ =>` wildcard. Downstream code that uses
`matches!(evt, ResyncedEvent::Resynced(_))` is unaffected.

## 5. Implementation phases

### Phase 1: Audit + add `#[non_exhaustive]` (mechanical)

1. Add `#[non_exhaustive]` to each type in §3 "in" list.
2. Compile-check: `cargo build -p nlink --all-features`.
3. Lib tests: `cargo test -p nlink --lib --all-features`. Note
   that any in-tree code that constructs `ReconcileOptions {
   ... }` literally needs to switch to a constructor or
   `Default::default().setter()` chain.
4. Update `ReconcileOptions` doc-string + add builder-style
   setters (`max_retries(n)`, `backoff(d)`) returning `Self`.

### Phase 2: Migration-guide entry

5. Add to `docs/migration_guide/0.15.1-to-0.16.0.md` under
   §Breaking changes:

   > ### Multiple 0.16-new pub structs/enums are `#[non_exhaustive]`
   >
   > To preserve forward-compatibility (future field/variant
   > additions in 0.16.x patch + 0.17 + 1.0 without semver
   > bumps), every new-in-0.16 public type that's likely to
   > grow is locked down with `#[non_exhaustive]`. Affected:
   > `RuleInfo`, `NftablesDiff`, `ReconcileOptions`,
   > `ReconcileReport`, `DpllDeviceReply`, `DpllPinReply`,
   > `DpllPinParentDevice`, `DpllPinParentPin`, `NetShaperReply`,
   > `NetShaperCapsReply`, `ConnectionPool`,
   > `ConnectionPoolBuilder`, `PooledConnection`, `DumpStream`,
   > `ResyncedEvent`, `ResyncMarker`.

   > Concrete user-visible change: `ReconcileOptions` can no

   > ```rust

   > constructed; the lockdown is invisible at call sites.

## 8. Risks

- **`ReconcileOptions` struct-literal construction in user
  code**: any 0.16-pre-cut adopter could be writing it as a
  struct literal already. Mitigation: the migration guide
  shows the rewrite; the `Default` + setter pattern is one
  line longer and reads better.
- **`cargo-semver-checks` false positive**: the tool can flag
  `#[non_exhaustive]` additions on new types as "field add
  needs a major bump" if it doesn't compare against 0.15.1.
  Mitigation: the tool is comparing the published 0.15.1
  surface, where these types don't exist; lockdown registers
  as a new type with the attr, not a transition. Verify with
  `cargo public-api diff` before cut.
- **Maintainer-judgement deferral**: if cut is imminent and
  Plan 163 doesn't land, the consequence is that 0.17 has to
  bump to 1.0 to add fields to any of these 16 types. That's
  enforceable but constraining. Strong recommendation to land
  pre-cut.

## 9. Why this matters more than it looks

This plan is small (~1 hour of mechanical work) but the
consequences of skipping it compound for years:

- Every 0.17 field addition that should have been additive
  becomes a major bump.
- Every kernel-attribute extension on DPLL / net_shaper
  (highly likely; both are young families) forces a 0.18 cut
  even when the user-side change is "I get one more `Option`
  field for free."
- Re-applying `#[non_exhaustive]` later is itself a major
  break; users get hit twice.

The Rust API guidelines list this as **C-STRUCT-PRIVATE** /
**C-FUTURE-PROOFING** for a reason: it's free pre-publish,
expensive after.

## 10. Out-of-scope follow-ups

- **Pre-existing 0.15.1 types**: `Flowtable`, `Rule`, `Chain`,
  `Table` (nftables), `XfrmSaBuilder`, message structs from
  rtnetlink. Adding `#[non_exhaustive]` to these would be a
  breaking change beyond what the 0.15.1→0.16.0 migration
  guide covers. Defer to the 1.0 surface-freeze cycle.
- **`#[non_exhaustive]` on `Error` variants** is already done
  in 0.16 (Plan 155 added it as part of the ext-ack work).

End of plan.
