into one final commit before tagging 0.16.0.

## 2. Why bundle as a single plan

Each fix is a one-line edit. Spreading them across 5 commits
adds noise to the release-cut history without adding value.
Bundling lets the cut PR contain exactly:

- The Plan 161 examples commit (already landed: `325d77c`).
- The Plan 162 PooledConnection commit (~1 hour).
- The Plan 163 `#[non_exhaustive]` lockdown commit (~1 hour).
- **The Plan 165 polish commit (~20 minutes)** — this one.
- The release-cut commit (consolidate `[Unreleased]` →
  `[0.16.0]`, tag, push).

Single-commit polish is more reviewable + cleaner in the
release-PR diff.

## 7. Why this matters

The audit deliberately separated **correctness/regression-risk
items** (handled in Plans 161, 162, 163) from **cosmetics**
(this plan). Skipping Plan 165 doesn't make the release wrong
— it makes it slightly less polished. Per the maintainer's
"0.16 is an important milestone, we cannot introduce
regression" framing, the polish bar is appropriate.

Total cost: 20 minutes. Total benefit: a release whose docs
read consistently from any entry point (CLAUDE.md, README,
CHANGELOG, plans/INDEX.md, individual plans).
