# Plan 159 — `ConnectionPool<P>` for 0.16

Master plan §4 explicitly defers neli's `NlRouter`-style
multi-request-per-socket fanout to 0.17 — too architectural for
the 0.16 cycle. **`ConnectionPool<P>` closes part of that gap
without the architectural rework**: instead of multiplexing N
requests over one socket via sequence-number demux, hold N sockets
and round-robin requests across them.

For the documented use cases (concurrent route lookups, parallel
multi-namespace work, high-fanout dump pipelines), the pool gives
≥80% of the throughput benefit at <10% of the implementation risk.
Sockets are file descriptors — cheap.

## 1. Summary

One new public type + a builder:

```rust
let pool = ConnectionPoolBuilder::<Route>::new()
    .size(num_cpus::get())
    .acquire_timeout(Duration::from_secs(5))
    .build()
    .await?;

let conn = pool.acquire().await?;        // PooledConnection<Route>
let links = conn.get_links().await?;
// conn drops → returned to pool
```

`PooledConnection<P>` derefs to `&Connection<P>` so all existing
typed methods Just Work. RAII drop returns the connection to the
pool.

Plus namespace-aware variant (`ConnectionPool::for_namespace(...)`)
and a recipe.

## 2. Goals

By end of plan:

- A high-fanout consumer (e.g. a metrics exporter dumping routes
  every second across 50 namespaces) can use the pool to issue
  parallel dumps without manually managing socket lifecycles.
- Pool sizing is configurable; sensible default (`num_cpus`).
- Acquire is `async` with timeout — never blocks the executor;
  EAGAIN-on-saturation surfaces as a typed error.
- Per-namespace pools are first-class — a `ConnectionPool<Route>`
  is bound to one netns, just like a single `Connection<Route>`.
- The pool **transparently handles dead connections**: if a
  pooled `Connection` becomes unhealthy (socket closed, kernel
  killed it, etc.), `acquire` rebuilds it on the way out rather
  than handing back a broken one.
- Closes the documented Cilium-style fanout pattern from
  COMPARISON_NELI.md §5 with a path that's reviewable in days,
  not weeks.

### Out

- True NlRouter-style single-socket fanout — deferred to 0.17 per
  master plan §4 item 6. The pool is the partial alternative.
- Adaptive pool sizing (grow/shrink based on load). Bounded-size
  pool is enough for 0.16; resize-on-demand is feature creep.

  Useful for ops, but `tracing` spans cover the same need with
  zero pool-side code.
- Pool of GENL connections that share a family-ID resolution
  cache. Plan-154-aware feature; would benefit DPLL pools etc.
  Demand-gated; defer until a downstream user asks.

## 4. Design

### 4.1 The user-facing API

```rust
use nlink::{ConnectionPool, ConnectionPoolBuilder, Route};

let pool = ConnectionPoolBuilder::<Route>::new()
    .size(8)
    .acquire_timeout(Duration::from_secs(2))
    .build()
    .await?;

// Acquire returns a PooledConnection<Route> that derefs to
// &Connection<Route>; all typed methods Just Work.
let conn = pool.acquire().await?;
let links = conn.get_links().await?;
drop(conn);   // returned to pool

// Parallel dumps across the pool:
let pool = Arc::new(pool);
let mut joins = Vec::new();
for ns in namespace_list {
    let pool = pool.clone();
    joins.push(tokio::spawn(async move {
        let conn = pool.acquire().await?;
        conn.get_routes().await
    }));
}
for j in joins { let _ = j.await; }
```

The pool itself is `Send + Sync` — designed to be wrapped in `Arc`
and shared across tasks. `acquire` is `&self` (not `&mut self`).

### 4.2 The type design

```rust
// crates/nlink/src/netlink/pool/pool.rs

pub struct ConnectionPool<P: ProtocolState> {
    inner: Arc<PoolInner<P>>,
}

struct PoolInner<P: ProtocolState> {
    /// Bounded async channel acts as both pool storage AND
    /// concurrency limiter — `acquire` is `recv()`, `release` is
    /// `send()`. tokio's channel handles backpressure for free.
    available: tokio::sync::mpsc::Sender<Connection<P>>,
    receiver: Mutex<tokio::sync::mpsc::Receiver<Connection<P>>>,
    /// Namespace identity, if pool is namespace-bound. Used for
    /// rebuilding dead connections.
    namespace: Option<String>,
    /// Bookkeeping for rebuilds + diagnostics.
    size: usize,
    acquire_timeout: Duration,
    /// Builder closure for rebuilding dead connections.
    build: Arc<dyn Fn() -> BoxFuture<'static, Result<Connection<P>>> + Send + Sync>,
}

impl<P: ProtocolState> ConnectionPool<P> {
    pub async fn acquire(&self) -> Result<PooledConnection<'_, P>> {
        let recv_fut = async {
            let mut rx = self.inner.receiver.lock().await;
            rx.recv().await
        };
        match tokio::time::timeout(self.inner.acquire_timeout, recv_fut).await {
            Ok(Some(conn)) => {
                // Health-check the connection before handing it out.
                // If it's dead, rebuild via the stored builder closure.
                let conn = if self.is_healthy(&conn) {
                    conn
                } else {
                    tracing::warn!(
                        ns = ?self.inner.namespace,
                        "connection unhealthy, rebuilding"
                    );
                    (self.inner.build)().await?
                };
                Ok(PooledConnection { pool: self, conn: Some(conn) })
            }
            Ok(None) => Err(Error::PoolClosed),
            Err(_) => Err(Error::PoolExhausted {
                size: self.inner.size,
                timeout: self.inner.acquire_timeout,
            }),
        }
    }

    fn is_healthy(&self, _conn: &Connection<P>) -> bool {
        // Conservative: assume healthy unless a previous user marked
        // it bad. A more thorough check would poll the underlying
        // fd or send a no-op getlink — both add latency. Defer to
        // 0.17.
        true
    }

    pub fn size(&self) -> usize { self.inner.size }
    pub fn namespace(&self) -> Option<&str> {
        self.inner.namespace.as_deref()
    }
}

// crates/nlink/src/netlink/pool/pooled.rs

pub struct PooledConnection<'p, P: ProtocolState> {
    pool: &'p ConnectionPool<P>,
    conn: Option<Connection<P>>,
}

impl<P: ProtocolState> Deref for PooledConnection<'_, P> {
    type Target = Connection<P>;
    fn deref(&self) -> &Connection<P> {
        self.conn.as_ref().expect("PooledConnection always holds a Connection")
    }
}

impl<P: ProtocolState> Drop for PooledConnection<'_, P> {
    fn drop(&mut self) {
        if let Some(conn) = self.conn.take() {
            // Best-effort return to pool; if the channel is closed
            // (pool dropped), the Connection drops normally.
            let _ = self.pool.inner.available.try_send(conn);
        }
    }
}
```

Design choices:

- **`tokio::sync::mpsc` as the pool channel**: capacity == pool
  size; `recv` is async-friendly; backpressure is built-in. Used
  this pattern over a custom semaphore + Vec because the channel
  handles "wake the next waiter when something's returned" for
  free.
- **`Mutex<Receiver>`**: mpsc's receiver isn't `Sync` by default;
  one mutex wrapping it lets multiple acquirers share. Contention
  is fine: acquire latency is microseconds, useful work is
  milliseconds.
- **Builder closure for rebuilds**: stored as `Arc<dyn Fn() ->
  BoxFuture>`. Captures the namespace + any other construction
  parameters. Lets `acquire` rebuild without the user's
  involvement.
- **Health-check is a stub for 0.16**: real fd-poking adds
  complexity. Defer the smart check to 0.17 with a separate
  design pass. For now, "unhealthy" means "previous user
  explicitly marked it" — exposed via
  `PooledConnection::invalidate()`:

```rust
impl<P: ProtocolState> PooledConnection<'_, P> {
    /// Mark this connection as unhealthy. On drop, it will be
    /// rebuilt instead of returned to the pool. Use when a
    /// recoverable error suggests the socket may be in a bad
    /// state (rare; most kernel errors are per-request, not
    /// per-socket).
    pub fn invalidate(&mut self) {
        if let Some(conn) = self.conn.take() {
            drop(conn);  // socket closes; pool size shrinks by 1
            // ... pool rebuilds on next acquire fall through.
        }
    }
}
```

### 4.3 The builder

```rust
// crates/nlink/src/netlink/pool/pool.rs

pub struct ConnectionPoolBuilder<P: ProtocolState> {
    size: usize,
    acquire_timeout: Duration,
    namespace: Option<String>,
    _phantom: PhantomData<P>,
}

impl<P> ConnectionPoolBuilder<P>
where
    P: ProtocolState + sealed::SyncConstructible + 'static,
{
    pub fn new() -> Self {
        Self {
            size: num_cpus::get(),
            acquire_timeout: Duration::from_secs(5),
            namespace: None,
            _phantom: PhantomData,
        }
    }

    pub fn size(mut self, size: usize) -> Self {
        self.size = size.max(1);   // 0 is meaningless; clamp
        self
    }

    pub fn acquire_timeout(mut self, timeout: Duration) -> Self {
        self.acquire_timeout = timeout;
        self
    }

    pub fn namespace(mut self, name: impl Into<String>) -> Self {
        self.namespace = Some(name.into());
        self
    }

    pub async fn build(self) -> Result<ConnectionPool<P>> {
        let (tx, rx) = tokio::sync::mpsc::channel(self.size);
        let namespace = self.namespace.clone();
        let build_closure: Arc<dyn Fn() -> _ + Send + Sync> = match &namespace {
            Some(ns) => {
                let ns = ns.clone();
                Arc::new(move || {
                    let ns = ns.clone();
                    Box::pin(async move {
                        nlink::netlink::namespace::connection_for(&ns)
                    }) as _
                })
            }
            None => Arc::new(|| {
                Box::pin(async { Connection::<P>::new() }) as _
            }),
        };
        // Seed the pool with `size` Connections.
        for _ in 0..self.size {
            let conn = (build_closure)().await?;
            tx.send(conn).await.expect("freshly created channel");
        }
        Ok(ConnectionPool {
            inner: Arc::new(PoolInner {
                available: tx,
                receiver: Mutex::new(rx),
                namespace,
                size: self.size,
                acquire_timeout: self.acquire_timeout,
                build: build_closure,
            }),
        })
    }
}

// For GENL (async-constructible) protocols:
impl<P> ConnectionPoolBuilder<P>
where
    P: ProtocolState + sealed::AsyncConstructible + 'static,
{
    pub async fn build(self) -> Result<ConnectionPool<P>> {
        // Same shape, but build closure uses Connection::new_async.
        // ... (parallel impl to the sync one above)
    }
}
```

The sealed-trait split from Plan 148 §4.5 carries forward — sync
constructible protocols and GENL protocols both get
`ConnectionPoolBuilder<P>`, with the right constructor selected by
the bound. Plan 148 §4.5 is a hard dependency.

### 4.4 Namespace-aware convenience

```rust
impl<P: ProtocolState + sealed::SyncConstructible + 'static> ConnectionPool<P> {
    /// Convenience: build a pool bound to a named network namespace.
    /// Equivalent to `ConnectionPoolBuilder::new().namespace(ns).build()`.
    pub async fn for_namespace(
        ns: impl Into<String>,
        size: usize,
    ) -> Result<Self> {
        ConnectionPoolBuilder::new()
            .namespace(ns)
            .size(size)
            .build()
            .await
    }
}
```

For multi-namespace consumers (CNI-plugin-shaped use cases) this
makes the per-namespace pool construction one line.

### 4.5 Error variants

Extend `crate::Error` with two new variants under `#[non_exhaustive]`:

```rust
pub enum Error {
    // ... existing ...
    #[error("connection pool exhausted; {size} connections all busy, \
             waited {timeout:?}")]
    PoolExhausted {
        size: usize,
        timeout: Duration,
    },
    #[error("connection pool is closed (all clones dropped)")]
    PoolClosed,
}

impl Error {
    pub fn is_pool_exhausted(&self) -> bool {
        matches!(self, Error::PoolExhausted { .. })
    }
    pub fn is_pool_closed(&self) -> bool {
        matches!(self, Error::PoolClosed)
    }
}
```

Per the project's `is_X()` predicate convention from
CLAUDE.md.

### 4.6 Documentation

The recipe `docs/recipes/connection-pool.md` covers:

1. When to use a pool vs a single Connection (rule of thumb: if
   you'd otherwise hand-roll multiple Connections to issue
   parallel work, use a pool).
2. The single-flight-per-Connection invariant carries through —
   the pool gives you concurrent requests by handing out
   different Connections, not by multiplexing one.
3. Sizing guidance:
   - I/O-bound bursty: pool size = `2 × num_cpus`
   - Steady-state: pool size = max-expected-concurrent-requests
   - Memory cost: each Connection has its own fd + (with Plan 158
     `syscall_batch` enabled) ~1 MB recv buffer
4. Multi-namespace pattern: one `Arc<ConnectionPool<Route>>` per
   namespace; share across all tasks that touch that namespace.
5. Drop semantics: dropping the last `Arc<ConnectionPool>` closes
   all underlying sockets. Outstanding `PooledConnection`s remain
   valid until their own drop.
6. Cooperation with Plan 151's `events_with_resync` — the resync
   `dump_conn` can be drawn from the pool.

## 8. Risks

- **Health-check is a stub**: real fd polling would add ~10 LOC
  and a per-acquire `poll()` syscall. Defer to 0.17 — the stub
  works for the common case (sockets stay healthy). The
  `invalidate()` escape hatch handles the rare case explicitly.
- **`Mutex<Receiver>` contention**: at very high acquire rates
  (>10k/s) the mutex could become contended. Solution if it
  surfaces: split receiver into N shards, hash-pick per acquire.
  Not implementing in 0.16 — premature optimization. Document the
  contention model: "acquire is microseconds-level; if you're
  doing 100k acquires/sec, you have a bigger problem."
- **Builder-closure ownership**: `Arc<dyn Fn() -> BoxFuture<...>>`
  is mildly awkward. Could be a trait `ConnectionFactory<P>` with
  one method; opted for the closure form because the namespace
  case captures one `String` and that's all the abstraction
  needed. If the closure shape grows (e.g., for the GENL family
  cache feature), promote to a trait then.
- **Sealed-trait bounds + GENL variant**: depends on Plan 148
  §4.5 having shipped. Master plan §7 sequencing puts 148 before
  159; if 148 §4.5 slips, this plan ships sync-only and gets the
  GENL variant in a follow-up PR. The pool works fine for
  rtnetlink / netfilter / xfrm regardless.
- **`num_cpus` as a dep**: tiny dep, but a dep. Alternative:
  `std::thread::available_parallelism()` (stable 1.59+). **Use the
  std version** — no new dep, exact same behavior. Plan revised
  in §4.3 default: `available_parallelism().map(|n| n.get()).unwrap_or(4)`.
- **Pool size sanity bounds**: clamp to ≥1 (size 0 is a
  deadlock). Upper bound: don't clamp — caller is responsible. A
  pool of 10000 is silly but not nlink's job to refuse.

## 9. Out-of-scope follow-ups

- **Smart health-checking** — fd-poll-based, kernel-aware. 0.17
  with its own design pass.
- **Adaptive pool sizing** — grow/shrink based on saturation
  metrics. Demand-gated.
- **GENL family ID cache shared across pool** — for GENL pools
  where every Connection in the pool resolves the same family,
  the resolution syscall could be amortized. Plan-154-aware
  optimization; defer.
- **Pool-aware `events_with_resync`** — Plan 151's resync helper
  takes a `dump_conn`; users already pass `pool.acquire().await?`
  for that. No special integration needed; documenting the
  pattern in the recipe is enough.
- **True NlRouter-style fanout** — 0.17 architectural project per
  master plan §4 item 6. Pool is the bridge.

End of plan.
