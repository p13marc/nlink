## 4. Approach — per fix

### 4.1 CRITICAL — `socket.rs:138` namespace-restore swallow

#### Diagnosis

```rust
// crates/nlink/src/netlink/socket.rs:114-145
pub fn new_in_namespace(protocol: Protocol, ns_fd: RawFd) -> Result<Self> {
    let current_ns = File::open("/proc/thread-self/ns/net")?;
    let current_ns_fd = current_ns.as_raw_fd();

    // Switch to target namespace.
    let ret = unsafe { libc::setns(ns_fd, libc::CLONE_NEWNET) };
    if ret < 0 { return Err(...); }

    // Create the socket (in the target namespace — correct).
    let result = Self::create_socket(protocol);

    // Restore — BUG: failure here doesn't propagate.
    let restore_ret = unsafe { libc::setns(current_ns_fd, libc::CLONE_NEWNET) };
    if restore_ret < 0 {
        eprintln!(           // ← writes to stderr, library shouldn't
            "warning: failed to restore original namespace: {}",
            std::io::Error::last_os_error()
        );
    }

    result    // ← still returns the socket; calling thread is now in target ns
}
```

When `restore` fails:
- The socket is fine — it lives in the target namespace, as designed.
- The **calling thread** is stuck in the target namespace.
- The next `*_by_name` call from this thread reads `/sys/class/net/`
  from the wrong namespace.
- Under tokio multi-thread runtime, another task scheduled on this
  thread also runs in the wrong namespace.
- The caller has zero signal beyond a stderr line.

#### Fix

Return an error variant that says "socket was created but the thread is
now in the wrong netns; caller must handle." Add a new variant to
`Error` (or extend an existing one with context):

```rust
// crates/nlink/src/netlink/error.rs
#[derive(thiserror::Error, Debug)]
#[non_exhaustive]
pub enum Error {
    // ... existing variants ...
    #[error("netns restore failed after socket creation; thread is stuck \
             in target netns (source: {source})")]
    NamespaceRestoreFailed {
        #[source]
        source: io::Error,
    },
}
```

Update `socket.rs:114-145`:

```rust
let restore_ret = unsafe { libc::setns(current_ns_fd, libc::CLONE_NEWNET) };
if restore_ret < 0 {
    let source = std::io::Error::last_os_error();
    // The socket itself is valid in the target namespace, but the
    // calling thread is now stuck there. Returning an error gives
    // the caller a chance to log, retry, or abort. Dropping the
    // result here also drops the (possibly successful) socket,
    // which the caller can't use safely anyway since they can't
    // rely on thread context.
    return Err(Error::NamespaceRestoreFailed { source });
}
result
```

#### Test

A regression test in `tests/integration/namespace_spawn.rs`:

```rust
#[tokio::test]
async fn netns_restore_failure_propagates_as_error() -> nlink::Result<()> {
    nlink::require_root!();
    nlink::require_module!("dummy");

    // Set up a namespace that we *can* enter.
    let ns = TestNamespace::new("netns-restore-fail")?;
    let ns_fd = std::fs::File::open(format!("/var/run/netns/{}", ns.name()))?;

    // Close the parent-ns fd before calling new_in_namespace so the
    // restore step fails. This is a contrived setup but reproduces
    // the failure path. (TODO: better simulation — likely needs to
    // hold the parent ns fd open and close it from another task.)
    // ...
    // Assertion: result is Err(Error::NamespaceRestoreFailed { .. }).
}
```

If reliable simulation is hard, ship the fix without a CI regression
test and gate behind a manual `--ignored` test instead. The fix is still
worth it; the regression test is a nice-to-have.

#### Effort

30 min for the fix + tests. Up to another 30 min if the regression test
setup proves tricky.

### 4.2 LOW (verify-first) — `route.rs:824-882` `RTA_METRICS` omitted on strict delete

#### Diagnosis

`Ipv4Route::write_add` (`route.rs:738-743`) writes `RTA_METRICS` when
`self.metrics.is_some()`. `Ipv4Route::write_delete_with_interfaces`
(`route.rs:824-882`) doesn't. Symmetry violation — `write_add` writes
N fields, `write_delete_with_interfaces` writes N-1. The kernel may or
may not key on `RTA_METRICS` for delete; in practice it likely doesn't.

#### Action

Two-step:

1. **Verify in kernel source first**: read `fib_table_delete` in
   `net/ipv4/fib_trie.c` (or for IPv6, `net/ipv6/route.c`
   `ip6_route_del`). Confirm whether `RTA_METRICS` is part of the
   match key. Likely answer: not on the IPv4 side (metrics are
   advisory at lookup time, not part of the route identity), but
   non-zero chance of a kernel-version surprise.
2. **If kernel ignores**: add a comment to
   `write_delete_with_interfaces` explaining why metrics are omitted
   (cite the kernel function), and close the loop. No code change.
3. **If kernel does match on it**: add the `RTA_METRICS` write to
   `write_delete_with_interfaces` (and the IPv6 variant), and add a
   regression test that adds a route with metrics, then deletes it
   with the same metrics.

Estimate the "kernel ignores" case at 95% likely. The verify step is
cheap insurance.

#### Effort

30 min (kernel read + comment or fix).

### 4.3 MEDIUM — `bitset.rs:283` clone in `sort_by_key`

#### Diagnosis

```rust
// crates/nlink/src/netlink/genl/ethtool/bitset.rs:283 (current)
let mut entries: Vec<(&u32, &String)> = self.names.iter().collect();
entries.sort_by_key(|(_, name)| (*name).clone());   // ← allocates per name
```

Clippy's `unnecessary_sort_by` lint pushed me toward `sort_by_key`, but
the auto-suggestion picked a key form that allocates. Real fix: use
`&str` reference as the key.

#### Fix

```rust
let mut entries: Vec<(&u32, &String)> = self.names.iter().collect();
entries.sort_unstable_by_key(|(_, name)| name.as_str());
```

`sort_unstable` is appropriate here — bitset name ordering is for
deterministic test fixtures, not for stability semantics.

#### Test

The existing `write_to_roundtrips_through_parse` test in
`bitset.rs::tests` already exercises this path. No new test needed; the
fix is verified by the existing one passing.

#### Effort

5 min.

### 4.4 HIGH — `config/diff.rs:434` qdisc parameter comparison

#### Diagnosis

```rust
// crates/nlink/src/netlink/config/diff.rs:430-440 (current)
if existing.kind() != Some(declared.kind()) {
    self.qdiscs_to_replace.push(...);
} else {
    // TODO: Could check detailed parameters here
}
```

For declarative `NetworkConfig::apply()` users, this means changing an
HTB's `rate` parameter from 100mbit to 1gbit produces zero kernel
operations — the diff sees "both sides HTB" and emits nothing. The
config silently doesn't take effect.

#### Fix

Two approaches, in increasing fidelity:

**Approach A (byte-compare, simple)**: serialize both `write_options`
outputs into ephemeral `MessageBuilder` buffers and `memcmp`. If
different, push to `qdiscs_to_replace`. Catches every parameter change
but requires `QdiscConfig::write_options` to be deterministic (already
is). ~30 LOC.

**Approach B (per-field, brittle)**: enumerate each typed qdisc config's
fields, compare. Requires per-kind code. Don't.

Approach A is the right pick. The lib's `QdiscConfig::write_options`
trait method already produces canonical bytes; reuse it.

```rust
// In config/diff.rs, around line 430:
fn qdisc_params_equal<Q: QdiscConfig>(a: &Q, b: &Q) -> bool {
    let mut buf_a = MessageBuilder::scratch();   // helper if not present
    let mut buf_b = MessageBuilder::scratch();
    a.write_options(&mut buf_a).ok();
    b.write_options(&mut buf_b).ok();
    buf_a.as_bytes() == buf_b.as_bytes()
}
```

The hitch: `existing` qdisc comes from kernel dump (raw attribute
bytes), not from a typed config. So we'd compare the existing
attributes blob against the declared config's serialized options. The
existing blob may include kernel-added attributes that the declared
side doesn't — false-positive risk. Worth a comment.

Possible simpler heuristic: compare a per-kind subset of attributes
(rate, ceil, burst for HTB; delay, jitter for netem; etc.). More
brittle but tighter signal.

**Recommendation**: ship Approach A, accept the false-positive risk
(it triggers harmless re-applies, not data loss). Document the
limitation in the diff module's docstring. If users hit churn from
false positives, refine in 0.17.

#### Test

`tests/integration/config.rs`: add a test that creates an HTB qdisc
with rate=100mbit, then re-applies with rate=1gbit, and verifies the
diff reports a `qdiscs_to_replace` entry. Reverting this fix should
make the test fail.

#### Effort

2 hours including the test.

## 8. Risks

- **Phase 1 regression-test contrivedness**: simulating `setns`
  restore-failure in a test is hard without root + manipulating ns
  fds. Acceptable to ship the fix without a green regression test, as
  long as the fix itself is reviewed (the mechanism is obvious from
  the diff). The fix is correct by construction.
- **Phase 2 false positives**: byte-compare may trigger re-apply when
  the existing dump has kernel-padded attributes the declared config
  doesn't. Symptom is "diff says replace when it could no-op"; this
  is harmless (the replace is idempotent), just noisy. Document.
- **Phase 4 kernel-source dive**: budget extending to 1 hour if the
  fib_table_delete trail goes deep. If so, defer to a follow-up
  commit during the cycle rather than blocking Phase 4.
