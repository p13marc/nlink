# Plan 154 — `nlink-macros` proc-macro crate for 0.16

The strategic centerpiece of the 0.16 cycle. Closes the single
biggest gap surfaced by the neli comparison.

## 0. Why this matters

[`COMPARISON_NELI.md`](../COMPARISON_NELI.md) §5 puts it bluntly:

> **Auto-derived wire codecs via proc macros.** `#[derive(Size,
> ToBytes, FromBytesWithInput, Header)]` plus `#[neli::neli_enum(...)]`
> lets a user define a new GENL family in ~20 lines (see
> `examples/nl80211.rs`). nlink's `zerocopy` approach is leaner at
> runtime but requires more hand-written code per new message type.
> **This is genuinely nicer for downstream crate authors.**

The downstream evidence is in §7:

> Real reverse deps (24 total per crates.io): `local-ip-address`,
> `socketcan`, `wireguard-uapi`, `neli-wifi`, `nl80211-ng`,
> `nl80211`, `netlink_wi`, `conntrack`, `i3status-rs`, `i3stat`,
> `foundations` / `rustfoundry` (Cloudflare's service-foundations
> fork), ...

Most of those ecosystem crates exist because neli's derives made
them cheap to build. **Without an equivalent in nlink, every future
GENL family that lands in the kernel goes to neli's ecosystem, not
nlink's.** This is the single most consequential strategic gap.

This plan adds `nlink-macros`: a proc-macro crate providing
`#[derive(GenlMessage)]`, `#[derive(GenlCommand)]`,
`#[derive(GenlAttribute)]` (and a few supporting macros) that let a
downstream crate define a new GENL family in roughly the same
~20 lines neli requires.

## 1. Summary

New crate `nlink-macros` in the workspace, providing proc-macro
derives:

| Derive / attr | Purpose |
|---|---|
| `#[derive(GenlMessage)]` | Serialize / deserialize a GENL message struct (header + body + attrs) |
| `#[derive(GenlCommand)]` + `#[genl_command(repr = u8)]` | Typed command enum, like neli's `#[neli::neli_enum(serialized_type = "u8")]` |
| `#[derive(GenlAttribute)]` + `#[genl_attribute(repr = u16)]` | Typed attribute-kind enum |
| `#[derive(GenlEnum)]` + `#[genl_enum(repr = "u32", base = 1)]` | Typed value enum (enum encoded *inside* an attribute payload, not as the attribute kind itself). Generates `From<MyEnum> for u32` + `TryFrom<u32> for MyEnum`. The `base` annotation handles the 1-based-vs-0-based kernel convention — kernels are usually 1-based but 0-based variants exist (e.g. `DPLL_FEATURE_STATE_*`). |
| `#[derive(NetlinkAttrs)]` | Auto-derive `ToAttrs` / `FromAttrs` for attribute-bearing payload structs that aren't full GENL messages (used for nested attribute groups — see §4.6) |
| `#[genl_family(name = "my_family", version = 1)]` | Crate-level annotation that links the typed enums to a `Connection<MyFamily>` marker |

Plus a runtime support shim in `nlink::macros::__rt` that the
generated code calls into. Most of the runtime work uses primitives
already exposed by nlink (`MessageBuilder`, `AttrIter`, `NlMsgHdr`).

**Total new public surface**: ~5 derive macros + 1 attr macro + a
small `nlink::macros` re-export module.

**Total new code**: ~800-1200 LOC across `nlink-macros/src/` plus
~200 LOC of runtime support in `nlink/src/macros/`.

## 2. Goals

By end of plan:

- A downstream crate can define a new GENL family in ~20 lines of
  declarative code (per the success criterion: matches neli's
  `nl80211.rs` example length).
- The generated code is `zerocopy`-friendly where possible —
  attribute-bearing structs use `MessageBuilder` for emission and
  `AttrIter` for parsing, matching nlink's existing patterns.
- Existing manual implementations (`Connection<Wireguard>`,
  `Connection<Macsec>`, etc.) continue to work unchanged. The
  macros are an **alternative path**, not a replacement.
- A worked example in `crates/nlink/examples/macros/`: define a
  fictional or real-but-small GENL family from scratch using the
  macros, exercise it end-to-end.
- Documentation: `docs/recipes/define-your-own-genl-family.md` and
  `crates/nlink-macros/README.md` showing the canonical pattern.

## 3. Scope

### In

- New `nlink-macros` proc-macro crate in the workspace.
- The five derives/attrs in §1.
- A runtime support module (`nlink::macros::__rt`) for the
  generated code to call into. Documented as "internal API; do
  not depend on directly."
- `nlink` re-exports the macros from `nlink_macros::*` for
  convenience: users can write `use nlink::macros::*;` instead of
  pulling in `nlink-macros` directly.
- Worked example: `crates/nlink/examples/macros/define_taskstats.rs`
  (taskstats is small, frozen, real — perfect for the showcase).
- Doc-comments on every macro with embedded example snippets.
- Recipe: `docs/recipes/define-your-own-genl-family.md`.
- README for `nlink-macros`.
- CHANGELOG entry under Added.

### Out (explicitly)

- Backwards-rewriting existing in-tree GENL families to use the
  macros. They work; rewriting just for the showcase would be
  busywork. Leave them as the "this is what the macros generate
  for you" reference points.
- Macros for non-GENL families (rtnetlink, netfilter, xfrm). These
  protocols are family-singletons; the in-tree implementations
  cover them. Macros wouldn't add ergonomic value.
- A `#[derive(Builder)]` for the typed-config structs (HtbQdiscConfig
  etc.) — different problem, different solution. STRATEGIC_ANALYSIS
  §3.2 lists this as a separate effort.
- Schema-driven codegen (YNL specs → Rust) — major separate plan
  ([STRATEGIC_ANALYSIS §4.2](../STRATEGIC_ANALYSIS.md#42-the-ynl-bet-—-biggest-architectural-decision-left)).
  Reserve for 0.17 or beyond. The macros in this plan are
  human-author-facing; YNL codegen is generated-Rust-from-YAML.

## 4. Design

### 4.1 Comparison target — what neli does

A new GENL family in neli looks like:

```rust
#[neli::neli_enum(serialized_type = "u8")]
pub enum MyCmd {
    Unspec = 0,
    Get    = 1,
    Set    = 2,
}

#[neli::neli_enum(serialized_type = "u16")]
pub enum MyAttr {
    Unspec  = 0,
    Name    = 1,
    Value   = 2,
}

// Use:
let (sock, _) = NlRouter::connect(NlFamily::Generic, None, Groups::empty())?;
let resp = sock.send::<_, _, NlTypeWrapper, Genlmsghdr<MyCmd, MyAttr>>(
    GenlId::Ctrl, NlmF::DUMP,
    NlPayload::Payload(
        GenlmsghdrBuilder::default()
            .cmd(MyCmd::Get).version(1)
            .attrs(GenlBuffer::<u16, Buffer>::new())
            .build()?,
    ),
)?;
```

The `#[neli_enum]` derive handles serialization. The
`Genlmsghdr<Cmd, Attr>` generic ties the typed command and attribute
enums together. The dispatch is generic over the message type.

### 4.2 nlink target — what we want users to write

```rust
use nlink::macros::*;

// Step 1: define the family marker.
#[genl_family(name = "my_family", version = 1)]
pub struct MyFamily;

// Step 2: define the command enum.
#[derive(GenlCommand)]
#[genl_command(repr = "u8")]
pub enum MyCmd {
    Unspec = 0,
    Get    = 1,
    Set    = 2,
}

// Step 3: define the attribute enum.
#[derive(GenlAttribute)]
#[genl_attribute(repr = "u16")]
pub enum MyAttr {
    Unspec = 0,
    Name   = 1,
    Value  = 2,
}

// Step 4 (optional): define a typed payload struct.
#[derive(GenlMessage)]
#[genl_message(cmd_enum = MyCmd, attr_enum = MyAttr)]
pub struct GetRequest {
    #[genl_attr(MyAttr::Name)]
    pub name: String,
}

// Use:
let conn = Connection::<MyFamily>::new_async().await?;
let response = conn.send_typed(GetRequest {
    name: "foo".to_string(),
}).await?;
```

The `#[genl_family]` attribute creates a marker struct that
implements `ProtocolState` + `AsyncProtocolInit` with family-ID
caching. The `Connection<MyFamily>` plugs into the existing
per-protocol pattern.

The `#[derive(GenlMessage)]` generates:
- `impl GenlMessage for GetRequest`
- `fn to_bytes(&self, builder: &mut MessageBuilder)` — serialize
- `fn from_bytes(bytes: &[u8]) -> Result<Self>` — parse
- `fn cmd() -> u8` — the command id
- Field-to-attribute mapping via the `#[genl_attr(...)]` markers

`Connection<MyFamily>::send_typed<M: GenlMessage>` is a generic
dispatch path added to nlink that works for any `GenlMessage` impl.

**Line count**: the snippet above is 22 lines plus imports. Matches
neli's pattern.

### 4.3 Generated runtime calls

The macros emit code that calls into a small runtime module. **Type
selection happens at macro-expand time**: the derive inspects each
field's Rust type and picks the matching helper. A
`#[genl_attr(MyAttr::Port)]` field of type `u32` expands to
`__rt::emit_u32_attr(...)`; the same attribute marker on a `String`
field expands to `__rt::emit_string_attr(...)`. Type mismatches
between the field type and the helper signature are caught at
**compile time**, not at runtime parse time.

```rust
// crates/nlink/src/macros/mod.rs

#[doc(hidden)]
pub mod __rt {
    pub use crate::netlink::{
        builder::MessageBuilder,
        attr::{AttrIter, NLA_F_NESTED},
        message::NlMsgHdr,
        error::{Error, Result},
    };

    /// Generated code calls this to emit a u32 attribute.
    pub fn emit_u32_attr(builder: &mut MessageBuilder, attr_type: u16, value: u32) {
        builder.append_attr_u32(attr_type, value);
    }
    pub fn emit_string_attr(builder: &mut MessageBuilder, attr_type: u16, value: &str) {
        builder.append_attr_str(attr_type, value);
    }
    /// ... helpers for each attribute kind ...

    /// Generated code calls this to parse a u32 attribute.
    pub fn parse_u32_attr(payload: &[u8]) -> Result<u32> {
        if payload.len() < 4 {
            return Err(Error::Truncated { expected: 4, actual: payload.len() });
        }
        Ok(u32::from_ne_bytes(payload[..4].try_into().unwrap()))
    }
    /// ... etc ...
}
```

Kept stable but documented as internal. The actual user-facing
surface is just the derives.

**Supported field types** (each maps to one `emit_*` / `parse_*`
pair):

| Rust type | Wire encoding | Helper |
|---|---|---|
| `u8` / `u16` / `u32` / `u64` | native-endian u-attr | `emit_u{8,16,32,64}_attr` |
| `i8` / `i16` / `i32` / `i64` | native-endian s-attr | `emit_i{8,16,32,64}_attr` |
| `String` / `&str` | NUL-terminated string | `emit_string_attr` |
| `Vec<u8>` / `&[u8]` | opaque binary | `emit_bytes_attr` |
| `Ipv4Addr` / `Ipv6Addr` | 4 / 16 byte network-order | `emit_ipv{4,6}_attr` |
| `MacAddr` | 6 byte | `emit_mac_attr` |
| `bool` | u8 (0/1) | `emit_u8_attr` (with cast) |
| `Option<T>` | attribute omitted when None | each helper has Option overload |
| nested `T: GenlMessage` | NLA_F_NESTED + recursive encode | `emit_nested_attr` |

A field of any *other* type is a compile error pointing at the field
with `"unsupported attribute type; supported types are u8/u16/u32/
u64, i8/.../, String, Vec<u8>, Ipv4Addr, Ipv6Addr, MacAddr, bool,
Option<T>, or T: GenlMessage"`.

This is the central type-safety guarantee: **the wire encoding for
each attribute is selected at compile time from the field's Rust
type**, not at runtime from a tag. Mismatched repr is a compile
error; missing-attribute logic (`Option`) is checked exhaustively.

### 4.4 GENL family registration

`#[genl_family(name = "my_family", version = 1)]` generates:

```rust
pub struct MyFamily {
    family_id: u16,  // resolved at Connection::new_async() time
    state: GenlFamilyCache,
}

impl ProtocolState for MyFamily {
    const PROTOCOL: Protocol = Protocol::Generic;
    // ...
}

impl AsyncProtocolInit for MyFamily {
    async fn resolve(socket: &NetlinkSocket) -> Result<Self> {
        let family_id = resolve_genl_family(socket, "my_family").await?;
        Ok(Self { family_id, state: GenlFamilyCache::default() })
    }
}

// Coordinated with Plan 148 §4.5 (sealed constructor) — the macro
// also impls the sealed AsyncConstructible trait so the user gets
// Connection::<MyFamily>::new_async().await as the canonical
// constructor, and Connection::<MyFamily>::new() is a compile error
// (consistent with the in-tree GENL families).
impl nlink::__sealed::AsyncConstructible for MyFamily {}
```

This integrates the family marker with nlink's existing GENL
family-resolution pattern. No new infrastructure needed —
`resolve_genl_family` already exists in nlink for built-in
families.

**Sequencing dependency on Plan 148 §4.5**: this macro generates
`impl nlink::__sealed::AsyncConstructible for MyFamily {}`, which
requires Plan 148's sealed-constructor work to have landed first.
Master plan §7 already sequences Plan 148 before Plan 154, so the
dependency is satisfied. If Plan 148 slips, this macro can ship
without the sealed-impl line and gain it later (the family marker
still works; it just allows the broken `new()` form until 148
lands).

### 4.6 `#[derive(NetlinkAttrs)]` for nested attribute groups

For attribute groups that are encoded as `NLA_F_NESTED` payloads
(e.g. the per-policy nested block in nl80211), the user defines a
plain struct and derives `NetlinkAttrs`:

```rust
#[derive(NetlinkAttrs)]
#[netlink_attrs(attr_enum = StaInfoAttr)]
pub struct StaInfo {
    #[genl_attr(StaInfoAttr::RxBytes)] pub rx_bytes: u64,
    #[genl_attr(StaInfoAttr::TxBytes)] pub tx_bytes: u64,
    #[genl_attr(StaInfoAttr::Signal)]  pub signal: Option<i8>,
}
```

The derive generates `ToAttrs` (writes the fields into a
caller-supplied `MessageBuilder` already positioned inside the
nested attribute) and `FromAttrs` (parses an `AttrIter` into Self).
It is **strictly weaker** than `GenlMessage` — there's no command
or header, just the field-to-attribute mapping. Used as the
building block for `GenlMessage`'s nested-attribute support; also
usable standalone for parsing nested blobs returned from raw GENL
calls.

### 4.7 Compatibility with existing in-tree GENL families

The in-tree families (`Wireguard`, `Macsec`, `Mptcp`, `Ethtool`,
`Nl80211`, `Devlink`) keep their hand-written implementations. They
predate the macros and are battle-tested; rewriting them just to
showcase the macros would be busywork.

The recipe + example point at them as **reference implementations**:
"this is what the macros generate; if you want to see the wire
format up close, read the manual ones."

For Plan 153's new GENL family (`net_shaper`): hand-write, matching
the existing in-tree pattern. Keeps Plan 153 reviewable on its own
without dragging the macro design into the same PR.

**Plan 156 is the in-tree dogfood.** The DPLL GENL family is added
*using* the macros — ~130 lines of macro-derived declaration for
the full family (12 commands, 14 device attrs, 31 pin attrs, 10
value enums). Plan 156 is intentionally sequenced immediately after
Plan 154 in master §7 so any macro-design issue surfaces on a real
subsystem before Plan 152's `aya` co-demo also depends on the
macros. See Plan 156 §7b for the `#[derive(GenlEnum)]` add-on
required to support DPLL's value enums.

## 5. Implementation phases

### Phase 1: scaffold + `GenlCommand` derive

1. Create `crates/nlink-macros/` workspace member (proc-macro lib).
2. Add to `[workspace.members]`.
3. Set `publish = true` (this crate IS publishable; users depend on
   it transitively via nlink's re-export).
4. Implement `#[derive(GenlCommand)]` + `#[genl_command(repr = u8|u16)]`.
5. Generate: `impl From<MyCmd> for u8` + `impl TryFrom<u8> for MyCmd`.
6. Unit tests via `trybuild` (compile-pass + compile-fail cases).

### Phase 2: `GenlAttribute` derive

7. Implement `#[derive(GenlAttribute)]` + `#[genl_attribute(repr = u16)]`.
8. Same shape as `GenlCommand` — typed kind enum.
9. Unit tests.

### Phase 3: `GenlMessage` derive

10. Implement `#[derive(GenlMessage)]` — the biggest derive.
11. Parse `#[genl_attr(MyAttr::Name)]` field markers.
12. Generate `to_bytes` + `from_bytes` + `cmd()`.
13. Type-resolution for common Rust types (u8, u16, u32, u64,
    String, Vec<u8>, IpAddr, etc.) — map each to the right
    emit/parse helper in the runtime.
14. Unit tests + trybuild.

### Phase 4: `#[genl_family]` attribute macro

15. Implement the family-marker generator.
16. Wire into `ProtocolState` + `AsyncProtocolInit` traits from
    nlink core.
17. Unit tests.

### Phase 5: `Connection<F>::send_typed<M: GenlMessage>`

18. Add the generic dispatch path on `Connection<F>` where `F:
    GenlFamily` (a new sealed marker trait that the `#[genl_family]`
    macro impls). Signature:

    ```rust
    impl<F: GenlFamily> Connection<F> {
        /// Send a typed GENL message and parse the typed response.
        /// The response type R is the message author's choice;
        /// typically a struct deriving `GenlMessage`. For requests
        /// that expect only an ACK use `R = ()`.
        pub async fn send_typed<M, R>(&self, msg: M) -> Result<R>
        where
            M: GenlMessage,
            R: GenlMessage + Default;

        /// Stream a dump response as typed messages.
        pub fn dump_typed_stream<M, R>(&self, request: M)
            -> impl Stream<Item = Result<R>> + Send + '_
        where
            M: GenlMessage,
            R: GenlMessage;
    }
    ```

    The `dump_typed_stream` builds on Plan 149's `dump_stream`
    primitive — the cross-plan integration point.
19. Wire to nlink's existing send_request / send_dump.
20. Unit tests (with mocked socket).

### Phase 6: worked example + recipe

21. `crates/nlink/examples/macros/define_taskstats.rs` — implement
    a subset of taskstats using the macros.
22. `docs/recipes/define-your-own-genl-family.md` — narrative
    walkthrough.
23. README for `nlink-macros`.

### Phase 7: nlink re-export + CHANGELOG

24. `crates/nlink/src/macros/mod.rs` — re-export
    `nlink_macros::*`. Behind no feature flag (the macro deps are
    light).
25. Update `crates/nlink/Cargo.toml` to depend on `nlink-macros`
    with version pinning (`nlink-macros = { path = "../nlink-macros",
    version = "0.16.0" }`).
26. CHANGELOG entry under Added — prominent placement, this is the
    headline 0.16 feature for downstream crate authors.

## 8. Risks

- **API design takes iteration.** The first macro shape may not
  feel right when you start writing the example. Budget at least
  one round of "redesign the attr-mapping syntax" before locking
  the public surface. Mitigation: write the
  `define_taskstats.rs` example **in parallel with** developing
  the derives, not after — the example is the design driver.
- **`syn` / `quote` boilerplate**: proc-macro crates accumulate
  helper functions for AST manipulation. The first 200 LOC are
  setup; real productivity kicks in after that. Don't be surprised.
- **Generated-code error messages**: when users misuse the
  derive, the compiler error should point at the user's source,
  not at the generated tokens. Use `syn::Error::new_spanned` for
  every error path; don't `panic!` in proc-macros.
- **`trybuild` flakiness**: trybuild compile-fail tests can be
  fragile when stable Rust's error messages change. Pin them to
  specific output via the `--overwrite` pattern; accept that
  they'll need re-baselining every 2-3 Rust releases.
- **Crate split adds release coordination**: `nlink-macros` ships
  before `nlink` in any cycle (nlink depends on it). Document the
  publish order in the release checklist.
- **Doc complexity**: explaining what the macros generate is
  harder than explaining a regular API. Counter with the example +
  recipe + "here's the manually-written `Connection<Wireguard>`
  for comparison" reference.
