# Plan 149 — streaming dump API for 0.16

The single biggest scalability cliff in nlink today: every dump
allocates a `Vec<Vec<u8>>` for the full kernel response before any
parsing happens. On a BGP router with 1M routes, `get_routes().await`
allocates and holds gigabytes before yielding the first row.

This plan adds a streaming alternative — `stream_routes()`,
`stream_links()`, `stream_neighbors()`, `stream_fdb()` — that yields
typed messages one at a time as they arrive. The eager `get_*` variants
stay in tree unchanged for convenience and existing-code compatibility.

## 1. Summary

Add `Stream<Item = Result<T>>` variants to the four dump APIs where
"large result sets are realistic." The internal `send_dump_inner` (in
`connection.rs`) already iterates frame-by-frame; the change is at the
boundary — return a stream instead of materializing.

Total surface added: 4 new public methods. ~250 LOC including
infrastructure. Zero changes to existing methods.

## 2. Goals

- Add `Connection<Route>::stream_routes()`, `stream_links()`,
  `stream_neighbors()`, `stream_fdb()`.
- Add a generic `dump_stream<T: FromNetlink>(msg_type, header)` building
  block on `Connection<P>` that the four typed methods adapt.
- Users querying large dumps can process N messages with O(1) memory.
- Backpressure: kernel keeps sending, we read at the consumer's pace.
- Cancellation safety: dropping the stream mid-iteration cleanly
  closes out the dump (drains remaining frames into the socket buffer,
  then leaves them — the kernel terminates the dump on its own when
  the socket goes silent).
- Keep `get_routes()` / `get_links()` / `get_neighbors()` / `get_fdb()`
  unchanged for backward compat. Document the streaming alternative in
  their doc comments.

## 3. Scope

### In

- `Connection<P>::dump_stream<T>()` low-level generic.
- `Connection<Route>::stream_routes()` / `stream_links()` /
  `stream_neighbors()` / `stream_fdb()` typed wrappers.
- Doc-cross-references from existing `get_*` methods.
- Unit tests for the dump_stream state machine (no kernel needed).
- Integration tests for at least one subsystem (routes) under
  `tests/integration/route.rs`, gated on `lab` feature + `dummy`
  module.
- CHANGELOG entry.

### Out

- Streaming for other subsystems (TC qdiscs/classes/filters, addresses,
  rules). Could add later if a downstream user asks; for now the four
  in §2 are the BGP-router-scale paths.
- Streaming for typed config dumps (e.g. `dump_xfrm_sa`). Smaller
  result sets; not the user pain point.
- Backpressure control beyond what tokio's recv naturally provides.
  Real flow control would need `NETLINK_RX_RING` (out of scope).

## 4. Design

### 4.1 Low-level building block

MSRV is 1.95 (RPITIT stable since 1.75), so the typed wrappers
return `impl Stream<...> + Send + '_` rather than a boxed trait
object. The low-level generic still hands out `Pin<Box<dyn Stream>>`
because `async_stream::try_stream!` produces an opaque type that
must be boxed once for the macro to compose with `T: FromNetlink`
generics. The boxing is one heap-alloc per dump (negligible vs the
dump itself).

```rust
// crates/nlink/src/netlink/connection.rs

use futures::Stream;
use std::pin::Pin;

impl<P: ProtocolState> Connection<P> {
    /// Stream a dump response as it arrives, one typed message per
    /// `next().await`. The stream terminates on `NLMSG_DONE`.
    ///
    /// Compared to [`dump_typed`], this method does **not** buffer the
    /// full response. On large dumps (e.g. `get_routes` on a BGP
    /// router with 1M+ routes), this avoids materializing gigabytes
    /// of intermediate `Vec<Vec<u8>>`.
    ///
    /// # Cancellation
    ///
    /// Dropping the stream is safe. The kernel terminates the dump
    /// when no more frames are read; any in-flight frames sit in
    /// the kernel socket buffer briefly until they age out. The next
    /// request on the same connection will see them first (with
    /// stale sequence numbers, which we skip). For a clean cancel,
    /// drain to `NLMSG_DONE` before dropping.
    ///
    /// # Errors
    ///
    /// The stream yields `Err` on any individual message-parse failure
    /// but continues iteration. Network-level errors (socket closed,
    /// timeout) terminate the stream after yielding the error.
    pub fn dump_stream<T: FromNetlink>(
        &self,
        msg_type: u16,
    ) -> Pin<Box<dyn Stream<Item = Result<T>> + Send + '_>> {
        // Build the request, capture sequence, return a stream that
        // sends on first poll and recv-and-yields on subsequent polls.
        Box::pin(async_stream::try_stream! {
            let mut builder = dump_request(msg_type);
            let mut header_buf = Vec::new();
            T::write_dump_header(&mut header_buf);
            builder.append_bytes(&header_buf);
            let seq = self.socket.next_seq();
            builder.set_seq(seq);
            builder.set_pid(self.socket.pid());
            let msg = builder.finish();
            self.socket.send(&msg).await?;

            'outer: loop {
                let data = self.with_timeout(self.socket.recv_msg()).await?;
                for result in MessageIter::new(&data) {
                    let (header, payload) = result?;
                    if header.nlmsg_seq != seq { continue; }
                    if header.is_error() {
                        let err = NlMsgError::from_bytes(payload)?;
                        if !err.is_ack() {
                            yield Err(Error::from_errno(err.error))?;
                            break 'outer;
                        }
                    }
                    if header.is_done() {
                        break 'outer;
                    }
                    if payload.len() < NLMSG_HDRLEN {
                        continue;
                    }
                    let item_payload = &payload[NLMSG_HDRLEN..];
                    yield T::from_bytes(item_payload)?;
                }
            }
        })
    }
}
```

Two design choices:

1. **`Pin<Box<dyn Stream + Send + '_>>` return type** instead of
   `impl Stream`. The async generator (`async_stream::try_stream!`)
   produces an opaque type that's hard to name without RPITIT and
   often ends up needing `Pin<Box<...>>` anyway. Boxing is one
   heap-alloc per dump — negligible compared to a 1M-route dump.
2. **`async_stream` dependency**: well-maintained,
   `tokio-rs`-adjacent, lightweight. Alternative is a hand-rolled
   `Stream` impl with state-machine enum — ~80 extra LOC, no new
   dep. Pick `async_stream` for readability.

If the project objects to a new dep, switch to manual state machine.
Document the trade-off in the PR.

### 4.2 Typed wrappers

For each of routes / links / neighbors / fdb, expose a one-line
wrapper. Typed wrappers use `impl Stream<...>` to name the return
without the `Pin<Box<dyn ...>>` ceremony — the caller pins via
`futures::pin_mut!` or `Box::pin` as needed. This matches how
existing async methods on `Connection<Route>` return non-boxed
futures.

```rust
impl Connection<Route> {
    pub fn stream_routes(&self)
        -> impl Stream<Item = Result<RouteMessage>> + Send + '_
    {
        self.dump_stream(NlMsgType::RTM_GETROUTE)
    }

    pub fn stream_links(&self)
        -> impl Stream<Item = Result<LinkMessage>> + Send + '_
    {
        self.dump_stream(NlMsgType::RTM_GETLINK)
    }

    pub fn stream_neighbors(&self)
        -> impl Stream<Item = Result<NeighborMessage>> + Send + '_
    {
        self.dump_stream(NlMsgType::RTM_GETNEIGH)
    }

    pub fn stream_fdb(&self)
        -> impl Stream<Item = Result<NeighborMessage>> + Send + '_
    {
        // FDB uses RTM_GETNEIGH with NDM_FAMILY = AF_BRIDGE
        // (see existing get_fdb() impl for the header customization).
        self.dump_stream_with_header(...)
    }
}
```

FDB needs a custom header (sets `NdMsg.family = AF_BRIDGE`). Extend
`dump_stream` to take an optional header-customizer closure if needed,
or expose a parallel `dump_stream_with_header` variant. Look at the
existing `get_fdb` impl to see the customization shape.

The four typed wrappers delegate to the boxed low-level — the
`impl Stream` return is still backed by `Pin<Box<dyn Stream>>` under
the hood, but the type is named opaquely so callers don't have to
think about boxing. RPITIT (stabilized in 1.75; nlink MSRV is 1.95)
lets us hide the boxing behind an opaque type.

### 4.3 Existing `get_*` methods — unchanged

Doc comment additions only:

```rust
/// ... existing docs ...
///
/// For large dumps (thousands of routes / interfaces / neighbors),
/// prefer the streaming variant [`stream_routes`] which yields one
/// `RouteMessage` at a time without materializing the full response.
pub async fn get_routes(&self) -> Result<Vec<RouteMessage>> { ... }
```

The streaming variant is **strictly opt-in** — zero risk of breaking
callers.

## 8. Risks

- **`async_stream` dependency**: small, well-maintained, but adds a
  dep. Alternative manual `Stream` impl is ~80 LOC. Decide before PR
  opens. If maintenance is a concern, manual impl.
- **Lifetimes**: the returned `impl Stream + '_` borrows `&self` for
  the stream's lifetime. Users wanting an owned stream (to send across
  task boundaries) need to spawn a task that owns the connection and
  the stream together. Document this. The opaque return type means
  `+ Send` is part of the named bound — verify on construction that
  `async_stream::try_stream!` produces a `Send` future (it does as
  long as captured values are `Send`).
- **Single-flight constraint**: only one streaming dump can be active
  per connection at a time (same as the existing `send_dump`
  invariant). Document with `// Note: do not call concurrently on the
  same connection — sequence numbers will race.`
- **Cancellation hazard**: if a user drops the stream mid-dump and
  immediately issues another request on the same connection, the new
  request's sequence number is different so stale frames from the
  dropped dump are skipped (good). But the socket buffer may fill if
  the kernel emits faster than userspace drains. The dump engine
  pauses when the buffer is full (the kernel's natural
  backpressure), so this is benign. Worth a sentence in the doc
  comment.
- **FDB header customization**: if `dump_stream` doesn't support
  custom headers cleanly, FDB ships in a follow-up (or via a
  `dump_stream_with_header` variant). Don't block the other three on
  this.

## 9. Additional streaming surfaces folded in (formerly §9 "out of scope")

Per the master plan's "no deferral to 0.17" stance, the additional
streaming surfaces previously listed as follow-ups are now in this
plan:

### 9.1 Stream TC dumps

Add `stream_qdiscs_by_index(ifindex)`, `stream_classes_by_index(ifindex)`,
`stream_filters_by_index(ifindex, parent)`, `stream_actions(kind)`.

Smaller per-interface scale than routes (typical interface has <100
qdiscs+classes+filters), but a host with many interfaces × many
qdiscs each can still accumulate. Plus consistency — once `stream_*`
is the pattern, every dump should support it.

**Effort**: 3 hours (4 thin wrappers on `dump_stream`, mirroring §4.2).
**Files**: `crates/nlink/src/netlink/{tc,filter,action}.rs`.

### 9.2 Stream typed config dumps

Add streaming variants for:
- `stream_xfrm_sa()`, `stream_xfrm_sp()` (IPsec)
- `stream_conntrack()` (netfilter conntrack table — can be huge on
  busy gateways)
- `stream_nftables_rules(table, chain)` (firewalls with thousands
  of rules)

**Effort**: 4 hours (more wrappers + verifying the GENL-family /
netfilter dump-header customization works through `dump_stream`).
**Files**: `crates/nlink/src/netlink/{xfrm,netfilter}.rs` +
`crates/nlink/src/netlink/nftables/connection.rs`.

### 9.3 What's still genuinely out of scope

- `NETLINK_RX_RING`-based zero-copy dumps. Multi-month project; not
  in 0.16 even with the expanded scope. The current `recv_msg`-based
  streaming is good enough for everything short of 100k+
  messages/second.

### 9.4 Updated total effort

With the additional surfaces, this plan's total effort grows from
1.5 days to **3 days**. Reflected in the master plan §2.

End of plan.
