# Plan 164 — `NftablesConfig::diff` perf hoist

The pre-release audit's bug-hunt agent flagged a perf gotcha in
the Plan 157 declarative-config diff path: for N declared
tables, the diff makes ~3N kernel round-trips
(`list_chains()` + `list_flowtables()` + `list_rules(table,
family)` per table) and re-parses the chain/flowtable lists
N times.

For the typical N=1-2 use case (single firewall ruleset) this
is fine. For the N=50-100 use case (multi-tenant operator
managing per-tenant tables, NixOS-style modular firewall
fragments) it's the difference between a 50ms diff and a
3-second one.

**Pure perf fix; no correctness issue. Non-blocking for 0.16.**

## 1. The current shape

`crates/nlink/src/netlink/nftables/config/diff.rs:170-431`:

```rust
for declared in &self.tables {
    // ...
    let current_chains = conn.list_chains().await?;       // ← per-table
    let chains_in_table: Vec<_> = current_chains.iter()
        .filter(|c| c.table == declared.name() && /*...*/)
        .collect();
    // ...
    let current_rules = conn.list_rules(
        declared.name(),
        declared.family(),
    ).await?;                                              // ← per-table
    // ...
    let current_flowtables = conn.list_flowtables().await?; // ← per-table
    // ...
}
```

For each declared table:

- One `list_chains()` round-trip returning ALL chains across ALL
  tables (then filtered in-memory to the current declared
  table). Quadratic memory + N×kernel round-trip.
- One `list_rules(table, family)` round-trip, filtered to the
  declared chain in `for declared_chain` loop. This one is
  table-scoped on the kernel side so it's not as wasteful, but
  still N round-trips.
- One `list_flowtables()` round-trip returning ALL flowtables,
  filtered in-memory. Same quadratic shape as chains.

Total: ~3N round-trips, with `list_chains` + `list_flowtables`
parsing the same data N times.

## 2. Goals

By end of plan:

- `NftablesConfig::diff(conn)` makes **O(1) kernel round-trips
  for non-rule data** (one `list_chains` + one `list_flowtables`,
  shared across all declared tables).
- Rule dumps remain O(N tables) because they're table-scoped
  server-side (the kernel returns only that table's rules; no
  win from hoisting).
- Behavior unchanged — same diff output, same wire shape on
  apply.

Result: ~3× speedup on N=10, ~30-50× on N=100 (the win grows
with N because chain/flowtable parse time is the dominant
cost).

## 3. Scope

### In

- Hoist `list_chains()` and `list_flowtables()` out of the
  table loop; build `HashMap<(Family, TableName), Vec<&...>>`
  indexes once.
- Keep `list_rules(table, family)` inside the loop (already
  optimal — server-side scoped).
- Unit test demonstrating the round-trip count is invariant
  in N for chains + flowtables.
- Doc-comment on `NftablesConfig::diff` updated to document
  the O(1)-for-non-rules shape.

### Out

- Switching `list_rules` to `stream_rules` for memory bounds:
  separate plan (the streaming-dump variant exists in 0.16 via
  Plan 149; integrating it into the diff path is its own
  refactor with potential interleave-with-other-mutator
  considerations).
- Caching diff results across calls: useless without a
  cache-invalidation story; out of scope.
- Concurrent dumps via `ConnectionPool`: marginal win for
  diff (each dump is its own seq); single-conn is fine.

## 4. Design

### 4.1 Hoist pattern

```rust
impl NftablesConfig {
    pub async fn diff(&self, conn: &Connection<Nftables>) -> Result<NftablesDiff> {
        let mut diff = NftablesDiff::default();

        // Tables presence diff (unchanged) ...

        // Hoist: one dump of all chains, indexed.
        let all_chains: Vec<ChainInfo> = conn.list_chains().await?;
        let chains_by_table: HashMap<(Family, &str), Vec<&ChainInfo>> = all_chains
            .iter()
            .fold(HashMap::new(), |mut acc, c| {
                acc.entry((c.family, c.table.as_str())).or_default().push(c);
                acc
            });

        // Hoist: one dump of all flowtables, indexed.
        let all_flowtables: Vec<FlowtableInfo> = conn.list_flowtables().await?;
        let flowtables_by_table: HashMap<(Family, &str), Vec<&FlowtableInfo>> = all_flowtables
            .iter()
            .fold(HashMap::new(), |mut acc, f| {
                acc.entry((f.family, f.table.as_str())).or_default().push(f);
                acc
            });

        for declared in &self.tables {
            if /* table is in tables_to_add */ {
                /* promote nested contents (unchanged) */;
                continue;
            }

            // Index lookup, not re-parse:
            let chains_in_table: &[&ChainInfo] = chains_by_table
                .get(&(declared.family(), declared.name()))
                .map(|v| v.as_slice())
                .unwrap_or(&[]);

            let fts_in_table: &[&FlowtableInfo] = flowtables_by_table
                .get(&(declared.family(), declared.name()))
                .map(|v| v.as_slice())
                .unwrap_or(&[]);

            // Rules: table-scoped server-side; keep inside loop.
            let current_rules = conn
                .list_rules(declared.name(), declared.family())
                .await?;

            // ... rest of diff logic unchanged (chain presence,
            //     chain delete, rule USERDATA-keyed identity,
            //     flowtable presence) ...
        }

        Ok(diff)
    }
}
```

The diff's per-table inner logic (chain presence check, rule
identity diff, flowtable presence check) is unchanged — only
the **source** of `chains_in_table` / `fts_in_table` changes
from "fresh dump each iteration" to "indexed view of a
hoisted dump."

### 4.2 Memory shape

- Before: peak = max(per-table chain list size). One iteration's
  worth of memory.
- After: peak = total chain list size. Held across the loop.

For typical N=1-2 with 5-20 chains: ~doubles a few KiB. Acceptable.

For very large N (hundreds of tables, thousands of chains each):
hoisting holds the whole dump in memory at once. **If this
becomes a concern, switch to `stream_chains` + per-table filter
on the stream side** — that's the 0.18 follow-up. For 0.17 the
hoist is the right shape (the maintenance cost of the hoist
version is much lower than the streaming variant).

### 4.3 Edge cases

- **Concurrent mutation between hoisted-list + per-rule dump**:
  the existing per-iteration shape has the same race (chains
  could change between the chain dump and the rule dump for the
  next iteration). The hoist doesn't make this worse.
- **EPERM on `list_chains`**: surfaces immediately, same as
  today. Test: ensure the error path returns `Error::permission_denied()`.
- **Empty kernel state**: hoisted lists are empty; index lookups
  return `&[]`; loop iterations are no-ops.

## 5. Implementation phases

### Phase 1: refactor (~1 hour)

1. Add the two `HashMap<(Family, &str), Vec<&...>>` index types
   + builders.
2. Hoist `list_chains()` + `list_flowtables()` out of the loop.
3. Switch inner reads to index lookups.
4. Keep `list_rules(table, family)` inside the loop (no change).

### Phase 2: tests (~1 hour)

5. Unit test mocking `Connection` (via a trait or test helper):
   ```rust
   #[tokio::test]
   async fn diff_hoists_chain_list_dump_to_O1() {
       let mock_conn = MockNftablesConn::with_call_log();
       let cfg = NftablesConfig::new()
           .table("t1", Family::Inet, |t| t.chain("c1", |c| c))
           .table("t2", Family::Inet, |t| t.chain("c1", |c| c))
           .table("t3", Family::Inet, |t| t.chain("c1", |c| c));
       let _ = cfg.diff(&mock_conn).await.unwrap();
       assert_eq!(mock_conn.list_chains_calls(), 1, "should hoist");
       assert_eq!(mock_conn.list_flowtables_calls(), 1, "should hoist");
       assert_eq!(mock_conn.list_rules_calls(), 3, "table-scoped");
   }
   ```
6. Existing diff tests still pass (output unchanged).

### Phase 3: doc + CHANGELOG (~15 min)

7. Doc comment on `NftablesConfig::diff`:

   > Computes the diff with **O(1) round-trips for chain /
   > flowtable enumeration** (single dump, indexed; reused
   > across all declared tables) and O(N tables) for rule
   > enumeration (rule dumps are server-side scoped by table).

8. CHANGELOG entry under Changed.

## 8. Risks

- **Mock `Connection<Nftables>` complexity**: nlink doesn't
  currently mock its `Connection` types for unit tests. Either
  add a trait-based abstraction (significant refactor) or
  count the round-trips via a different mechanism (e.g. a test
  that runs against a `LabNamespace` and counts kernel events
  via netlink-monitor). Mitigation: trait-based abstraction is
  out of scope; use a `LabNamespace`-based integration test
  gated on `lab` + `require_root!()` (deferred to Plan 140 CI).
  Alternative: in-tree call-count test using a thin wrapper
  type that records `list_chains` calls.

- **Memory regression at very large N**: documented in §4.2.
  Mitigation: doc the trade-off; defer streaming variant to
  0.18.

## 9. Why this is 0.17, not 0.16

The current implementation is **correct**. The audit flagged it
as "perf, not correctness." Correctness fixes block a release;
perf fixes don't. The hoist is a clean, isolated 2.5-hour
follow-up that fits naturally into the 0.17 backlog.

Landing it in 0.17 means downstream users with large N
declared tables get a 3-30× speedup; downstream users with
small N see no change. Either way, no semver impact.

End of plan.
