# Plan 146 — 0.16 cycle master

0.15.1 shipped quality patches + one CI-discovered bug fix (`del_route`
strict-key matching for OIF-bound routes). Since then, two external
PRs landed (avionix-g/p13marc#1 + #2) adding:

- `Neighbor::write_delete` ndm_flags propagation (proxy NDP fix)
- IPv6 source/destination match helpers on the nftables `Rule` builder

Plus the trait method `RouteConfig::write_delete_with_interfaces` from
commit `ba2270b` — semver-additive, forces the next release to be
**0.16.0** rather than 0.15.2.

This master plan orchestrates the 0.16 cycle: scope, acceptance gates,
sub-plan structure, and the tier split between "must ship" and "ship
if cycle has room."

## 1. Why 0.16 now

0.15.1 closed the patch-release surface. The accumulated work for 0.16
pulls in **everything from `CODE_ANALYSIS.md`** that has real user
value — no tier-1/tier-2 split, no "defer to 0.17" for items with
real demand. The cycle gets bigger, but the result is a single
substantive minor release rather than a sequence of half-releases.

Estimated cycle length: **~5-6 focused weeks** — grown from ~3
weeks as 156-159 (DPLL family, declarative `NftablesConfig`,
syscall batching, ConnectionPool) were pulled in. See §2 table for
per-plan effort + §7 for sequencing. Compared to the original
~1-week tier-1-only plan, this is a substantial-but-bounded
single-cycle scope.

Three items are kept out of 0.16 with explicit rationale — see §4
"Genuinely out of scope."

### Genuinely out of scope (with rationale)

Six items I'm explicitly **not** pulling in, with reasons that hold up:

1. **CNI plugin demo bin** — research confirmed there's no
   production Rust CNI plugin in 2026 (sart, masap/rust_cni,
   passcod/cni-plugins are all hobby projects; Cilium/Calico/AWS-VPC
   are Go). Building a CNI demo is **market speculation**, not value
   delivery. Reconsider once a real Rust CNI exists upstream.
2. **`tcx` BPF attach helpers** — `aya` does this in-tree
   (`aya/programs/tc/`, verified via docs). Adding parallel surface
   in nlink would duplicate `aya`'s work and confuse the boundary
   between the two libs. The `aya` co-demo in Plan 152 is the right
   path; it shows the integration without forcing nlink to absorb
   BPF program-loading responsibility.
3. **nl80211 MLO (Wi-Fi 7 multi-link operation)** — Wi-Fi 7
   hardware is not widely deployed in mid-2026; the `wifi` bin is
   already marked as a low-priority POC. Adding MLO without a
   downstream user asking is speculation. Will revisit once a real
   request comes in.
4. **MPTCP userspace path-manager** — the in-kernel PM covers the
   common case; userspace PM is niche. nlink already has MPTCP
   GENL support; specific userspace-PM features can be added
   on-demand without a strategic push.
5. **Sync API surface (mirror of neli's `--features sync`)** —
   surfaced by COMPARISON_NELI.md §5 item 3. neli provides parallel
   sync + async APIs from one type tree. nlink commits to async-only.
   **Rationale to NOT add sync:**
   - tokio is the de facto async runtime; ~95%+ of greenfield Rust
     network code in 2026 is async-on-tokio. Sync is the fallback for
     legacy / embedded / specific-use-case codebases, not the default.
   - Adding sync doubles the surface area to maintain. Every Connection
     method needs a sync sibling; tests for both; docs for both. The
     project's sole maintainer should not absorb that overhead absent
     concrete demand.
   - neli's sync default reflects its history (sync came first; async
     was retrofitted). nlink was async-from-day-one; the symmetry
     argument doesn't apply.
   - If a downstream user files an issue saying "I can't use nlink
     because I need sync," reconsider. Until then, document the
     stance in the README and move on.
6. **NlRouter-style fan-out dispatch (concurrent unicast on one socket)**
   — surfaced by COMPARISON_NELI.md §5 item 2. neli's `NlRouter`
   demuxes responses by sequence number, allowing concurrent unicast
   requests on the same socket. nlink's "one Connection per concern;
   single-flight per Connection" pattern is different.
   **Rationale to DEFER (not reject) to 0.17:**
   - The change is architectural. It requires restructuring
     `Connection<P>::send_request` / `send_dump` to allow N requests
     in-flight per socket, with a background task fanning replies out
     by sequence number.
   - It's a multi-week effort, with real risk of regression in
     existing single-flight callers if the borrow semantics shift.
   - 0.16 is already a 4-5 week cycle with Plan 154. Adding this
     would push the cut past 2 months.
   - The current pattern (open multiple connections for concurrent
     traffic — cheap; netlink sockets are file descriptors) works
     fine for the documented use cases.
   - **0.17 candidate**: write a design plan (similar shape to
     Plan 154's), prototype on `Connection<Generic>` first, evaluate
     real-world benefit.

These are documented in the CHANGELOG `## [Out of scope]` section
so users searching for them find the rationale.

## 8. Risks

- **Plan 149 scope creep**: Streaming dump touches the lower-level
  `send_dump_inner`. If that refactor cascades into broader connection
  changes, defer the broader cleanup to 0.17 rather than blocking
  0.16. The eager `get_*` variants stay in tree; streaming is purely
  additive.
- **Plan 150 kernel-test risk**: Flowtable testing needs kernel
  module `nf_flow_table` loaded. Workflow modprobe wave (Plan 144 §4.2)
  already covers this; flowtable tests gate with `require_module!`.
- **Plan 154 macro-design iteration**: proc-macro design typically
  needs 1-2 redesign passes after the first real consumer surfaces
  issues. Plan 156 (DPLL) is sequenced immediately after Plan 154 to
  catch these early — but if 156 reveals a macro-design issue late,
  the fallback is to ship 154 with the broken syntax behind
  `#[doc(hidden)]` "experimental" framing and stabilize it in 0.16.1.
  Plan 152's aya demo also depends on the macros and would have to
  defer to 0.16.1 in that scenario.
- **Plan 156 first-dogfood discoveries**: building DPLL via macros
  is the first time the macros meet a real subsystem with nested
  attributes + value enums + transparent newtypes. Expect to
  surface ≥1 type-mapping gap. Mitigation: develop 156 in parallel
  with 154 Phase 6 (example); the example + 156 jointly drive the
  design, not 154 in isolation.
- **Plan 157 canonicalization edge cases**: typed-level rule sort
  + canonical re-emit catches the common cases (per §4.3); the
  documented limitations (`NFTA_RULE_COMPAT`, unrecognized future
  attrs) produce false-positive re-applies that are harmless.
  Risk: discovering a *fourth* false-positive class that wasn't
  predicted, producing user-visible churn. Mitigation: integration
  test 6 (idempotent reapply) catches this on the most common
  shapes; ship knowing the remaining tail-risk; document.
- **Plan 158 perf may not match prediction**: the 2-5× speedup
  claim from research is based on quinn-udp's UDP scenarios.
  Netlink dumps have different message-size distributions; the
  actual win may be smaller. Mitigation: ship the feature
  regardless (it can't be slower than current); record actual
  numbers in PR; let the criterion benches speak.
- **Plan 159 sealed-trait dependency**: depends on Plan 148 §4.5
  having shipped. If 148 §4.5 slips, 159 ships sync-only and
  gains the GENL-pool variant in a follow-up. The pool works fine
  for rtnetlink / netfilter / xfrm regardless — the slip is
  contained.
- **Trait-method addition surface inflation**: The 0.16 cut adds at
  least one trait method (`write_delete_with_interfaces`). If we add
  many more across the 13 sub-plans, the `cargo-semver-checks` output
  becomes hard to review at release time. Be deliberate about
  trait-surface additions; prefer free functions where possible.
- **Workspace dep additions** (`bitflags`, `nlink-macros`,
  possibly `serde` behind a feature for Plan 157): each is small,
  but together they bump nlink's transitive dep count. `cargo
  machete` clean is in the acceptance gates; `cargo tree` review
  at release time catches accidental duplicates.

  ENOBUFS automatically.
- **Diagnostics**: extended-ack TLV parsing turns `errno = 22`
  into actionable strings.
- **Kernel features**: DPLL (6.7+), XFRM offload, devlink rate,
  TX H/W shaping (6.13+), nftables flowtable.
