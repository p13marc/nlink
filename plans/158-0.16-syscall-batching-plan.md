# Plan 158 — `recvmmsg` / `sendmmsg` syscall batching for 0.16

The next perf cliff after Plan 149's streaming dumps. Each netlink
dump frame currently costs one `recvmsg(2)` syscall. On a 100k-link
dump that's 100k syscalls = ~50ms of pure syscall overhead on a
modern host. `recvmmsg(2)` returns up to N frames per syscall;
typical wins per quinn-udp's prior art are **2-5×** for dump-heavy
workloads.

This plan adds `recvmmsg` / `sendmmsg`-backed batched I/O to
`NetlinkSocket`, then wires it into the dump-heavy paths
(routes, links, neighbors, conntrack, nft ruleset, FDB).

## 1. Summary

Three deliverables:

1. **`NetlinkSocket::recv_batch(&self, out: &mut Vec<Vec<u8>>,
   max: usize) -> Result<usize>`** + complementary
   `send_batch(&self, msgs: &[&[u8]]) -> Result<usize>` —
   `libc::{recvmmsg, sendmmsg}` wrapped behind the existing
   `AsyncFd` readiness path.
2. **Per-socket reused buffer pool** (~1 MB per socket: 32 slots ×
   32 KiB). Replaces the existing per-`recv_msg` allocation —
   a separate small allocator win independent of the batching.
3. **Opt-in feature flag `syscall_batch`** for one release while
   the benchmarks (Plan 156-as-annex) confirm the speedup on real
   workloads. Default-on in 0.17 after a release cycle of soak.

Plus benchmark + integration tests + CHANGELOG.

## 2. Goals

By end of plan:

- `cargo bench -p nlink --bench dump_throughput` shows ≥2×
  reduction in syscall count and ≥1.5× wall-clock speedup on the
  100k-route synthetic dump (Plan 156 annex provides the
  bench; this plan provides the speedup to measure).
- All existing tests pass with `syscall_batch` enabled; behavior
  is byte-identical from the caller's perspective.
- A new `recv_batch` / `send_batch` public API is documented as
  the lower-level building block; the existing `recv_msg` /
  `send` wrappers remain unchanged for callers that don't want
  batching.
- AsyncFd lifecycle is unaffected — `recv_batch` integrates via
  the same `ready().await` → `try_io(...)` pattern as
  `recv_msg`.

## 3. Scope

### In

- `NetlinkSocket::recv_batch` + `send_batch` per §4.1.
- Per-socket owned buffer pool (`Vec<Vec<u8>>` × batch-size, 32
  KiB each).
- Internal switch in `connection.rs`'s dump paths to call
  `recv_batch` when the `syscall_batch` feature is on.
- Feature flag `syscall_batch` defined in `crates/nlink/Cargo.toml`.
- Benchmark `crates/nlink/benches/dump_throughput.rs` (criterion;
  bundled with Plan 156's bench suite if that lands as an annex).
- Unit tests for the unsafe wrapping code (boundary conditions on
  batch size, MSG_TRUNC handling).
- Integration test: a 1000-link dump round-trips identically with
  and without `syscall_batch` (same parsed `LinkMessage` set).
- CHANGELOG entry.

### Out

- `NETLINK_RX_RING` zero-copy receive. Multi-month project; remains
  the 0.17+ frontier per CODE_ANALYSIS.md §3.3.
- `recvmsg`-side I/O uring backend. Different architecture entirely.
- Per-connection statistics on batch fill ratio. Useful for
  tuning but not blocking; demand-gated.
- Tuning the batch size at runtime. Hardcoded const for 0.16;
  expose as `Connection<P>::with_batch_size(usize)` if anyone
  needs it.

## 4. Design

### 4.1 The batched recv path

Wrap `libc::recvmmsg` with the same `AsyncFd` discipline as
`recv_msg`. Pass `MSG_DONTWAIT`; pass `NULL` for the broken
`timeout` arg; rely on tokio's `timeout(...)` for deadline
management.

```rust
// crates/nlink/src/netlink/socket.rs

pub const NL_BATCH_SIZE: usize = 32;
pub const NL_BUF_SIZE: usize = 32 * 1024;

pub struct NetlinkSocket {
    fd: AsyncFd<RawFd>,
    pid: u32,
    /// Per-socket batched-recv buffers, reused across calls.
    /// Allocated lazily on first batched recv.
    batch_bufs: Mutex<Option<BatchBufs>>,
}

struct BatchBufs {
    /// Owned buffer storage; never moved while batch_bufs is held.
    storage: Vec<Vec<u8>>,
    /// iovec entries pointing into `storage`. Rebuilt on each call
    /// (cheap — 32 entries).
    iovecs: Vec<libc::iovec>,
    /// mmsghdr array; msg_hdr.msg_iov points into `iovecs`.
    msgs: Vec<libc::mmsghdr>,
}

impl NetlinkSocket {
    /// Receive up to `max` netlink datagrams in one syscall. Returns
    /// the count received and pushes each into `out` as an owned
    /// `Vec<u8>` of the message's actual length (not the buffer
    /// capacity). `out` is appended to, not cleared.
    ///
    /// On `EAGAIN` (no data ready), returns `Ok(0)` after one
    /// AsyncFd re-arm; the caller can poll again or back off.
    /// Other errors propagate.
    ///
    /// `max` is clamped to `NL_BATCH_SIZE`.
    pub async fn recv_batch(
        &self,
        out: &mut Vec<Vec<u8>>,
        max: usize,
    ) -> Result<usize> {
        let max = max.min(NL_BATCH_SIZE);
        loop {
            let mut guard = self.fd.ready(Interest::READABLE).await?;
            match guard.try_io(|inner| {
                self.recv_batch_inner(inner.get_ref(), max)
            }) {
                Ok(Ok(frames)) => {
                    let n = frames.len();
                    out.extend(frames);
                    return Ok(n);
                }
                Ok(Err(e)) => return Err(e.into()),
                Err(_would_block) => continue, // re-arm AsyncFd
            }
        }
    }

    fn recv_batch_inner(&self, fd: &RawFd, max: usize) -> io::Result<Vec<Vec<u8>>> {
        let mut bufs_guard = self.batch_bufs.lock().unwrap();
        let bufs = bufs_guard.get_or_insert_with(BatchBufs::new);
        bufs.prepare(max);

        let n = unsafe {
            libc::recvmmsg(
                *fd,
                bufs.msgs.as_mut_ptr(),
                max as libc::c_uint,
                libc::MSG_DONTWAIT,
                std::ptr::null_mut(),
            )
        };
        if n < 0 {
            return Err(io::Error::last_os_error());
        }

        let mut frames = Vec::with_capacity(n as usize);
        for i in 0..n as usize {
            let len = bufs.msgs[i].msg_len as usize;
            let flags = bufs.msgs[i].msg_hdr.msg_flags;
            if flags & libc::MSG_TRUNC != 0 {
                // 32 KiB ought to be enough; if not, this is a
                // protocol-level bug we want to scream about.
                return Err(io::Error::new(
                    io::ErrorKind::InvalidData,
                    format!("netlink frame {} exceeded NL_BUF_SIZE ({} > {})",
                        i, len, NL_BUF_SIZE),
                ));
            }
            frames.push(bufs.storage[i][..len].to_vec());
        }
        Ok(frames)
    }
}

impl BatchBufs {
    fn new() -> Self {
        let storage: Vec<Vec<u8>> = (0..NL_BATCH_SIZE)
            .map(|_| vec![0u8; NL_BUF_SIZE])
            .collect();
        let mut iovecs = Vec::with_capacity(NL_BATCH_SIZE);
        let mut msgs: Vec<libc::mmsghdr> = (0..NL_BATCH_SIZE)
            .map(|_| unsafe { std::mem::zeroed() })
            .collect();
        // Iovecs + msgs wired up in `prepare()` since storage
        // pointers must be re-fetched (Vec<u8> can grow; storage
        // pointers never change after init but we redo it for
        // clarity).
        for buf in &storage {
            iovecs.push(libc::iovec {
                iov_base: buf.as_ptr() as *mut libc::c_void,
                iov_len: buf.len(),
            });
        }
        for (i, msg) in msgs.iter_mut().enumerate() {
            msg.msg_hdr.msg_iov = &mut iovecs[i] as *mut _;
            msg.msg_hdr.msg_iovlen = 1;
            msg.msg_hdr.msg_name = std::ptr::null_mut();
            msg.msg_hdr.msg_namelen = 0;
            msg.msg_hdr.msg_control = std::ptr::null_mut();
            msg.msg_hdr.msg_controllen = 0;
            msg.msg_hdr.msg_flags = 0;
            msg.msg_len = 0;
        }
        BatchBufs { storage, iovecs, msgs }
    }

    fn prepare(&mut self, max: usize) {
        // Reset per-call mutable fields (msg_len, msg_flags) on the
        // active slots. iovecs + msg_iov bindings stay valid.
        for i in 0..max {
            self.msgs[i].msg_len = 0;
            self.msgs[i].msg_hdr.msg_flags = 0;
        }
    }
}
```

Key design points:

- **`Mutex<Option<BatchBufs>>`** because `&self` must be enough for
  the call. The mutex contention is fine for nlink's single-flight
  model (one recv at a time per Connection).
- **Owned `Vec<u8>` returned to caller**, not borrowed slices.
  Matches `recv_msg`'s contract; simpler ownership story; the
  `.to_vec()` copy is amortized vs the syscall cost.
- **`MSG_TRUNC` → hard error**, not silent truncation. nlink's
  buffer is 32 KiB, which exceeds the kernel's per-frame limit
  for everything except pathological dumps. If we ever hit one,
  it's worth a panic-loud error rather than silent data loss.
- **No `MSG_WAITFORONE`** — would block the executor thread.
  Always `MSG_DONTWAIT`.
- **`timeout` arg is `NULL`** — the kernel implementation has been
  buggy for >15 years (per `man recvmmsg` BUGS). tokio
  `timeout(...)` is the right tool.

### 4.2 The batched send path

```rust
impl NetlinkSocket {
    /// Send up to `msgs.len()` netlink request frames in one
    /// syscall. Returns the count successfully sent (≥1 on
    /// success; partial-send is possible per `sendmmsg(2)`
    /// semantics — see kernel man page).
    ///
    /// Each successful slot's `msg_len` is verified to match the
    /// input slice length; mismatch is treated as `ShortWrite`.
    ///
    /// **Important**: per-request kernel processing errors (the
    /// kernel's `NLMSG_ERROR` responses) are NOT surfaced here —
    /// they arrive asynchronously on the receive side, matched by
    /// `nlmsg_seq`. `send_batch` only reports syscall-level
    /// errors (ENOBUFS, EMSGSIZE, etc.).
    pub async fn send_batch(&self, msgs: &[&[u8]]) -> Result<usize> {
        if msgs.is_empty() { return Ok(0); }
        let n = msgs.len().min(NL_BATCH_SIZE);

        loop {
            let mut guard = self.fd.ready(Interest::WRITABLE).await?;
            match guard.try_io(|inner| {
                self.send_batch_inner(inner.get_ref(), &msgs[..n])
            }) {
                Ok(Ok(sent)) => return Ok(sent),
                Ok(Err(e)) => return Err(e.into()),
                Err(_would_block) => continue,
            }
        }
    }

    fn send_batch_inner(&self, fd: &RawFd, msgs: &[&[u8]]) -> io::Result<usize> {
        let mut iovecs: Vec<libc::iovec> = msgs.iter()
            .map(|m| libc::iovec {
                iov_base: m.as_ptr() as *mut libc::c_void,
                iov_len: m.len(),
            })
            .collect();
        let mut mmsg: Vec<libc::mmsghdr> = (0..msgs.len())
            .map(|i| unsafe {
                let mut hdr: libc::mmsghdr = std::mem::zeroed();
                hdr.msg_hdr.msg_iov = &mut iovecs[i] as *mut _;
                hdr.msg_hdr.msg_iovlen = 1;
                hdr
            })
            .collect();
        let n = unsafe {
            libc::sendmmsg(
                *fd,
                mmsg.as_mut_ptr(),
                msgs.len() as libc::c_uint,
                libc::MSG_DONTWAIT,
            )
        };
        if n < 0 {
            return Err(io::Error::last_os_error());
        }
        // Verify each successful slot wrote the full buffer length.
        for (i, slot) in mmsg.iter().enumerate().take(n as usize) {
            if slot.msg_len as usize != msgs[i].len() {
                return Err(io::Error::new(
                    io::ErrorKind::WriteZero,
                    format!("sendmmsg slot {i}: wrote {} of {} bytes",
                        slot.msg_len, msgs[i].len()),
                ));
            }
        }
        Ok(n as usize)
    }
}
```

Notes on `send_batch`:

- **Partial-send is normal**; the caller (typically a higher-level
  helper) loops calling from the returned offset until all sent or
  a real error.
- **Per-request kernel errors don't surface here** — they come back
  as `NLMSG_ERROR` frames on the recv side, matched by `nlmsg_seq`.
  The doc string is explicit about this.
- **iovecs allocated per-call**: 30 LOC + 1 small allocation per
  call. Could reuse like recv side; not worth it (sends are sparse
  in nlink today).

### 4.3 Feature flag

```toml
# crates/nlink/Cargo.toml
[features]
syscall_batch = []
```

When `syscall_batch` is off:
- `recv_batch` / `send_batch` still compile and work (they're not
  cfg-gated; only the wiring into dump paths is).
- Dump paths use `recv_msg`-in-loop as today. No behavior change.

When `syscall_batch` is on:
- Dump paths in `connection.rs` use `recv_batch` with batch=32.
- All other behavior unchanged.

Rationale for the flag: this is a syscall-level change with
performance implications across the board. Letting users opt in
for one release gives feedback on edge cases (kernel version
sensitivity, RAM-constrained deployments) before defaulting it on.

**0.17 plan**: remove the flag, make it default behavior. Plan 158
itself doesn't ship that switch.

### 4.4 Wiring into dump paths

The dump path in `connection.rs`'s `send_dump_inner` currently
loops on `recv_msg` until `NLMSG_DONE`. Replace the loop body
under `cfg(feature = "syscall_batch")`:

```rust
#[cfg(feature = "syscall_batch")]
{
    let mut batch = Vec::with_capacity(NL_BATCH_SIZE);
    let got = self.socket.recv_batch(&mut batch, NL_BATCH_SIZE).await?;
    if got == 0 { continue; }
    for frame in batch.drain(..) {
        // existing per-frame parse loop
    }
}

#[cfg(not(feature = "syscall_batch"))]
{
    let frame = self.socket.recv_msg().await?;
    // existing per-frame parse loop
}
```

Plan 149's `dump_stream` benefits identically — the streaming path
also runs through `send_dump_inner`'s receive loop, so the same
cfg gate covers it.

### 4.5 Allocation discipline

`BatchBufs::new()` allocates ~1 MB per socket (32 × 32 KiB). This
is paid **once per Connection**, not per recv. For nlink's typical
usage (one Connection per concern, small number of Connections per
process), the memory cost is negligible. For pool-heavy usage (Plan
159), the cost scales with pool size — document that
`ConnectionPool<P, N>` allocates ~N MB of recv buffers.

For users where 1 MB/socket is genuinely too much: expose
`with_batch_size(usize)` on Connection construction (defaulting to
`NL_BATCH_SIZE = 32`), letting them tune down.

### 4.6 Safety annotations

All `unsafe` blocks in this plan correspond to direct `libc::*`
calls with documented preconditions:

```rust
// SAFETY: `recvmmsg` requires the mmsghdr array to be valid for
// reads + writes for `vlen` entries. We own the array (Vec
// allocation), it's properly initialized (zeroed then partially
// filled in BatchBufs::new), and the iovec pointers reference
// owned storage that outlives this call (held in the same struct
// behind the Mutex guard).
let n = unsafe { libc::recvmmsg(...) };
```

Pattern matches what existing `socket.rs` does for `recvmsg`.

### 4.7 Test strategy

Unit tests (no kernel needed):

- `BatchBufs::new` produces a fully-wired array; every iovec
  points into its corresponding storage.
- `prepare(max)` resets only the first `max` slots' mutable
  fields.
- `MSG_TRUNC` on any slot is converted to `InvalidData` error.

Integration tests (`tests/integration/`):

- `recv_batch_round_trip_100_link_dump`: create 100 dummy
  interfaces, dump links with `syscall_batch` enabled, verify the
  same `LinkMessage` set as a non-batched dump.
- `recv_batch_partial_fill`: send a single request, recv_batch with
  max=32, verify it returns 1.
- `send_batch_atomic_failure_partial`: send 5 messages where the
  4th has an invalid family ID, verify slots 0-2 succeed and slot
  3 produces `NLMSG_ERROR` on the recv side.

Benchmark (`benches/`):

- `bench_dump_routes_batched_vs_unbatched`: synthetic 10k-route
  setup, criterion comparison. Target: ≥2× wall-clock speedup.
- `bench_send_batch_throughput`: tight-loop sender, criterion.
  Target: ≥3× throughput vs serial send.

## 8. Risks

- **Kernel-version sensitivity**: `recvmmsg` has been in Linux
  since 2.6.33 (2010); `sendmmsg` since 3.0 (2011). No realistic
  kernel-version risk. The `timeout` arg buggy-forever caveat is
  handled by passing `NULL`.
- **MSG_TRUNC on a real workload**: the 32 KiB per-slot buffer
  exceeds every realistic netlink frame, but a pathological dump
  (e.g., a 65k-entry route attribute) could overflow. The
  treatment (hard error) is correct — silent truncation would
  produce parse failures or wrong data later. Document the
  symptom: "if you hit `InvalidData: netlink frame N exceeded
  NL_BUF_SIZE`, file an issue; this suggests either a kernel bug
  or a never-seen-before attribute size."
- **`Mutex<BatchBufs>` contention under high concurrency**:
  irrelevant because nlink Connections are single-flight (one
  request at a time per Connection). ConnectionPool (Plan 159)
  shards across multiple Connections, each with its own mutex,
  so still no contention.
- **Allocation surprise**: 1 MB per Connection is paid lazily on
  first batched recv. Document so users who open many Connections
  don't get caught out. Tune via `with_batch_size`.
- **Behavior parity**: batched and non-batched paths must produce
  identical parsed message sets for the same kernel state.
  Integration test in §4.7 covers this — gating PR on it passing.
- **Bench setup complexity**: synthesizing 10k routes for the
  bench requires `LabNamespace` + many `add_route` calls. The
  benchmark *setup* may dominate per-iteration wall time; use
  criterion's `setup`-once pattern to amortize.
