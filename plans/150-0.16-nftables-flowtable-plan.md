# Plan 150 — nftables flowtable for 0.16

The headline fastpath feature for nftables: a per-table object that
caches established connection-tracking flows, letting the kernel
bypass the full nftables rule traversal for matching packets. On
capable NICs, can be hardware-offloaded via
`NF_FLOWTABLE_HW_OFFLOAD`. Used by routers / firewalls doing
high-throughput forwarding.

nlink today has no flowtable surface (confirmed via grep). This plan
adds it.

## 1. Summary

Three new public surfaces:

1. `Flowtable` builder + `add_flowtable` / `del_flowtable` Connection
   methods.
2. A `flow add @table` expression on `Rule` to populate the table from
   the rule path.
3. Recipe `docs/recipes/nftables-flowtable.md` showing the typical
   bridge-firewall + flowtable layout.

Plus integration tests gated on `nf_flow_table` kernel module.

## 2. Goals

By end of plan:

- Users can declaratively create a flowtable bound to one or more
  ingress devices.
- Rules can `add @ft` to populate the flowtable with matched flows.
- Hardware offload is selectable via a builder flag
  (`Flowtable::hw_offload(true)`) — emits
  `NFTA_FLOWTABLE_FLAGS = NF_FLOWTABLE_HW_OFFLOAD`.
- The lib emits the correct messages: `NFT_MSG_NEWFLOWTABLE` (= 22)
  for create, `NFT_MSG_DELFLOWTABLE` (= 24) for delete,
  `NFT_MSG_GETFLOWTABLE` (= 23) for dump.
- Recipe shows the typical setup: bridge filter table + forward chain
  + flowtable + rule that adds flows.

## 3. Scope

### In

- New `Flowtable` struct in `nftables/types.rs` (next to `Table`,
  `Chain`, `Rule`).
- New `Connection<Nftables>` methods: `add_flowtable`, `del_flowtable`,
  `get_flowtables` (dump), `get_flowtable` (single by name).
- New `Expr::FlowOffload { table: String }` variant in `nftables/expr.rs`
  for the `flow add @<table>` rule expression.
- New `NFT_MSG_NEW/GET/DELFLOWTABLE` constants in `nftables/mod.rs`.
- New `NFTA_FLOWTABLE_*` attribute IDs.
- Integration tests under `tests/integration/` — possibly a new
  `nftables.rs` test file if there isn't one (audit existing tests
  first).
- Recipe `docs/recipes/nftables-flowtable.md`.
- CHANGELOG entry.

### Out

- nftables flowtable counters (separate feature; demand-gated).
- `softwarefallback` toggle on hw-offload flowtables — covered by the
  flags field but no typed setter; defer if non-trivial.
- Per-flowtable conntrack zone scoping — niche.
- Dynamic flow eviction commands — handled by kernel timers, no
  netlink surface.

## 4. Design

### 4.1 Kernel ABI quick reference

From `include/uapi/linux/netfilter/nf_tables.h`:

```c
enum nf_tables_msg_types {
    /* ... */
    NFT_MSG_NEWFLOWTABLE        = 22,
    NFT_MSG_GETFLOWTABLE        = 23,
    NFT_MSG_DELFLOWTABLE        = 24,
};

enum nft_flowtable_attributes {
    NFTA_FLOWTABLE_UNSPEC,
    NFTA_FLOWTABLE_TABLE,        // NLA_STRING — owning table name
    NFTA_FLOWTABLE_NAME,         // NLA_STRING — flowtable name
    NFTA_FLOWTABLE_HOOK,         // NLA_NESTED — hook info (priority, devs)
    NFTA_FLOWTABLE_USE,          // NLA_U32 — refcount (read-only)
    NFTA_FLOWTABLE_HANDLE,       // NLA_U64 — kernel handle (read-only)
    NFTA_FLOWTABLE_PAD,
    NFTA_FLOWTABLE_FLAGS,        // NLA_U32 — flags
    __NFTA_FLOWTABLE_MAX,
};

enum nft_flowtable_flags {
    NFT_FLOWTABLE_HW_OFFLOAD     = 0x1,
    NFT_FLOWTABLE_COUNTER        = 0x2,
};

enum nft_flowtable_hook_attributes {
    NFTA_FLOWTABLE_HOOK_UNSPEC,
    NFTA_FLOWTABLE_HOOK_NUM,     // NLA_U32 — must be NF_NETDEV_INGRESS (5)
    NFTA_FLOWTABLE_HOOK_PRIORITY,// NLA_U32 — typically 0 or +5
    NFTA_FLOWTABLE_HOOK_DEVS,    // NLA_NESTED — list of NFTA_DEVICE_NAME
};
```

The flowtable lives in the `inet` or `ip`/`ip6` family table. Its hook
is always `NF_NETDEV_INGRESS` (kernel constant 5), attaches to one or
more netdevs.

### 4.2 Builder shape (per project convention)

```rust
// crates/nlink/src/netlink/nftables/types.rs

#[derive(Debug, Clone)]
pub struct Flowtable {
    /// Owning table family (e.g. "inet", "ip").
    pub family: Family,
    /// Owning table name.
    pub table: String,
    /// Flowtable name (unique within the table).
    pub name: String,
    /// Device names to attach the ingress hook to.
    pub devs: Vec<String>,
    /// Hook priority (typically 0; `-300` for early ingress).
    pub priority: i32,
    /// Flags bitmap (`FlowtableFlag::HW_OFFLOAD | FlowtableFlag::COUNTER`).
    pub flags: FlowtableFlag,
}

impl Flowtable {
    pub fn new(family: Family, table: impl Into<String>, name: impl Into<String>) -> Self {
        Self {
            family,
            table: table.into(),
            name: name.into(),
            devs: Vec::new(),
            priority: 0,
            flags: FlowtableFlag::empty(),
        }
    }

    /// Attach the flowtable's ingress hook to a device. Call multiple
    /// times to attach to several devices (e.g. both ends of a bridge).
    pub fn device(mut self, dev: impl Into<String>) -> Self {
        self.devs.push(dev.into());
        self
    }

    /// Set the ingress hook priority. Default is 0.
    pub fn priority(mut self, p: i32) -> Self {
        self.priority = p;
        self
    }

    /// Request hardware offload. Requires a NIC with flow-table
    /// offload support (mlx5, hns3, etc.) and the kernel built with
    /// CONFIG_NF_FLOW_TABLE_HW.
    ///
    /// Failure mode if the NIC doesn't support offload: the kernel
    /// accepts the add and silently falls back to software. There
    /// is no in-band signal of whether offload actually engaged;
    /// check `ethtool -k <dev>` or the `tc -s filter` counters to
    /// confirm.
    pub fn hw_offload(mut self, on: bool) -> Self {
        self.flags.set(FlowtableFlag::HW_OFFLOAD, on);
        self
    }

    /// Request per-flow counter tracking. Adds overhead.
    pub fn counter(mut self, on: bool) -> Self {
        self.flags.set(FlowtableFlag::COUNTER, on);
        self
    }
}

bitflags::bitflags! {
    /// Flowtable flags (`NFTA_FLOWTABLE_FLAGS`). See `bitflags` 2.x
    /// for the operations (`union`, `difference`, `set`, `contains`).
    #[derive(Debug, Default, Clone, Copy, PartialEq, Eq)]
    pub struct FlowtableFlag: u32 {
        const HW_OFFLOAD = 0x1;
        const COUNTER    = 0x2;
    }
}
```

## 8. Risks

- **Kernel ABI confirmation**: NFTA_FLOWTABLE_HOOK_DEVS encoding is a
  nested list of string attributes. Verify against
  `include/uapi/linux/netfilter/nf_tables.h` rather than trusting
  documentation; the header is the source of truth.
- **Family validation**: `NFT_MSG_NEWFLOWTABLE` accepts only certain
  family values (typically `inet`, `ip`, `ip6`; not `bridge` or
  `arp`). If the lib doesn't validate, the kernel returns `ENOTSUP` —
  fine, but a typed error would be nicer.
- **Device-name resolution**: the flowtable's `NFTA_FLOWTABLE_HOOK_DEVS`
  attribute takes device **names**, not ifindexes. Inside foreign
  netns this raises the usual "name resolution happens in the calling
  process's namespace" question — but since we're just passing the
  string through to the kernel which resolves in the netlink socket's
  netns, this is actually fine. Document the contract.
- **Recipe verification**: writing the recipe in detail requires
  having a working flowtable to test against. If the author hasn't
  used flowtables before, budget 1-2 hours of `nft -i` interactive
  exploration before writing.
- **HW offload test reliability**: real hw-offload requires capable
  NIC; CI doesn't have one. The `hw_offload_flag_accepted` test
  should either skip when not present or assert only that the kernel
  doesn't error (kernels with no HW support typically return
  `EOPNOTSUPP` when you try to add an hw-offload flowtable; gate
  accordingly).

## 9. Additional flowtable items folded in (formerly §9 "out of scope")

Per the master plan's "no deferral to 0.17" stance:

### 9.1 Flowtable counter introspection — closed without new API surface

**Closed for 0.16** (revisited 2026-05-24 after research). The
original design assumed `NFT_MSG_GETFLOWTABLE` would return
per-flow counter attributes when the flowtable was created with
`NFT_FLOWTABLE_F_COUNTER`. **This premise was wrong.**

Verified against kernel UAPI: `enum nft_flowtable_attributes` in
`include/uapi/linux/netfilter/nf_tables.h` exposes only
configuration attrs — `NFTA_FLOWTABLE_TABLE`, `_NAME`, `_HOOK`,
`_USE`, `_HANDLE`, `_FLAGS`. No per-flow tuple or counter
attribute exists. `nf_tables_fill_flowtable_info` in
`net/netfilter/nf_tables_api.c` emits only those config attrs.

**Where per-flow counters actually live**: inside the
**conntrack entries** that the flowtable has offloaded. The
`flags counter` flowtable option tells the offload fastpath to
keep the existing conntrack entry's `CTA_COUNTERS_ORIG` /
`CTA_COUNTERS_REPLY` in sync with packets that bypass the slow
path. Standard `IPCTNL_MSG_CT_GET` dumps surface them, with the
`IPS_OFFLOAD_BIT (= 14)` / `IPS_HW_OFFLOAD_BIT (= 15)` bits set
in `CTA_STATUS` (`include/uapi/linux/netfilter/nf_conntrack_common.h`).

**Status as of 0.16**: the infrastructure is **already in place**
via Plan 149's `Connection::<Netfilter>::stream_conntrack` +
`ConntrackStatus::OFFLOAD` / `::HW_OFFLOAD` + the existing
`ConntrackEntry::counters_orig` / `counters_reply` fields. The
correct pattern is:

```rust,no_run
use nlink::{Connection, Netfilter};
use nlink::netlink::netfilter::ConntrackStatus;
use tokio_stream::StreamExt;

# async fn run() -> nlink::Result<()> {
let conn = Connection::<Netfilter>::new()?;
let mut stream = conn.stream_conntrack(libc::AF_INET as u8).await?;
while let Some(entry) = stream.next().await {
    let entry = entry?;
    let Some(status) = entry.status else { continue };
    if !status.contains(ConntrackStatus::OFFLOAD)
        && !status.contains(ConntrackStatus::HW_OFFLOAD)
    {
        continue; // not offloaded → counters not flowtable-related
    }
    if let Some(c) = &entry.counters_orig {
        println!("orig: packets={} bytes={}", c.packets, c.bytes);
    }
    if let Some(c) = &entry.counters_reply {
        println!("reply: packets={} bytes={}", c.packets, c.bytes);
    }
}
# Ok(())
# }
```

This is what `conntrack -L` does internally (the `[OFFLOAD]` /
`[HW_OFFLOAD]` markers in its output come from the same two
status bits). No third API exists; kube-proxy, Cilium, and
firewalld don't expose flowtable per-flow counters via NFT
either — they read ctnetlink or skip the feature.

**No new code needed for 0.16.** The deferred §9.1 design is
formally closed with the rationale recorded here.

**Recipe addition**:
[`docs/recipes/nftables-stateful-fw.md`](../docs/recipes/nftables-stateful-fw.md)
gets a short "Per-flow counters for offloaded flows" section
showing the `stream_conntrack` + status-filter pattern above.

### 9.2 Multicast events for flowtable + table/chain/rule changes

The nftables netlink family already emits `NFT_MSG_NEW*` events on
mutation. Adding subscription support is a small addition:

```rust
impl Connection<Nftables> {
    pub fn subscribe(&mut self, groups: &[NftablesGroup]) -> Result<()>;
    pub fn events(&mut self) -> impl Stream<Item = Result<NftablesEvent>>;
}

pub enum NftablesEvent {
    NewTable(Table), DelTable(...),
    NewChain(Chain), DelChain(...),
    NewRule(Rule), DelRule(...),
    NewFlowtable(Flowtable), DelFlowtable(...),
}
```

Mirrors the existing `Connection<Netfilter>::subscribe` /
`Connection<Netfilter>::events` API for conntrack events.
