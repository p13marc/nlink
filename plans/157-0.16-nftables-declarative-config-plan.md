# Plan 157 — declarative `NftablesConfig` for 0.16

`NetworkConfig` (`crates/nlink/src/netlink/config/`) already gives
users `diff + apply + reconcile` for links, addresses, routes, and
qdiscs. **nftables is the glaring gap** — the highest-value
declarative-config surface still imperative-only. After Plan 150
adds flowtables and Plan 148 adds `NFT_TABLE_F_PERSIST`, every
nftables primitive nlink supports is mature enough to participate
in declarative reconciliation.

This plan mirrors `NetworkConfig`'s shape (`NftablesConfig::new()`,
`.diff(&conn).await?`, `diff.apply(&conn).await?`) for nftables —
atomic-via-Transaction commit included.

## 1. Summary

Three deliverables:

1. **`NftablesConfig`** declarative builder covering tables,
   chains, rules, flowtables, and sets. Maps and counters covered
   as `Option<…>` fields on the relevant objects (no top-level
   `.map()` builder yet — demand-gated).
2. **`NftablesDiff`** — the parallel of `ConfigDiff`. Categorized
   add/update/delete ops per object kind. `Display` summary like
   `ConfigDiff::summary()`.
3. **`NftablesDiff::apply(&conn)`** — atomic transactional install
   using the existing `Connection<Nftables>::transaction()` API.
   Per-object idempotency where it makes sense; "delete absent
   table" is success, not error.

Plus reconcile-mode (`apply_reconcile` with retry-on-conflict),
recipe (`docs/recipes/nftables-declarative-config.md`), and
CHANGELOG entry.

## 2. Goals

By end of plan:

- A firewall operator can declare their entire ruleset in Rust,
  run `cfg.diff(&conn).await?`, inspect the proposed changes
  (`.summary()`), and apply transactionally — either getting the
  whole new state or rolling back fully on any error.
- Re-running `apply` on the same `NftablesConfig` against an
  already-applied state is a no-op (empty diff).
- Adding one rule to an otherwise-unchanged config produces a diff
  with exactly one add op (not a full reset).
- Rule equivalence is deterministic enough for **CI-tested
  reproducibility** — running the same `NftablesConfig` from two
  callers produces byte-identical diffs against the same kernel
  state.
- Recipe walks through the canonical "load my firewall on boot,
  reconcile on config-file change" pattern with `notify` for the
  file-watch part.

## 3. Scope

### In

- New module `crates/nlink/src/netlink/nftables/config/`
  containing:
  - `mod.rs` — re-exports + module doc
  - `types.rs` — `NftablesConfig`, `DeclaredTable`,
    `DeclaredChain`, `DeclaredRule`, `DeclaredFlowtable`,
    `DeclaredSet` + their fluent builders
  - `diff.rs` — `NftablesDiff`, per-object change collections,
    `summary()`, `is_empty()`, `change_count()`
  - `apply.rs` — `NftablesDiff::apply` + `apply_reconcile`
- Public re-exports from `crates/nlink/src/lib.rs`:
  `pub use netlink::nftables::config::{NftablesConfig, NftablesDiff};`
- Integration tests:
  - `tests/integration/nftables_config.rs` covering all six
    standard scenarios (see §4.5 below)
  - All gated `require_root!()` + `require_module!("nf_tables")`
- Recipe + CHANGELOG entry.

### Out

- nftables maps (`nft add map ...`) — out for 0.16. Sets cover
  the most common case; maps add another dimension that's worth
  its own design pass.
- Named counters as top-level objects (counters embedded in rules
  ARE in scope). Named counters add the same dim as maps; defer.
- nftables objects of type `quota` / `synproxy` / `ct helper` /
  `ct expectation` — all demand-gated.
- "Diff against a YAML/TOML file" — that's a downstream concern.
  This plan gives the Rust API; serde derives on the declarative
  types come for free for users who want them.
- Cross-namespace reconciliation. Namespace-correctness is
  per-`Connection` as everywhere else in nlink.

## 4. Design

### 4.1 The user-facing shape

```rust
use nlink::{Connection, Nftables, NftablesConfig};
use nlink::nftables::{Family, Hook, Priority};

let cfg = NftablesConfig::new()
    .table("filter", Family::Inet, |t| t
        .persist(true)                            // NFT_TABLE_F_PERSIST
        .chain("input", |c| c
            .hook(Hook::Input).priority(Priority::Filter).policy(Policy::Drop)
            .rule(|r| r.match_iif("lo").accept())
            .rule(|r| r.match_ct_state(CtState::ESTABLISHED | CtState::RELATED).accept())
            .rule(|r| r.match_proto(Proto::Icmp).accept())
            .rule(|r| r.match_proto(Proto::Tcp).match_dport(22).accept()))
        .chain("forward", |c| c
            .hook(Hook::Forward).priority(Priority::Filter).policy(Policy::Drop)
            .rule(|r| r.match_iif("eth0").match_oif("eth1").accept()))
        .flowtable("ft", |f| f
            .device("eth0").device("eth1").hw_offload(true))
        .rule_chain("forward", |c| c
            .rule(|r| r.match_ct_state(CtState::ESTABLISHED).flow_offload("ft").accept())));

let conn = Connection::<Nftables>::new()?;
let diff = cfg.diff(&conn).await?;
println!("{}", diff.summary());
// "NftablesDiff: 1 table to add, 2 chains to add, 5 rules to add, 1 flowtable to add"

diff.apply(&conn).await?;   // atomic; rolls back on any failure
```

Builder closures keep the syntax tree-shaped, matching the existing
`NetworkConfig::link("eth0", |b| b.dummy().up())` convention.

### 4.2 Type design

```rust
// crates/nlink/src/netlink/nftables/config/types.rs

#[derive(Debug, Clone, Default)]
pub struct NftablesConfig {
    tables: Vec<DeclaredTable>,
}

#[derive(Debug, Clone)]
pub struct DeclaredTable {
    name: String,
    family: Family,
    flags: TableFlag,                  // bitflags (NFT_TABLE_F_PERSIST etc.)
    chains: Vec<DeclaredChain>,
    flowtables: Vec<DeclaredFlowtable>,
    sets: Vec<DeclaredSet>,
}

#[derive(Debug, Clone)]
pub struct DeclaredChain {
    name: String,
    /// Hook + priority + policy; None means "regular (non-base) chain".
    hook: Option<ChainHookSpec>,
    rules: Vec<DeclaredRule>,
}

#[derive(Debug, Clone)]
pub struct DeclaredRule {
    /// Canonicalized representation — see §4.3.
    body: RuleBody,
    /// Optional user-supplied stable key for diff matching. If
    /// absent, identity falls back to `RuleBody` byte equality.
    handle_key: Option<String>,
}

#[derive(Debug, Clone)]
pub struct DeclaredFlowtable {
    name: String,
    devs: Vec<String>,
    priority: i32,
    flags: FlowtableFlag,             // bitflags from Plan 150
}

#[derive(Debug, Clone)]
pub struct DeclaredSet {
    name: String,
    key_type: SetKeyType,             // ipv4_addr | ipv6_addr | inet_proto | ...
    elements: Vec<SetElement>,
    flags: SetFlag,                   // interval | constant | dynamic | ...
}
```

Builders consume `self` and return `Self` per project convention
(`pub fn name(mut self, ...) -> Self`). Tree closures are
boilerplate but type-safe: `chain(|c| c.hook(...).rule(|r| r...))`
gets full IDE completion.

### 4.3 The hard part — rule equivalence

`nft` rules can be semantically identical but **byte-different** on
the wire — register allocation choices, optional attributes the
kernel adds (`NFTA_RULE_COMPAT`), and so on. Naive byte-compare on
the wire form produces a stream of false-positive replaces.

A common temptation is "sort expressions by kind enum then
byte-compare." **This is wrong.** nftables expressions have data
dependencies: `match_iif("eth0")` lowers to `meta load iif → reg N`
followed by `cmp reg N, eth0_ifindex`. The load and the cmp
*cannot* be reordered without breaking the rule's semantics. So
canonicalization at the expression-bytecode layer is the wrong
abstraction.

**Correct approach**: canonicalize at the **typed-`Rule`** layer,
*before* lowering to expression bytecode. The typed `Rule`'s
matches and statements are independent at the API level — that's
where the commutativity assumption holds. Sort the typed `Match`
collection deterministically, then lower deterministically; the
emitted expression bytes are canonical by construction.

```rust
fn canonicalize_typed(rule: &Rule) -> Rule {
    let mut canon = rule.clone();
    // 1. Sort the typed match collection by a stable key.
    //    Matches are independent at the API level — sorting them
    //    is semantically safe; sorting EXPRESSIONS would not be.
    canon.matches.sort_by_key(Match::canonical_key);
    // 2. Stable register-allocation ordering: the lowering pass
    //    walks matches in order and allocates registers
    //    monotonically from Reg1. With matches sorted, register
    //    assignment is deterministic.
    // 3. Statement collection (verdict / nat / counter / etc.)
    //    keeps source order — statements are NOT commutative
    //    (counter-before-verdict ≠ verdict-before-counter).
    canon
}

fn canonical_bytes(rule: &Rule) -> Vec<u8> {
    let canon = canonicalize_typed(rule);
    let mut buf = MessageBuilder::scratch();
    canon.write_to(&mut buf);
    buf.into_bytes()
}
```

The canonical *bytes* (the output of `canonical_bytes`) are what
the diff actually compares; the typed-rule canonicalization step
is the substrate that makes the byte form deterministic.

The canonical form is computed once when the `DeclaredRule` enters
`NftablesConfig`; not re-derived on every diff call. Cached as
`Vec<u8>` on the struct.

**Kernel-side comparison**: dump the current rules via
`Connection<Nftables>::list_rules(table, family)`, parse each into
the typed `Rule`, run the same `canonicalize_typed` + `write_to`
pipeline, compare bytes. The pipeline is a pure function of the
typed `Rule`, independent of whether the struct came from the
user's declaration or from a kernel dump.

**`Match::canonical_key`** — the sort key for the type-level sort.
Implementation:

```rust
impl Match {
    fn canonical_key(&self) -> (u8, Vec<u8>) {
        // First byte: stable kind ordinal (Meta = 1, Payload = 2,
        // Ct = 3, ...). Second component: kind-specific bytes
        // (the operand bytes — IP, port number, ifname, etc.)
        // so two matches of the same kind sort by their value.
        match self {
            Match::Meta { key, op, value } =>
                (1, kind_value_bytes(*key, *op, value)),
            Match::Payload { base, offset, len, op, value } =>
                (2, payload_value_bytes(*base, *offset, *len, *op, value)),
            // ... etc per typed match variant ...
        }
    }
}
```

**Documented limitation**: kernels may add attributes
(`nft_log` flags, `NFTA_RULE_COMPAT` for legacy iptables-nft) that
the declared side doesn't carry. These produce false-positive
"declared rule differs from kernel rule" diffs, triggering harmless
re-apply (delete + add). Document in the diff module's rustdoc:

> ### Rule equivalence — known limitations
>
> Diff matching uses a canonicalized byte-comparison of the rule's
> expression bytecode. Two situations may produce false-positive
> "differs" diffs that trigger harmless re-applies:
>
> 1. The kernel attached `NFTA_RULE_COMPAT` (set on rules
>    originally added via `iptables-nft`). Declared rules never
>    carry compat; if the dump shows one, it's treated as
>    "different" and gets replaced with a clean compat-less rule.
> 2. Hypothetical future kernel attributes that nlink's parser
>    doesn't recognize. These are ignored on parse, so the
>    canonical form omits them, while the dump's bytes carry them.
>
> Re-apply is idempotent (delete + add in one transaction).
> False-positive cost is one transaction round-trip per affected
> rule, no data loss.

### 4.4 Diff structure

```rust
// crates/nlink/src/netlink/nftables/config/diff.rs

#[derive(Debug, Clone, Default)]
pub struct NftablesDiff {
    pub tables_to_add: Vec<DeclaredTable>,
    pub tables_to_delete: Vec<TableId>,
    pub chains_to_add: Vec<(TableId, DeclaredChain)>,
    pub chains_to_delete: Vec<ChainId>,
    pub chains_to_replace: Vec<(ChainId, DeclaredChain)>,
    pub rules_to_add: Vec<(ChainId, DeclaredRule)>,
    pub rules_to_delete: Vec<RuleHandle>,   // kernel-assigned handle
    pub flowtables_to_add: Vec<(TableId, DeclaredFlowtable)>,
    pub flowtables_to_delete: Vec<FlowtableId>,
    pub flowtables_to_replace: Vec<(FlowtableId, DeclaredFlowtable)>,
    pub sets_to_add: Vec<(TableId, DeclaredSet)>,
    pub sets_to_delete: Vec<SetId>,
    pub sets_to_replace: Vec<(SetId, DeclaredSet)>,
    /// Set elements diffed independently — adding/removing entries
    /// from an existing set is much cheaper than replacing the set.
    pub set_elements_to_add: Vec<(SetId, Vec<SetElement>)>,
    pub set_elements_to_delete: Vec<(SetId, Vec<SetElement>)>,
}

impl NftablesDiff {
    pub fn is_empty(&self) -> bool { /* ... */ }
    pub fn change_count(&self) -> usize { /* ... */ }
    pub fn summary(&self) -> String { /* matches ConfigDiff::summary shape */ }
}

#[derive(Debug, Clone, PartialEq, Eq, Hash)]
pub struct TableId(pub Family, pub String);

#[derive(Debug, Clone, PartialEq, Eq, Hash)]
pub struct ChainId { pub table: TableId, pub name: String }

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub struct RuleHandle(pub u64);   // kernel-assigned, returned by list_rules
```

Object IDs (`TableId`, `ChainId`, `FlowtableId`, `SetId`) are
typed newtypes so diff entries are self-describing — no `(String,
String, Family)` tuples leaking into user code.

### 4.5 Apply — transactional commit

```rust
impl NftablesDiff {
    /// Apply the diff atomically via a single nft transaction.
    /// All ops succeed together or none do (kernel-side rollback
    /// on any per-op failure).
    pub async fn apply(&self, conn: &Connection<Nftables>) -> Result<()> {
        let mut txn = conn.transaction();
        // Order matters: table > chain > rule, delete > add per kind
        // (per nft's accepted ordering for atomic commits).
        for tid in &self.tables_to_delete {
            txn = txn.del_table(&tid.1, tid.0);
        }
        for table in &self.tables_to_add {
            txn = txn.add_table(&table.name, table.family);
            // (chains + flowtables + sets attach to fresh table)
        }
        for cid in &self.chains_to_delete { txn = txn.del_chain(...); }
        for (tid, chain) in &self.chains_to_add { txn = txn.add_chain(...); }
        // ... rest ...
        txn.commit(conn).await
    }

    /// Apply with retry on transient conflicts (EAGAIN / EBUSY).
    /// Useful when other processes are mutating the ruleset
    /// concurrently. Bounded retry; falls through to error on
    /// real failures.
    pub async fn apply_reconcile(
        &self,
        conn: &Connection<Nftables>,
        opts: ReconcileOptions,
    ) -> Result<ReconcileReport> { /* ... */ }
}

#[derive(Debug, Clone)]
pub struct ReconcileOptions {
    pub max_retries: usize,    // default 3
    pub backoff: Duration,     // default 100ms, exponential
}

#[derive(Debug, Clone, Default)]
pub struct ReconcileReport {
    pub attempts: usize,
    pub final_diff: NftablesDiff,
}
```

The `apply` method **fully reuses** the existing `Transaction`
plumbing (`crates/nlink/src/netlink/nftables/connection.rs:747`).
No new wire-level code; only orchestration.

### 4.6 Six standard reconciliation scenarios — integration test matrix

Every integration test covers one canonical scenario:

| # | Initial state | Declared state | Expected diff |
|---|---|---|---|
| 1 | Empty | One table + one chain + 3 rules | adds: 1 table, 1 chain, 3 rules |
| 2 | Cfg from #1 applied | Same Cfg | empty (no-op) |
| 3 | Cfg from #1 applied | Same Cfg + 1 extra rule | adds: 1 rule |
| 4 | Cfg from #1 applied | Cfg with 1 rule removed | deletes: 1 rule |
| 5 | Cfg from #1 applied | Cfg with chain's `policy` changed | replaces: 1 chain |
| 6 | Cfg from #1 applied | Empty Cfg | deletes: 1 table (cascade) |

Test names map directly: `reconcile_empty_to_full`,
`reconcile_idempotent_reapply`, `reconcile_add_one_rule`, etc.

Each test runs in its own `LabNamespace` for isolation.

## 8. Risks

- **Rule equivalence false-positives**: documented in §4.3 as
  acceptable (harmless re-apply, no data loss). If users hit
  excessive churn, refine in 0.17 with a third normalization pass.
- **Transaction-size limits**: kernel's atomic-commit limit is in
  number of operations (`NFNL_MAX_BATCH = 1024` historically;
  bumped to higher in recent kernels). A diff that exceeds this
  needs splitting. Mitigation: in `apply()`, if the proposed
  transaction has > 512 ops, split into multiple commits (lose
  atomicity at split boundaries; document). Real-world: only
  matters for sites with thousands of rules.
- **Concurrent mutation race**: between `diff()` reading current
  state and `apply()` committing, another process can mutate.
  Result: apply may fail or produce a state different from what
  the diff predicted. Mitigation: `apply_reconcile` retries on
  EBUSY/EAGAIN; documented invariant is "diff is a *plan*, apply
  is what actually happens — re-run diff after apply if you need
  to confirm convergence."
- **`list_rules` dump cost**: on hosts with 10k+ rules, the dump
  is non-trivial (multi-MB response). Streaming dump from Plan
  149 (`stream_rules`) is the right primitive — once it lands,
  switch the diff path to use it instead of the eager
  `list_rules`. Document in §3 "Out" that diff still uses eager
  list in 0.16; streaming swap is a follow-up.
- **`Transaction` API coverage gaps**: the existing
  `Transaction` (`connection.rs:747`) supports add/del table,
  add/del chain, add/del rule. Need to verify it also supports
  add/del set, add/del flowtable, add/del set-elements. If gaps,
  Phase 4 has to extend `Transaction` before `apply` can wire
  everything. **Action: audit `Transaction` coverage during
  Phase 1; flag any gap as a Phase 0 prereq.**
