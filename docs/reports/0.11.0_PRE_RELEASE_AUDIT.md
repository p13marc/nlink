# nlink 0.11.0 Pre-Release API Audit

**Date:** 2026-03-28
**Scope:** Full API consistency review, breaking change proposals, feature ideas
**Based on:** Codebase audit, nlink-lab feedback, Rust API guidelines

---

## Executive Summary

The nlink API is **good overall** -- return types are consistent, builder patterns are uniform, protocol state types are well-designed, and naming follows netlink conventions. However, there is one major inconsistency (interface parameter types) that affects ~40 public methods, plus several smaller issues worth fixing before a minor release.

This document proposes **4 breaking changes** and **8 non-breaking improvements** for 0.11.0.

---

## API Consistency Audit

### What's Already Excellent

| Area | Status | Details |
|------|--------|---------|
| Get return types | Consistent | `Option<T>` for singles, `Vec<T>` for collections |
| Builder pattern | Consistent | All builders return `Self`, all use method chaining |
| Config trait naming | Consistent | `LinkConfig`, `QdiscConfig`, `AddressConfig`, `RouteConfig`, `FilterConfig` |
| Protocol state exports | Excellent | All 17 protocol types exported at crate root |
| Feature flag coverage | Correct | `sockdiag`, `tuntap`, `tc`, `output` properly gated |
| Naming verbs | Mostly consistent | `add_*`/`del_*`/`replace_*`/`set_*`/`get_*`/`flush_*` |

### What Needs Fixing

#### 1. Interface Parameter Types (Major Inconsistency)

Three incompatible patterns coexist for the same concept (identifying a network interface):

| Pattern | Count | Example Methods |
|---------|-------|----------------|
| `impl Into<InterfaceRef>` | ~15 methods | `set_link_up`, `set_link_down`, `set_link_mtu`, `del_link`, `set_link_master`, `enslave` |
| `&str` (name only) | ~22 methods | `add_qdisc`, `del_qdisc`, `add_class`, `del_class`, `add_filter`, `del_fdb`, `del_bridge_vlan`, `flush_fdb`, `add_address_by_name` |
| `u32` (index only) | `*_by_index` variants | `add_qdisc_by_index`, `del_link_by_index`, etc. |

The `impl Into<InterfaceRef>` pattern is the most ergonomic -- it accepts `&str`, `String`, and `u32` transparently. But the majority of methods don't use it.

**Affected methods in `tc.rs`:**
- `add_qdisc(dev: &str, ...)` / `del_qdisc(dev: &str, ...)` / `change_qdisc(dev: &str, ...)`
- `replace_qdisc(dev: &str, ...)` / `add_class(dev: &str, ...)` / `del_class(dev: &str, ...)`
- `change_class(dev: &str, ...)` / `replace_class(dev: &str, ...)`
- `add_filter(dev: &str, ...)` / `del_filter(dev: &str, ...)` / `flush_filters(dev: &str, ...)`
- `get_tc_chains(ifname: &str, ...)`

**Affected methods in `addr.rs`:**
- `del_address(ifname: &str, ...)` / `del_address_v4(ifname: &str, ...)`
- `del_address_v6(ifname: &str, ...)` / `add_address_by_name(ifname: &str, ...)`
- `replace_address_by_name(ifname: &str, ...)` / `flush_addresses(ifname: &str)`

**Affected methods in `fdb.rs`:**
- `get_fdb(bridge: &str)` / `get_fdb_for_port(bridge: &str, port: &str)`
- `del_fdb(dev: &str, ...)` / `flush_fdb(bridge: &str)`

**Affected methods in `bridge_vlan.rs`:**
- `get_bridge_vlans(dev: &str)` / `set_bridge_pvid(dev: &str, ...)`
- `add_bridge_vlan_tagged(dev: &str, ...)` / `del_bridge_vlan(dev: &str, ...)`
- `add_bridge_vlan_range(dev: &str, ...)` / `del_bridge_vlan_range(dev: &str, ...)`
- `del_vlan_tunnel(dev: &str, ...)` / `get_vlan_tunnels(dev: &str)`

#### 2. Naming Inconsistency: `remove_netem` vs `del_*`

Every deletion method uses `del_*` except two:
- `remove_netem(iface)` -- should be `del_netem(iface)`
- `remove_netem_by_index(ifindex)` -- should be `del_netem_by_index(ifindex)`

#### 3. Error Context Coverage

Only 5 methods wrap errors with `KernelWithContext`:
- `add_link` (link.rs)
- `del_link_by_index` (connection.rs)
- `set_link_state_by_index` (connection.rs)
- `del_qdisc_by_index_full` (tc.rs)
- `change_qdisc_by_index_full` (tc.rs)

All other mutation operations (`add_address`, `del_address`, `add_route`, `del_route`, `add_fdb`, `del_fdb`, `add_bridge_vlan`, etc.) return raw kernel errors without operation context.

#### 4. `Send + Sync` Bounds on Config Traits

Inconsistent across traits:

| Trait | `Send + Sync` required? |
|-------|------------------------|
| `QdiscConfig` | Yes |
| `ClassConfig` | Yes |
| `FilterConfig` | Yes |
| `ActionConfig` | Yes |
| `LinkConfig` | **No** |
| `AddressConfig` | **No** |
| `RouteConfig` | **No** |
| `NeighborConfig` | **No** |

This means `LinkConfig` implementations can't be sent across async task boundaries, while `QdiscConfig` implementations can.

#### 5. `LinkStats` Fields Are `pub(crate)`

All fields (`rx_packets`, `tx_packets`, `rx_bytes`, `tx_bytes`, `rx_errors`, `tx_errors`, `rx_dropped`, `tx_dropped`, `multicast`, `collisions`) are `pub(crate)` with public accessor methods. There's no invariant to protect -- these are simple `u64` counters from the kernel.

Consumers doing metric conversion must call a method per field:
```rust
// Current: verbose
metrics.rx_errors = stats.rx_errors();
metrics.tx_errors = stats.tx_errors();

// With public fields: cleaner
metrics.rx_errors = stats.rx_errors;
```

## Proposed Breaking Changes for 0.11.0

### B1. Unify Interface Parameters to `impl Into<InterfaceRef>`

**Impact:** ~22 methods change signature from `&str` to `impl Into<InterfaceRef>`
**Migration for callers:** None in most cases -- `&str` implements `Into<InterfaceRef>` via `From<&str>`, so existing code compiles unchanged. Only callers relying on turbofish or explicit type annotations may need adjustment.

**Approach:**
- Change all `dev: &str` / `ifname: &str` parameters to `dev: impl Into<InterfaceRef>`
- Keep `*_by_index` variants for namespace-safe operations (documented)
- Remove `add_address_by_name` / `replace_address_by_name` -- they become redundant since the base methods now accept names directly

### B2. Rename `remove_netem` to `del_netem`

**Impact:** 2 methods renamed
**Migration:** Find-and-replace `remove_netem` -> `del_netem`

### B3. Make `LinkStats` Fields Public

**Impact:** `pub(crate)` -> `pub` on 10 fields
**Migration:** None -- adding visibility is backwards-compatible for consumers, but technically the struct layout commitment changes.

### B4. Add `Send + Sync` Bounds to All Config Traits

**Impact:** `LinkConfig`, `AddressConfig`, `RouteConfig`, `NeighborConfig` gain `Send + Sync` requirement
**Migration:** Most implementations already satisfy this. Only those holding `Rc` or non-Send types would break (unlikely for network config structs).

## Proposed Non-Breaking Improvements for 0.11.0

### N1. Add `#[non_exhaustive]` to Public Enums

Critical for semver stability. Without this, adding a variant is a breaking change.

**Enums to annotate:**
- `NetworkEvent` (most important -- frequently matched)
- `Error`
- `OperState`
- `BondMode`, `MacvlanMode`, `IpvlanMode`
- `EthtoolEvent`, `Nl80211Event`, `DevlinkEvent`
- `NamespaceEvent`, `NamespaceNetlinkEvent`
- `SELinuxEvent`, `ProcEvent`
- `QdiscOptions`

### N2. Expand Error Context to All Mutation Operations

Add `.with_context()` wrapping to every `send_ack()` call in mutation methods. Pattern:

```rust
self.send_ack(builder).await
    .map_err(|e| e.with_context(format!("add_route({})", dest)))
```

**Methods to update:** ~30-40 across `addr.rs`, `route.rs`, `fdb.rs`, `bridge_vlan.rs`, `tc.rs`, `neigh.rs`, `rule.rs`, `nexthop.rs`

### N3. Add `prelude` Module

```rust
// nlink::prelude re-exports the most common types
pub use crate::{Connection, Error, Result, Route, NetworkEvent, RtnetlinkGroup};
pub use crate::netlink::namespace;
pub use crate::netlink::link::{DummyLink, VethLink, BridgeLink, BondLink, VlanLink, VxlanLink};
pub use crate::netlink::tc::NetemConfig;
pub use crate::netlink::route::{Ipv4Route, Ipv6Route};
pub use crate::netlink::nftables::{Rule, Family, Chain, ChainType, Hook, Priority, Policy};
```

### N4. Add `#[must_use]` on Builders

All builder types should have `#[must_use = "builders do nothing unless passed to a Connection method"]` to catch cases where a builder is constructed but never used:

```rust
// This currently compiles silently but does nothing:
VethLink::new("a", "b").mtu(9000);
// With #[must_use], the compiler warns.
```

### N5. Newtype for `Ifindex`

Replace raw `u32` with `struct Ifindex(pub u32)` for interface indices. This prevents accidentally passing a TC handle where an ifindex is expected, and vice versa.

```rust
pub struct Ifindex(pub u32);

impl From<u32> for Ifindex { ... }
impl From<Ifindex> for u32 { ... }
```

This is technically a breaking change if done for public APIs, but could be introduced gradually by accepting `impl Into<Ifindex>` which still accepts `u32`.

### N6. `serde` Feature Flag

Add optional `Serialize`/`Deserialize` derives on public data types behind a `serde` feature flag:

```toml
[features]
serde = ["dep:serde"]
```

**Types to annotate:** `LinkMessage`, `AddressMessage`, `RouteMessage`, `TcMessage`, `NeighborMessage`, `NetworkEvent`, `OperState`, `LinkStats`, diagnostics types.

This is highly valuable for monitoring tools that export state as JSON.

### N7. Unified Async Constructor

Currently `Connection::<Route>::new()` is sync, `Connection::<Wireguard>::new_async().await` is async. Since this is a tokio-native library, a unified async constructor would be cleaner:

```rust
// Unified:
let conn = Connection::<Route>::open().await?;     // resolves immediately
let wg = Connection::<Wireguard>::open().await?;    // resolves GENL family
```

The sync `new()` can remain for backward compatibility on `Default` types.

### N8. Interface Name Validation on All Paths

Currently only `add_link` validates interface names. Extend validation to `set_link_name`, `add_qdisc` (the dev parameter), `add_filter`, `del_fdb`, and any other method that takes an interface name string before sending to kernel.

## Priority Matrix

| # | Item | Breaking? | Effort | Impact |
|---|------|-----------|--------|--------|
| B1 | Unify interface params to `InterfaceRef` | Yes | Medium | High -- fixes the biggest inconsistency |
| N1 | `#[non_exhaustive]` on enums | Soft | Low | High -- critical for semver |
| B2 | `remove_netem` -> `del_netem` | Yes | Trivial | Low -- naming consistency |
| B3 | Public `LinkStats` fields | Yes | Trivial | Medium -- unblocks metric consumers |
| N2 | Error context on all mutations | No | Medium | High -- eliminates ~90 map_err wrappers |
| B4 | `Send + Sync` on all config traits | Yes | Low | Medium -- consistency + async safety |
| N3 | `prelude` module | No | Low | Medium -- better onboarding |
| N4 | `#[must_use]` on builders | No | Low | Low -- catches bugs |
| N5 | `Ifindex` newtype | Soft | Medium | Medium -- type safety |
| N6 | `serde` feature flag | No | High | High -- enables JSON export |
| N7 | Unified async constructor | No | Medium | Medium -- cleaner API |
| N8 | Name validation on all paths | No | Low | Medium -- better errors |

---

## Recommended Execution Order

**Phase 1 (breaking changes):** B1, B2, B3, B4 -- do all breaking changes together
**Phase 2 (hardening):** N1, N2, N4, N8 -- non_exhaustive, error context, must_use, validation
**Phase 3 (ergonomics):** N3, N5, N7 -- prelude, Ifindex, unified constructor
**Phase 4 (features):** N6 -- serde (can be done independently, larger scope)
