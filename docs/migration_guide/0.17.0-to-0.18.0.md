# Upgrading from nlink 0.17.0 to 0.18.0

0.18.0 is a **purely additive** release driven by the
upstream-asks report from nlink-lab (the canonical downstream
consumer). Every change is either a new method, a new variant
on a `#[non_exhaustive]` enum, or a wire-format extension that
the kernel ignores when unset — no public-API breaks for code
that compiled against 0.17.0.

The one nuance: **`ChainInfo` gained `#[non_exhaustive]`** for
the new `device: Option<String>` field. The only construction
site is internal (`parse_chain`), so downstream code never
constructed `ChainInfo { ... }` literals; the attribute is
defensive forward-compat. If you *were* destructuring
`ChainInfo` via `let ChainInfo { name, table, family, .. } =
info`, your code keeps working unchanged because the rest
pattern (`..`) already swallows any new fields.

## What's new

### Plan 180 — `chain_type` + `device` on declarative chains

`DeclaredChainBuilder` gained the two setters the imperative
`Chain` builder already had:

```rust
use nlink::netlink::nftables::config::NftablesConfig;
use nlink::netlink::nftables::types::{ChainType, Family, Hook, Priority};

let cfg = NftablesConfig::new()
    .table("nat", Family::Inet, |t| {
        t.chain("prerouting", |c| {
            c.hook(Hook::Prerouting)
                .priority(Priority::Filter)
                .chain_type(ChainType::Nat)   // <- new
        })
    })
    .table("netdev", Family::Netdev, |t| {
        t.chain("ingress", |c| {
            c.hook(Hook::Ingress)
                .priority(Priority::Filter)
                .device("eth0")              // <- new
        })
    });
```

Without `chain_type(ChainType::Nat)`, a NAT chain hooking
`prerouting`/`postrouting` defaulted to `ChainType::Filter` —
any `masquerade`/`snat`/`dnat` verdict then refused to load
with `EOPNOTSUPP`. `device(...)` binds a netdev base chain to a
specific interface (`type filter hook ingress device eth0
priority -150`). Both setters mirror the imperative builders
exactly.

The `Chain::device(...)` imperative setter was added at the same
time. `ChainInfo` now exposes:
- `chain_type: Option<ChainType>` — typed; mapped from the
  kernel's `"filter"`/`"nat"`/`"route"` strings via
  `ChainType::from_kernel_string`. Unknown values yield `None`.
- `device: Option<String>` — raw, since netdev binds are
  arbitrary interface names.

If you wrote `info.chain_type == Some("nat".into())`, update to
`info.chain_type == Some(ChainType::Nat)`. Match arms on the
typed enum keep working unchanged.

### Plan 181 — `list_*_in` server-side filtered dumps

Four new methods on `Connection<Nftables>` that mirror the
existing `list_rules(table, family)` shape:

```rust
let tables  = nft.list_tables_in(Family::Inet).await?;
let chains  = nft.list_chains_in("filter", Family::Inet).await?;
let fts     = nft.list_flowtables_in("filter", Family::Inet).await?;
let sets    = nft.list_sets_in("filter", Family::Inet).await?;
```

Each emits the matching `NFTA_*_TABLE` attribute + `nfgen_family`
filter on the dump request. The kernel returns only matching
entities, so this is more efficient than
`list_chains().filter(|c| c.table == "filter")` on hosts with
many tables. The unfiltered counterparts (`list_chains()`,
etc.) keep working unchanged.

**Known quirk**: the kernel ignores `NFTA_CHAIN_TABLE` /
`NFTA_FLOWTABLE_TABLE` / `NFTA_SET_TABLE` on *dump* requests
(only on single-`get` requests), so nlink applies a defensive
client-side `retain()` filter post-dump to ensure the
contract holds across kernel versions. Discovered during
Plan 181's CI cycle; harmless but worth knowing about.

### Plan 182 — `Error::ext_ack()` accessors

The kernel-error variants (`Kernel`, `KernelWithContext`) gained
two inherent accessors that elide the 5-line `match | _ =>`
ceremony forced by `#[non_exhaustive]`:

```rust
match conn.add_link(spec).await {
    Err(e) => {
        if let Some(msg) = e.ext_ack() {
            eprintln!("kernel says: {msg}");
        }
        if let Some(off) = e.ext_ack_offset() {
            eprintln!("attribute offset: {off}");
        }
    }
    Ok(()) => {}
}
```

### Plan 183 — `Display` for diff types

```rust
println!("{diff}");                  // works for both NftablesDiff
println!("{}", network_diff);        //   and ConfigDiff
```

Wraps the existing `summary()` methods, so the rendered output
matches `diff.summary()` byte-for-byte.

### Plan 184 — `Ipv4Route::default_route()` constructors

```rust
let v4 = Ipv4Route::default_route().gateway("192.0.2.1");
let v6 = Ipv6Route::default_route().gateway("2001:db8::1");
```

Equivalent to `Ipv4Route::new("0.0.0.0", 0)` /
`Ipv6Route::new("::", 0)`; pick whichever reads better in
context.

### Plan 185 — ENOBUFS-resilient nftables watching

The headline feature. `Connection<Nftables>` gained two new
methods that mirror `kube_rs::watcher(api, cfg) -> Stream`:

```rust
use std::sync::Arc;
use nlink::netlink::{Connection, Nftables};
use nlink::netlink::nftables::NftablesEvent;
use nlink::netlink::resync::{ConnectionFactory, ResyncedEvent, ResyncMarker};
use tokio_stream::StreamExt;

let factory: ConnectionFactory<Nftables> = Arc::new(|| {
    Box::pin(async { Connection::<Nftables>::new() })
});

let conn = Connection::<Nftables>::new()?;
let mut events = conn.into_events_with_resync(factory)?;

while let Some(item) = events.next().await {
    match item? {
        ResyncedEvent::Event(NftablesEvent::NewTable(t)) => { /* ... */ }
        ResyncedEvent::Marker(ResyncMarker::ResyncStart) => { /* drop incremental state */ }
        ResyncedEvent::Resynced(ev) => { /* rebuild from snapshot */ }
        ResyncedEvent::Marker(ResyncMarker::ResyncEnd) => { /* state is consistent */ }
        _ => {}
    }
}
```

When the kernel drops events under pressure (`-ENOBUFS`), the
wrapper opens a fresh connection via the factory, walks the
ruleset, and emits the snapshot as `Resynced(...)` items between
`ResyncStart` / `ResyncEnd` markers. The fresh connection is
required — Plan 178 documented the "subscribe + unicast on the
same socket" race that motivates not reusing the subscribe
socket for the snapshot.

`into_events_with_resync` consumes `self` and returns a
`'static + Send` stream that's `tokio::spawn`-friendly.
`subscribe_all_with_resync(&mut self)` is the borrowed sibling
when you need to keep the connection around for ad-hoc queries.

Bundled with this plan: `NftablesEvent::NewSet(SetInfo)` +
`DelSet(SetInfo)` variants — the snapshot enumerates sets, so
drift-detection consumers need the matching event variants for
live updates.

Recipe: [`docs/recipes/nftables-watch-with-resync.md`].

### Internal-only — `events_with_resync` lifetime-generic

`nlink::netlink::resync::events_with_resync`'s snapshot-future
bound went from `Send + 'static` to `Send + 'a`. Existing
call sites with `'static` closures keep working unchanged via
lifetime elision. The change unlocks the borrowed
`subscribe_all_with_resync` form whose snapshot future doesn't
need to outlive the caller's stack.

If you implemented your own wrapper using `events_with_resync`,
you don't need to do anything — the `'a` parameter defaults
back to `'static` wherever you wrote `'static` before.

## No deprecations

0.18 doesn't deprecate anything. The cycle's theme was
downstream-driven additions, not API consolidation.
