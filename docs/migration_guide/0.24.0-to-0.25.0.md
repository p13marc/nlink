# Upgrading from 0.24.0 to 0.25.0

0.25.0 is a **bug-fix release**. Almost everything in it makes nlink do what
it always said it did, so most upgrades need no code changes at all.

But several of the fixes change what your existing code **does**, with no
compiler error to tell you — because the old behaviour was wrong, and correcting
it necessarily changes the result. Read the sections below even if you read
nothing else.

Sections 1–3 change the meaning of values you already pass. Sections 4–7 change
what nlink sends to the kernel for the same input.

---

## ⚠️ Silent behaviour changes — read these

### 1. `mbps` / `kbps` / `gbps` now mean BYTES per second (8x)

**Affects:** any tc-style rate *string* — `Rate::parse`, `"…".parse::<Rate>()`,
every `parse_params` that takes a rate, `nlink-tc` command lines, and any
config file or preset that round-trips one.

nlink parsed the whole `bps` family as **bits** per second. `tc(8)` matches its
suffix table with `strcasecmp`, so lowercase `mbps` hits the `MBps = 8_000_000`
entry — mega**bytes**. nlink was 8x low on every one of them.

```rust
// 0.24 and earlier
Rate::parse("100mbps")?  ==  Rate::mbit(100)    // 100 Mbit — wrong

// 0.25
Rate::parse("100mbps")?  ==  Rate::mbit(800)    // 100 MByte/s = 800 Mbit
```

**What to do:** grep your configs and call sites for the `bps` spellings.

- If you *meant* megabits (most people did — that's what nlink gave you),
  change the string to `mbit`. Same value, no surprise.
- If you meant megabytes, you were silently getting 1/8th of it and this fix
  gives you what you asked for.

Typed constructors (`Rate::mbit(100)`, `Rate::bytes_per_sec(n)`) are unaffected
and never had this problem. **Prefer them** — the string grammar exists for
`tc(8)` compatibility, not for new code.

### 2. A bare number in a time slot now means MICROseconds

**Affects:** `parse_params` on any config with a time-valued token — most
visibly `netem delay`.

`tc(8)`'s internal time unit is `TIME_UNITS_PER_SEC` = 1e6, so a suffix-less
number is microseconds. nlink read it as **seconds** — a 1,000,000x error.

```
nlink-tc qdisc add dev eth0 root netem delay 100
```

| | interpretation |
|---|---|
| 0.24 | **100 seconds** of delay — the interface effectively stops passing traffic |
| 0.25 | 100 microseconds — what `tc` does |

**What to do:** if you relied on the old behaviour you were almost certainly
already writing an explicit suffix (`delay 100ms`), which is unchanged. Bare
numbers meaning seconds was not a thing anyone deliberately depended on.

`m` / `min` / `h` are **no longer accepted** as time suffixes. `tc(8)` has no
minute suffix, and mapping `m` to minutes made a mistyped `ms` silently 60,000x
too long. Spell it `s` / `ms` / `us` / `ns`.

### 3. HTB / TBF / police bursts are now correct — your shaping will change

**Affects:** every `TbfConfig`, `HtbClassConfig`, `PoliceAction`, and therefore
every `RateLimiter`, `PerHostLimiter` and `PerPeerImpairer`.

nlink wrote byte counts and microsecond values into wire fields the kernel reads
as **psched ticks**. The token buckets were wrong by factors of 15.625 to 125.
See the CHANGELOG for the full accounting; the short version:

- **TBF** dropped every packet over ~262 bytes at `rate 1mbit` with the default
  burst. If you had a working TBF config, you had worked *around* this.
- **HTB** classes got a token bucket below one MTU, so they under-ran their
  configured rate with heavy added latency.
- **police** with a rate **could not install at all** (the kernel rejected the
  action for a missing rate table).

**What to do:** re-check any burst values you hand-tuned. If you compensated for
the old behaviour by inflating a burst, that compensation is now double-counted.
Bursts you left at the default are simply correct now.

### 4. `NftablesConfig::apply` no longer deletes undeclared tables

**Affects:** `NftablesConfig::diff` + `apply`, and `apply_reconcile`.

This one changes behaviour in the **safe** direction, and you almost certainly
want the new behaviour. It is listed here because if you were *relying* on
full-reconcile semantics, they are now opt-in.

`diff()` used to schedule **every table in the kernel** not named in your config
for deletion, and `apply()` committed it. `NFT_MSG_DELTABLE` cascades to all
chains, rules and sets in the table, and `list_tables()` is unscoped — so on a
host running Docker, firewalld or libvirt, applying a config with one table
destroyed the box's entire firewall in a single atomic commit.

```rust
// 0.25: foreign tables survive. This is now safe on a real host.
cfg.diff(&conn).await?.apply(&conn).await?;

// Opt in to the old behaviour — and look at the diff first.
use nlink::netlink::nftables::config::NftDiffOptions;
let diff = cfg
    .diff_with_options(&conn, &NftDiffOptions::default().purge_tables(true))
    .await?;
println!("{diff}");           // the `-` lines are tables about to be destroyed
diff.apply(&conn).await?;
```

**What to do:** nothing, unless you wanted the deletions. Purge remains unscoped
— it cannot tell your tables from Docker's — so only enable it where you own
every table in the namespace.

### 5. Rules now install in the order you declared them

**Affects:** `Connection::add_rule`, `Transaction::add_rule`, and every
declarative apply.

`NLM_F_APPEND` was never set, so the kernel **prepended** each rule. Rules
landed in reverse declaration order, and since nftables is first-match-wins,
that inverted your policy:

```yaml
rules:
  - { match: "tcp dport 22", action: accept }
  - { match: "",             action: drop   }
```

| | kernel order |
|---|---|
| 0.24 | `drop`, `accept` — **SSH blocked** |
| 0.25 | `accept`, `drop` — what you wrote |

**What to do:** if you reversed your rule lists to compensate, un-reverse them.

### 6. Filters now install with the protocol and priority you asked for

**Affects:** `add_filter`, `add_filter_by_index`, `replace_filter`,
`replace_filter_by_index`.

Every filter config has `.protocol()` and `.priority()` setters. Nothing read
them — `add_filter` hardcoded `ETH_P_IP` and priority 0 (kernel auto-assign).

```rust
conn.add_filter(dev, TcHandle::CLSACT,
    MatchallFilter::new().protocol(ETH_P_ALL).priority(1)).await?;
```

| | what got installed |
|---|---|
| 0.24 | `protocol ip`, priority auto — IPv6/ARP/VLAN never matched |
| 0.25 | `protocol all`, priority 1 — what you wrote |

**What to do:** if you were compensating for the old behaviour (e.g. installing
a second filter to catch IPv6), you can drop the workaround. Note
`MatchallFilter` and `FlowerFilter` **default** to `ETH_P_ALL`, so a filter you
never called `.protocol()` on now matches all traffic rather than IPv4 only.
That is what those builders always claimed. If you genuinely want IPv4-only,
say `.protocol(0x0800)`.

The `FilterConfig` trait gains `protocol()` and `priority()`. Both are
**defaulted** (`None`), so an external `FilterConfig` impl keeps compiling.

### 7. `TcMessage::is_filter()` / `is_class()` now need the message type

**Affects:** anyone calling these predicates. (Nothing inside nlink did.)

`is_filter()` used to test `tcm_info != 0`. For a qdisc the kernel puts the
**refcount** there — always ≥ 1 — so **every qdisc reported `is_filter() ==
true`**, and `filter_protocol()` returned refcount bits as an ethertype.

`struct tcmsg` is byte-identical for qdiscs, classes and filters; only
`nlmsg_type` distinguishes them, and it lives in the netlink header.
`TcMessage` now carries it, and the predicates read it.

**What to do:** messages you got from `get_qdiscs()` / `get_classes()` /
`get_filters()` / `events()` are classified correctly with no change on your
side. Only a `TcMessage` you built yourself, or parsed from a bare payload with
no header, has an unknown type — and there all three predicates return `false`
rather than guessing. `is_qdisc()` is new.

## Additions

- **`nlink::netlink::psched`** — the packet-scheduler clock. `Psched` (cached
  `/proc/net/psched` parameters), `tc_calc_xmittime` / `tc_calc_xmitsize` /
  `tc_calc_rtable`. Ports of iproute2's `tc_core.c`. You need
  `tc_calc_xmitsize` if you read a raw tick value back off the wire yourself;
  the typed decode paths now convert for you.
- `get_size` accepts the binary `kib` / `mib` / `gib` / `tib` spellings its own
  rustdoc had always advertised.
- **`TcMessage::stats_basic_hw()`** — the hardware-offloaded *subset* of the
  basic stats (`TCA_STATS_BASIC_HW`). It used to silently overwrite the
  software total, so on an offloading NIC `bytes()`/`packets()` reported only
  the offloaded portion — often **0**. They now return the software total, and
  the hw subset has its own accessor. Not a counter to add to the total: it is
  the part of that same total the NIC counted in hardware.
- **`TcHtbGlob::with_r2q`**, and `HtbQdiscConfig::r2q` now actually reaches the
  wire (it was hardcoded to 10).
- **`TcMessage::is_qdisc()`** and `TcMessage::msg_type()`.
- **`FilterConfig::protocol()` / `::priority()`** (defaulted), and
  `filter::DEFAULT_FILTER_PROTOCOL`.

## Stricter errors (previously silent)

- Negative rates and sizes now **error** instead of saturating to `0`.
  `get_rate("-5mbit")` used to return `Ok(0)`, which the kernel then rejected
  with a bare `EINVAL` instead of nlink's contract-mandated
  `"htb: invalid rate ..."`. A negative *duration* used to **panic** outright.

## Doc fixes worth knowing about

- `Percent::from_str`'s doc claimed `"0.5"` was "treated as a fraction". It never
  was — the code reads it as **half of one percent**, which is the `tc(8)`-correct
  behaviour. The doc was wrong, not the code. If you believed the doc, you have a
  100x error in your loss/corruption rates. Use `Percent::from_fraction(0.5)` for
  50%.
