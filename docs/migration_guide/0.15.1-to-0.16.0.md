# Upgrading from nlink 0.15.1 to 0.16.0

0.16.0 is a **substantial-but-mostly-additive** release.
Everything that compiled against 0.15.1 should compile against
0.16.0 with one cosmetic exception (the `Error` enum gained
`#[non_exhaustive]` markers — `match` arms need a wildcard if you
didn't already have one). The cycle's headline is a new
proc-macro crate (`nlink-macros`) that lets downstream code
declare custom Generic Netlink families in ~30 lines, plus two
in-tree dogfoods (DPLL, net_shaper) that validate the macros
end-to-end.

If you're using `#[deny(deprecated)]` or `#[deny(warnings)]`,
there are no new warnings to clean up — 0.16 doesn't add
deprecations.

## Breaking changes

### MSRV bumped from 1.85 to 1.95

`rust-version = "1.95"` in the workspace `Cargo.toml`. Required
because the nlink-macros crate uses `&&` short-circuit patterns
in `if let` (stabilized 1.95) for clearer macro-emit code, and
the new declarative-nftables diff path uses the same in scattered
places. Downstream code can target older rustc — they just can't
build nlink with it.

If you're on a CI matrix that pins rustc < 1.95, update the
toolchain spec to `stable` (or pin >= 1.95.0). All transitive
deps remain MSRV-compatible.

### Tightened generic bounds on `Connection::<P>::new*` and `namespace::connection_for*`

The sync constructor `Connection::<P>::new()` now requires
`P: SyncConstructible`, and `Connection::<P>::new_async()` requires
`P: AsyncConstructible` (both sealed traits introduced by Plan
148.5). This makes a long-standing footgun a compile error:

```rust
// 0.15.1 — compiled but produced a useless connection (the
// Wireguard GENL family ID was 0, never resolved).
let conn = Connection::<Wireguard>::new()?;

// 0.16.0 — compile error: Wireguard: !SyncConstructible.
// Use the async constructor instead, which resolves the family ID:
let conn = Connection::<Wireguard>::new_async().await?;
```

The same tighter bounds apply to `namespace::connection_for(name)`
(now requires `SyncConstructible`) and `namespace::connection_for_async(name)`
(requires `AsyncConstructible`). Use the `_async` variant for any
Generic Netlink family (`Wireguard`, `Macsec`, `Mptcp`, `Ethtool`,
`Nl80211`, `Devlink`, `Dpll`, `NetShaper`); the sync variant is
fine for `Route` / `Netfilter` / `Xfrm` / `Nftables` etc. that don't
need a family-ID round-trip.

If you have a generic function like
`fn connect<P: ProtocolState + Default>()`, you'll need to add
`+ SyncConstructible` (or `+ AsyncConstructible` for the async
form) to match. The traits are sealed — only nlink can `impl`
them for crate types — so generic helpers using them are
constrained to the in-tree protocol markers.

This is a deliberate type-safety improvement that mirrors the
0.15.0 typed-units rollout philosophy: prefer a compile error
to silently-wrong runtime behavior.

### `Error::Kernel` and `Error::KernelWithContext` are now `#[non_exhaustive]`

Plan 155 added an `ext_ack: Option<String>` field to both
variants for `NETLINK_EXT_ACK` TLV parsing (richer error
messages). The `#[non_exhaustive]` attribute is the semver-
correct way to add fields without breaking destructuring `match`
arms — but it does require an existing `match` on those variants
to include a wildcard:

```rust
// 0.15.1 — works
match err {
    Error::Kernel { errno } => { /* ... */ }
    _ => { /* ... */ }
}

// 0.16.0 — same code; the wildcard arm catches the new fields.
// If you destructured without a wildcard, add one now.
```

If you constructed `Error::Kernel { errno }` literally anywhere
in your code, that's a hard break — switch to
`Error::Kernel { errno, ext_ack: None }` or use the predicate
helpers (`err.is_busy()`, `err.is_not_found()`, …) which are
unchanged.

This single change is what bumped the cycle from `0.15.2` to
`0.16.0` per Cargo's 0.x.y semver rules.

### `RuleInfo` (nftables) gained three fields

Plan 157b v2 added per-rule reconciliation identity via
`NFTA_RULE_USERDATA`. `RuleInfo` now carries:

```rust
pub struct RuleInfo {
    // existing fields ...
    pub comment: Option<String>,        // parsed "nlink:<key>" tag
    pub userdata_raw: Option<Vec<u8>>,  // raw bytes (preserves foreign comments)
    pub expression_bytes: Vec<u8>,      // for diff body-equivalence check
}
```

Additive — the fields default to `None` / empty on rules that
don't carry a comment. Only affects code that constructs
`RuleInfo` literally (rare; it's a parser output, not a
constructor input) or matches it exhaustively.

### `#[non_exhaustive]` lockdown — 9 new-in-0.16 pub structs (Plan 163)

`RuleInfo`, `NftablesDiff`, `ReconcileOptions`, `ReconcileReport`,
`DpllDeviceReply`, `DpllPinReply`, `NetShaperReply`,
`NetShaperCapsReply`, `ConnectionPool`, `ConnectionPoolBuilder`,
`PooledConnection`, `DumpStream`, `ResyncStream`,
`GenlTypedDumpStream`, and `RouteInfo` (in
`nlink::netlink::diagnostics`) are now `#[non_exhaustive]`.
This was applied pre-cut to preserve forward-compatibility —
re-applying the attribute after publish would itself be a
breaking change.

**Only `ReconcileOptions` has a caller-visible impact** because
it's the only one users construct directly:

```rust
// Before — broke at 0.16.0
let opts = ReconcileOptions {
    max_retries: 5,
    backoff: Duration::from_millis(50),
};

// After — builder pattern
let opts = ReconcileOptions::default()
    .max_retries(5)
    .backoff(Duration::from_millis(50));
```

`Default::default()` and field access continue to work
unchanged. All other types in the list are constructed by
nlink internals, not user code; their match arms gain the
standard `_ => { ... }` requirement if you matched them
exhaustively.

### `PooledConnection::invalidate` now consumes `self` (Plan 162)

```rust
// Before — `&mut self`: subsequent `&*p` panicked at runtime.
let p = pool.acquire().await?;
p.invalidate();
let _ = &*p; // PANIC

// After — `self`: subsequent `&*p` is a compile error
// (E0382 — use of moved value).
let p = pool.acquire().await?;
p.invalidate();
let _ = &*p; // does not compile
```

**Source-compatible** for the typical "invalidate then drop"
use case (`p.invalidate();` still works; the guard is gone
after the call either way). Only the use-after-move bug-shape
is broken — that's the point: a footgun that used to panic at
runtime is now caught by the compiler.

## Behavior changes

### Namespace-restore failures now error instead of silently logging

`Connection::<P>::new_in_namespace(...)` used to log a stderr line
and return `Ok(socket)` when the post-construction `setns()`
back to the original namespace failed — a real bug, because the
caller would then operate in the wrong namespace without knowing.
Plan 147.1 promoted this to a hard error:

```rust
// 0.15.1
match Connection::<Route>::new_in_namespace("myns") {
    Ok(conn) => { /* might silently be in "myns" instead of the calling ns */ }
    Err(_) => { /* only fires for setns(myns) failures */ }
}

// 0.16.0
match Connection::<Route>::new_in_namespace("myns") {
    Ok(conn) => { /* guaranteed back in the original namespace */ }
    Err(e) if e.is_namespace_restore_failed() => {
        // NEW: the setns-back failed. The fd is closed; the
        // calling process may now be in `myns`. Recovery is
        // process-level (re-exec, or accept the new namespace
        // and proceed).
    }
    Err(e) => { /* original-namespace setns failure as before */ }
}
```

The new variant `Error::NamespaceRestoreFailed { source }` is the
sole break. If your error-handling code matches `Error` exhaustively
(without a wildcard), add either a wildcard arm or an explicit
`is_namespace_restore_failed()` predicate check.

### `set_ext_ack(true)` is now the default

Every `Connection<P>::new*` constructor enables
`NETLINK_EXT_ACK` on the socket automatically. When the kernel
attaches a `NLMSGERR_ATTR_MSG` TLV to an error response (most
common: `EINVAL` on a malformed attribute), nlink parses it and
folds the human-readable text into `Error::Kernel::ext_ack` —
so error display messages get richer for free:

```text
# 0.15.1
"add_link(veth0, kind=veth): Invalid argument (errno 22)"

# 0.16.0
"add_link(veth0, kind=veth): Invalid argument (errno 22): \
 attribute IFLA_MTU rejected: value 0 out of range (at request offset 24)"
```

To restore the 0.15.1 minimal-output behavior on a connection:

```rust
let mut conn = Connection::<Route>::new()?;
conn.set_ext_ack(false)?;
```

But really, don't — the richer messages are strictly better.

### Devlink / Nl80211 / Ethtool subscribe paths

These three GENL families were refactored onto the new shared
`__rt::resolve_genl_family_with_groups` + `GenlFamily::mcast_group`
infrastructure (Plan 156 Phase 5 follow-up). The bespoke
`Connection::<F>::subscribe()` methods still work and are
unchanged — they're now thin wrappers over the generic
`subscribe_group(name)`. Internal-only `pub(crate)` field renames
on the marker structs (`monitor_group_id: Option<u32>` → 
`mcast_groups: HashMap<String, u32>`); no public-API impact.

If you reach into `state()` via `Connection::<Devlink>::state()`,
that's a private path — opening an issue is more useful than
trying to keep it working.

## New surface — at a glance

These are all additive; no migration needed. Documented here so
you know what's new to adopt.

### `nlink-macros` proc-macro crate (Plan 154)

Declare a complete custom GENL family + typed request/response
structs in ~30 lines:

```rust
use nlink::macros::{
    genl_family, GenlAttribute, GenlCommand, GenlEnum,
    GenlMessage, NetlinkAttrs,
};

#[genl_family(name = "taskstats", version = 1)]
pub struct Taskstats;

#[derive(GenlCommand, Debug, Clone, Copy)]
#[genl_command(repr = "u8")]
pub enum TaskstatsCmd { Get = 1, /* ... */ }

#[derive(GenlAttribute, Debug, Clone, Copy)]
#[genl_attribute(repr = "u16")]
pub enum TaskstatsAttr { Pid = 1, Stats = 4 }

#[derive(GenlMessage, Debug, Default, Clone)]
#[genl_message(cmd = TaskstatsCmd::Get)]
pub struct TaskstatsGet {
    #[genl_attr(TaskstatsAttr::Pid)]
    pub pid: u32,
}

let conn = Connection::<Taskstats>::new_async().await?;
let reply: TaskstatsGet = conn.send_typed(TaskstatsGet { pid: 1 }).await?;
```

Full walkthrough:
[`docs/recipes/define-your-own-genl-family.md`](../recipes/define-your-own-genl-family.md).

### DPLL family (Plan 156)

Typed `Connection<Dpll>` API for kernel 6.7+ clock-synchronization
hardware (SyncE / PTP / GNSS). Telco-RAN / time-sync / SmartNIC
use case. Push-based multicast monitor:

```rust
use nlink::netlink::{Connection, genl::dpll::{Dpll, DpllEvent}};
use tokio_stream::StreamExt;

let mut conn = Connection::<Dpll>::new_async().await?;
conn.subscribe_monitor()?;
let mut events = conn.events();
while let Some(evt) = events.next().await {
    match evt? {
        DpllEvent::DeviceChanged(dev) => { /* lock-status change */ }
        DpllEvent::PinChanged(pin) => { /* pin-state change */ }
        _ => {}
    }
}
```

Recipe: [`docs/recipes/dpll-monitor.md`](../recipes/dpll-monitor.md).

### `net_shaper` family (Plan 153.3)

Kernel 6.13+ TX hardware shaping (per-NIC / per-queue /
intermediate-node bandwidth / burst / priority / weight) on
shaper-capable drivers (Intel `ice` E810/E830, Mellanox `mlx5`
ConnectX-7+, Broadcom `bnxt`). Second in-tree macro dogfood.

Recipe: [`docs/recipes/tx-hw-shaping.md`](../recipes/tx-hw-shaping.md).

### Streaming dump API (Plan 149)

O(1)-memory iteration over scale-out dump responses. Available
on every dumpable kernel object:

```rust
// rtnetlink
conn.stream_links().await?;
conn.stream_routes().await?;
conn.stream_neighbors().await?;
conn.stream_addresses().await?;

// TC
conn.stream_qdiscs().await?;
conn.stream_classes("eth0").await?;
conn.stream_filters("eth0", parent).await?;

// XFRM (Plan 149 follow-up)
xfrm_conn.stream_sas().await?;
xfrm_conn.stream_sps().await?;

// Netfilter / nftables (Plan 149 closeout)
nf_conn.stream_conntrack(libc::AF_INET as u8).await?;
nf_conn.stream_conntrack_v4().await?;
nf_conn.stream_conntrack_v6().await?;
nft_conn.stream_rules("filter", Family::Inet).await?;
```

All return `DumpStream<'_, P, T>` which implements
`tokio_stream::Stream<Item = Result<T>>`. Prefer over the eager
`get_*` / `list_*` methods on hosts with BGP / conntrack /
IPsec / nftables-scale state.

### Declarative `NftablesConfig` (Plan 157)

Mirror of `NetworkConfig` for nftables — diff + atomic apply +
reconcile. Per-rule identity via `handle_key` (encoded as
`NFTA_RULE_USERDATA` so it shows up as `comment "nlink:<key>"` in
`nft list ruleset` output):

```rust
let cfg = NftablesConfig::new().table("filter", Family::Inet, |t| {
    t.chain("input", |c| c.hook(Hook::Input).policy(Policy::Drop))
        .rule_keyed("input", "ssh-allow", |r| r.match_tcp_dport(22).accept())
});

cfg.diff(&conn).await?.apply(&conn).await?;     // first time: 2 adds
cfg.diff(&conn).await?.apply(&conn).await?;     // re-apply: 0 ops (idempotent)
```

Recipe:
[`docs/recipes/nftables-declarative-config.md`](../recipes/nftables-declarative-config.md).

### `ConnectionPool<P>` (Plan 159)

Bounded-mpsc-channel-backed pool for parallel netlink fanout.
Lets workers borrow a `PooledConnection<'_, P>` and return it
on drop:

```rust
let pool = ConnectionPool::<Route>::new(4)?;
let leases: Vec<_> = (0..32)
    .map(|i| {
        let p = pool.clone();
        tokio::spawn(async move {
            let conn = p.acquire().await?;
            conn.get_links().await
        })
    })
    .collect();
```

Recipe: [`docs/recipes/connection-pool.md`](../recipes/connection-pool.md).

### `ResyncedEvent<T>` / `ResyncMarker` (Plan 151)

Types for handling `ENOBUFS` overflow during multicast event
subscription. **The pre-baked `events_with_resync()` Stream
wrapper landed in the pre-cut audit window**, so 0.16 ships
both the typed sum (`ResyncedEvent<T>` / `ResyncMarker`) and
the wrapper:

```rust
use nlink::netlink::resync::events_with_resync;

let live = conn.subscribe_links().await?;          // Stream<Item = Result<LinkMessage>>
let stream = events_with_resync(live, move || {
    let conn = conn.clone();
    Box::pin(async move { conn.get_links().await })
});

while let Some(item) = stream.next().await {
    match item? {
        ResyncedEvent::Event(link) => /* live event */,
        ResyncedEvent::Resynced(link) => /* replayed from snapshot */,
        ResyncedEvent::Marker(ResyncMarker::ResyncStart) => /* sync began */,
        ResyncedEvent::Marker(ResyncMarker::ResyncEnd)   => /* sync done */,
    }
}
```

The hand-rolled loop pattern in the recipe still works (and
remains the right shape when you want full control over the
snapshot lifecycle). Use the wrapper for the common case.

Recipe: [`docs/recipes/events-with-resync.md`](../recipes/events-with-resync.md).

### `syscall_batch` opt-in feature (Plan 158)

Wraps `recvmmsg` / `sendmmsg` around the dump and streaming
paths for ≥1.5× speedup on dump-heavy workloads. Opt-in for one
release of soak; default-on planned for 0.17.

```toml
[dependencies]
nlink = { version = "0.16", features = ["syscall_batch"] }
```

No source changes — the feature transparently speeds up
`get_routes()` / `stream_links()` / `stream_conntrack()` / etc.

### Shared GENL multicast-group resolution

`#[genl_family]`-derived markers automatically populate a
`mcast_groups: HashMap<String, u32>` map at construction time;
the generic `Connection::<F: GenlFamily>::subscribe_group(name)`
helper looks up the kernel ID. This is the substrate Plan 156's
DPLL multicast monitor uses; downstream code defining custom
GENL families with `#[genl_family]` gets it for free.

### `enable_strict_checking(bool)` sockopt (Plan 155.2)

`Connection::enable_strict_checking(true)?` enables
`NETLINK_GET_STRICT_CHK` (kernel 5.0+). Surfaces
client/kernel-version mismatches as errors instead of silent
misbehavior. Off by default; opt in when developing against a
specific kernel.

### `wait_link_up(iface, timeout)` + `get_link_stats(iface)` (Plan 148)

Convenience helpers for two common patterns. `wait_link_up`
event-subscribes + polls until the interface flag flips up (or
`timeout` elapses); `get_link_stats` returns the typed
`LinkStats64` struct.

## What did NOT change

- Every typed config from 0.15.0 (`HtbQdiscConfig`, `NetemConfig`,
  the 18 qdisc kinds + 4 class kinds + 9 filter kinds + 14
  action kinds) — wire shape, parsing, and Connection methods
  are byte-identical.
- `Rate` / `Bytes` / `Percent` / `TcHandle` / `FilterPriority`
  newtypes — no API change.
- `NetworkConfig` (the existing declarative-config primitive
  for links/addresses/routes/qdiscs) — no API change.
- Error predicates (`is_busy()`, `is_not_found()`, …) — no API
  change.
- The 16 GENL families from 0.15.0 (WireGuard, MACsec, MPTCP,
  Ethtool, nl80211, Devlink) — bespoke `Connection<P>` APIs
  unchanged; the internal refactor onto shared infra is
  invisible.
