# Upgrading from 0.22.0 to 0.23.0

**A large, mostly additive release with a small breaking surface.**
0.23.0 closes the `#134`–`#137` follow-on epic — full nl80211 read
audit, XFRM monitor events, declarative nftables sets, the `Store`
reflector watch-cache, declarative-config purge, JSON Schema, the
opt-in dispatcher mode, and more. Almost all of it lands additively.

The breaking changes are three, all mechanical: `Chain::new` now
returns `Result`, the parsed `LinkStats` is read through accessors, and
five nl80211 result structs became `#[non_exhaustive]`. If you don't
use those APIs, **bump the version and nothing else changes.**

For the full list of what's new, see
[`CHANGELOG.md ## [0.23.0]`](../../CHANGELOG.md). If you're upgrading
from an older release, read the intervening guides too — start at
[`README.md`](README.md).

## Breaking changes

### 1. `Chain::new(table, name)` now returns `Result`

The nftables table-name boundary is now the validated `TableName`
newtype (mirroring `ChainName`), so `Chain::new` validates the names it
is given and returns a local `Err` for an empty / overlong /
interior-NUL name instead of letting a kernel `EINVAL` surface later.
The arguments also became `impl TryInto<TableName / ChainName>`, which
makes swapping the table and chain a compile error.

Plain `&str` / `String` still work (now with validation) — the only
change is the `Result`:

```rust
// Before (0.22):
let chain = Chain::new("filter", "input").hook(Hook::Input);

// After (0.23): add `?` (or handle the Err)
let chain = Chain::new("filter", "input")?.hook(Hook::Input);
```

The name-taking `Connection` methods (`add_table`,
`add_table_with_flags`, `del_table`, `del_table_if_exists`,
`flush_table`, `list_chains_in`, `del_chain`, `del_chain_if_exists`)
gained the same `impl TryInto<…>` bound but are **not** breaking for
`&str` callers — `conn.add_table("filter", Family::Inet)` compiles
unchanged (and now validates). Already-typed values pass through
infallibly. Both newtypes also gain `FromStr` (`"filter".parse()?`) and
`From<TableName>/From<ChainName> for String`.

> The transaction/batch builder name args remain `&str` (they return
> `Self`, so can't surface a validation error mid-chain) — unchanged.

### 2. `LinkStats` counters are read through accessors

The parsed `LinkStats` (the `rtnl_link_stats64` wrapper reachable via
`LinkMessage`, **not** the user-facing `stats::LinkStats` DTO) had bare
`pub` fields; they're now `pub(crate)` behind the existing accessor
methods and the struct is `#[non_exhaustive]`:

```rust
// Before (0.22):
let rx = link.stats().map(|s| s.rx_bytes);

// After (0.23): call the accessor
let rx = link.stats().map(|s| s.rx_bytes());
```

The separate `stats::LinkStats` aggregate (with `name` + `total_*` /
rate helpers) is a convenience DTO and is **unchanged**.

### 3. nl80211 result structs are now `#[non_exhaustive]`

`SurveyInfo`, `StationInfo`, `PhyInfo`, `Band`, and `Frequency` gained
fields this release (channel survey, richer station counters, PHY band
capabilities) and are now `#[non_exhaustive]` + `Default`. You can
still read every field; the change only affects code that **constructs
them with a struct literal** or **matches them exhaustively** — add
`..` :

```rust
// Exhaustive match now needs a rest pattern:
match survey {
    SurveyInfo { frequency, noise, .. } => { /* … */ }
}
```

These are dump-result types you normally read, not build, so most code
is unaffected.

## Behaviour changes (now-correct data)

These are wire-format / protocol fixes — same API, but data that was
previously wrong or missing is now correct. No code change required;
re-test if you depended on the old (buggy) behaviour.

- **nftables named sets now work.** `Connection::add_set` (and the new
  declarative `DeclaredSet`) failed with `ERANGE` on every create
  because `NFTA_SET_ID` / `NFTA_SET_HANDLE` were the wrong attribute
  ids. Fixed; `SetInfo::handle` now parses too.
- **nl80211 `BAND_ATTR_VHT_CAPA` and `STA_INFO_RX_BITRATE`** were
  parsed off the wrong attributes — VHT band capabilities and station
  RX bitrate are now populated correctly (were silently wrong / absent).
- **nl80211 `get_phys` no longer truncates rich PHYs.** It now requests
  `NL80211_ATTR_SPLIT_WIPHY_DUMP` and reassembles the multi-message
  reply per wiphy, so bands / channels / HE caps that didn't fit one
  message are no longer dropped (and no longer parsed as separate
  `PhyInfo`s).
- **GENL command paths no longer desync the socket.** The WireGuard /
  MACsec / MPTCP / ethtool `SET`/query helpers routed through bespoke
  single-`recv` blocks that could return `Ok` without consuming their
  own ACK on a stale/interleaved frame; they now use the canonical
  looped-seq-filter helpers (the "H9" class).

## Notable additive — new capabilities

None of these force a migration; they supersede or extend older
patterns. Highlights (see the CHANGELOG for the full set):

- **Reflector / watch-cache** — `Store<K, V>` + `ReflectExt::reflect`
  keep an in-memory cache up to date from a resync event stream
  (`netlink::reflector`). See `docs/library.md` §"Reflector /
  Watch-Cache".
- **`NetworkConfig` declarative purge** — opt-in full reconcile
  (`DiffOptions::purge` / `ApplyOptions::with_purge`), fenced to
  global-scope addresses on managed interfaces and `static`/`boot`
  main-table routes; links and qdiscs are never purged. (Re-instates,
  correctly, the knob removed in 0.19.)
- **JSON Schema** for `NetworkConfig` via the new `schemars` feature
  (`NetworkConfig::json_schema()`).
- **Declarative nftables sets** — `DeclaredSet` + element-level diff
  inside the atomic batch.
- **Opt-in dispatcher mode** — `Connection::with_dispatcher()` lets a
  long-lived event stream and concurrent requests coexist on one
  connection (the default lean mutex is unchanged).
- **XFRM monitor** — `Connection<Xfrm>: EventSource` (`ip xfrm monitor`
  equivalent).
- **Ergonomic newtype conversions** — `Bytes ⇄ u64`, `TcHandle ⇄ u32`,
  `FilterPriority → u16`, `Percent → f64`, `TableName`/`ChainName →
  String` (lossless `From`/`Into`; `Rate` deliberately excluded to keep
  its unit-safety guarantee).
- Bridge `BRIDGE_VLANDB_ENTRY` / `GOPTS` per-VLAN options, net_shaper
  group shaping, `WireguardConfig::{from_wg_quick, client}`, TC pedit
  `munge` DSL, `Connection<Ovpn>::attach_socket`.

## Deferred

Two Plan 195/#137 items are intentionally not in 0.23.0:

- `backon`-based `StreamBackoff` — the resync wrapper already handles
  `ENOBUFS`; reconnect backoff composes at the spawn-loop level.
- `ip vrf exec` — needs cgroup2 + eBPF `sock_bind`, untestable under
  the non-root CI gate.
