# Upgrading from nlink 0.18.0 to 0.19.0

**Status: STUB — populated by each 0.19 plan as it lands.**

This file is created by Plan 193 (the first 0.19 plan in the
recommended landing order). Each subsequent plan appends its
own section below; the cycle cut polishes the headline + the
intro paragraph.

The 0.19 cycle theme is **closing the loop with nlink-lab's
158-arc feedback** (`nlink-feedback.md`) — six feedback-driven
plans plus three defensive plans surfaced during the 0.19
consolidation-pass ecosystem audit, plus a kube-rs-shaped
stream-combinator plan.

The cycle ships **multiple breaking changes**, all small and
mechanically migrable. See per-plan sections below.

## Plan 187 — `Error` API sign normalization + Io-shape predicate fix

### What changed

- **Factory sign normalization**: `Error::from_errno`,
  `from_errno_with_context`, `from_errno_ext_ack`,
  `from_errno_with_context_ext_ack` now call `.abs()` on the
  input. Both `from_errno(1)` and `from_errno(-1)` produce
  stored `errno = 1`.

- **Predicate Io-shape fix**: `Error::errno()` unwraps
  `Error::Io(io_err)` via `raw_os_error()` so every predicate
  goes through one code path. Three predicates that previously
  matched on the variant directly (`is_busy`,
  `is_already_exists`, `is_permission_denied`) now go through
  `errno()` and catch the Io shape too. Critical for
  `apply_reconcile` retry classification.

### Migration steps

**For most consumers:** no action required. Production code
using `?`-bubbled kernel errors keeps working.

**For direct tests / mocks** that asserted `Some(-1)` after
calling `from_errno_ext_ack(1, ..)`:

```rust
// Before 0.19
assert_eq!(err.errno(), Some(-1));

// 0.19+
assert_eq!(err.errno(), Some(1));
```

**For boxed-source wrappers**: still a footgun until Plan 187
phase 2 (chain_walk helper) lands. Until then, work around by
unwrapping the box manually:

```rust
let inner = source.downcast_ref::<Box<nlink::Error>>()
    .map(|b| b.as_ref())
    .or_else(|| source.downcast_ref::<nlink::Error>());
```

**Phase 2 shipped**: `Error::chain_walk` + `root_cause` +
`contexts` now handle both shapes:

```rust
use nlink::Error;
// Find the first ENOBUFS anywhere in the chain.
let enobufs = Error::chain_walk(&outer_err).find(|e| e.is_no_buffer_space());
// Get the deepest nlink error.
let bottom = outer_err.root_cause();    // when outer_err is &nlink::Error
// All nlink layers as a Vec.
let layers: Vec<_> = err.contexts();
```

If you've been writing the manual ladder, switch to `chain_walk`.

## Plan 190 — `LinkBuilder` gaps (VRF first; more in follow-ups)

### What changed (this commit)

`LinkBuilder::vrf(table)` ships. The `DeclaredLinkType` enum
grew a `Vrf { table }` variant. Members enslave via the
existing `LinkBuilder::master` chain.

```rust
let cfg = NetworkConfig::new()
    .link("vrf-red", |b| b.vrf(100))
    .link("eth0",   |b| b.dummy().master("vrf-red"));
cfg.apply(&conn).await?;
```

### Migration steps

The `DeclaredLinkType` enum is `#[non_exhaustive]`, so this
addition is **strictly additive** for callers using `..`
rest-patterns or `_` catch-alls. If you wrote an exhaustive
match WITHOUT `..`, add `DeclaredLinkType::Vrf { .. } => ...`
to keep compiling.

```rust
// Add this arm (or use `_ => ...`) when widening to 0.19.0
DeclaredLinkType::Vrf { .. } => { /* your handler */ }
```

### VLAN protocol — Plan 190 §2.2 (also shipped)

`LinkBuilder::vlan_protocol(VlanProtocol)` selects the VLAN
tagging protocol. `DeclaredLinkType::Vlan` widened with
`protocol: Option<VlanProtocol>`.

```rust
let cfg = NetworkConfig::new()
    .link("eth0.100", |b| {
        b.vlan("eth0", 100).vlan_protocol(VlanProtocol::Dot1ad)
    });
```

**Pattern-match update required** — direct matches on the
Vlan variant now need a `..` rest-pattern OR an explicit
`protocol` binding:

```rust
// Before 0.19
DeclaredLinkType::Vlan { parent, vlan_id } => ...
// 0.19+
DeclaredLinkType::Vlan { parent, vlan_id, .. } => ...
// OR
DeclaredLinkType::Vlan { parent, vlan_id, protocol } => ...
```

`VlanProtocol` is `#[non_exhaustive]` — future kernel
additions will widen the enum.

### VXLAN extras — Plan 190 §2.1 (also shipped)

Three new VXLAN knobs on `LinkBuilder`:

```rust
let cfg = NetworkConfig::new().link("vx0", |b| {
    b.vxlan(42)
        .vxlan_remote(IpAddr::V4(Ipv4Addr::new(10, 0, 0, 99)))
        .vxlan_local(IpAddr::V4(Ipv4Addr::new(10, 0, 0, 2)))
        .vxlan_port(4790)
        .vxlan_underlay_dev("eth0")
});
```

`DeclaredLinkType::Vxlan` widened with `local` / `port` /
`underlay_dev` fields; update direct pattern matches to use
`..` rest-pattern. IPv6 local source addresses are silently
ignored today — the imperative VxlanLink is IPv4-only for
tunnel-source IPs, matching the existing `remote` shape.

Idempotent re-apply for these knobs is deferred — VXLAN
`compute_diff` parity against the kernel's `IFLA_VXLAN_*`
dump requires an `IFLA_INFO_DATA` parser that isn't in
nlink yet. Re-applies replay the create (the kernel's
"already exists" surfaces as an error).

### Bond options gap-fill — Plan 190 §8 (also shipped)

5 new declarative bond setters:

```rust
let cfg = NetworkConfig::new().link("bond0", |b| {
    b.bond()
        .bond_mode(BondMode::Ieee8023ad)
        .bond_ad_select(BondAdSelect::Bandwidth)
        .bond_lacp_rate(BondLacpRate::Fast)
        .bond_downdelay(200)
        .bond_updelay(500)
        .bond_resend_igmp(3)

```

`DeclaredLinkType::Bond` widened with 5 matching

from `nlink::netlink::config::types` as `BondAdSelect` +
`BondLacpRate` for ergonomic import — they're aliases for
the imperative enums, not separate types. Pattern matches

## Plan 197 — ovpn GENL declarative (deferred to 0.20)

### Status: deferred

The ovpn link half shipped in Plan 190 §2.3b
(`LinkBuilder::ovpn()` + `OvpnLink`). The declarative-GENL
path the plan calls for cannot ship in 0.19 because the
**imperative ovpn GENL family doesn't exist in nlink yet**.

### What's missing

Plan 197 assumes there's a `Connection<Ovpn>` with
`get_peer` / `set_peer` / `del_peer` methods to build the
declarative path on top of. nlink has zero ovpn GENL
infrastructure — no protocol marker, no family-ID
resolution, no command/attribute enums, no per-method
implementations.

### Why this isn't "just" a coverage gap

The kernel `ovpn` GENL family is fresh in 6.16 (released
spring 2026); the UAPI (`include/uapi/linux/ovpn.h`) is
still being refined. nlink's other GENL families
(WireGuard, Devlink, DPLL, ethtool, MACsec, MPTCP, nl80211)
were implemented against APIs with at least 2 years of
stability. Shipping ovpn GENL before the ABI settles risks
locking nlink into a shape that needs breaking changes
once the kernel iterates.

### Path forward (0.20+)

1. Implement the imperative `Connection<Ovpn>` family —
   protocol marker, async family-resolution path,
   command + attribute enums, peer-config methods.
   Estimated 400–600 LOC, plus integration tests behind
   `require_module!("ovpn")` (kernel 6.16+ gate).
2. Once shipped + a soak release, layer the declarative
   `OvpnConfig` + `apply` / `diff` on top (mirrors Plan
   196 — declarative WG).
3. Wire ovpn into the `nlink::facade::Stack` bundle
   (currently absent — see Plan 200's stack.rs doc).

This sequencing matches how every other GENL family in
nlink shipped (imperative first, declarative after the
imperative API has had time to bake).

### See also

- `crates/nlink/src/netlink/genl/wireguard/config.rs` —
  Plan 196's declarative WG is the template the ovpn
  declarative path will follow.
- `include/uapi/linux/ovpn.h` (kernel) — UAPI as of 6.16.
- OpenVPN ovpn-dco userspace docs.

## Plan 194 — Concurrent stress regression tests + F1 fix

### What changed

Two regression tests landed under
`crates/nlink/tests/integration/concurrent_stress.rs`,
preempting two bug-shapes that hit the `rtnetlink` Rust crate
recently (`rtnetlink` issues #131 + #132):

1. `concurrent_dumps_on_shared_connection_route_correctly` —
   16 concurrent `get_links()` calls on a shared
   `Arc<Connection>`. Initially failed in CI; **fixed in 0.19**
   by closing F1 (see below).
2. `concurrent_namespaces_dont_corrupt_each_other` —
   16 concurrent namespace creations, each with its own
   `Connection`. Always passed; nlink's `LabNamespace::new`
   is hermetic.

### F1 — shared `Arc<Connection>` concurrent ops (CLOSED in 0.19)

**The bug.** nlink's `send_*_inner` recv loops in
`connection.rs` filter by `nlmsg_seq` and `continue` past
non-matching frames. The filter is *destructive*: when two
concurrent tasks share an `Arc<Connection>`, task A's
`recv_msg().await` reads task B's response, sees
`seq != A`, and discards it. Task B is now waiting on a
response that's already been consumed; it blocks until the
30s operation timeout fires.

This is the rtnetlink #131 shape — every Rust netlink crate
has hit it. `netlink-proto` (used by `rtnetlink`) and `neli`
(`NlRouter`) both solve it by spawning a dedicated background
task that owns the fd and dispatches incoming frames to
per-request `mpsc::Sender`s keyed by seq in a shared `HashMap`.

**The fix in 0.19.** nlink takes a smaller, equally-correct
fix: add a `tokio::sync::Mutex` to `Connection<P>` and acquire
it at the top of every method that does
`socket.send(...) + recv-loop-until-DONE`. Concurrent callers
on a shared `Arc<Connection>` now serialize cleanly instead of
racing.

```rust
// New in 0.19 — connection.rs send_dump_inner
async fn send_dump_inner(&self, builder: MessageBuilder) -> Result<...> {
    // F1 fix — serialize the send + recv-loop pair so concurrent
    // tasks on a shared `Arc<Connection>` don't race on the
    // recv side.
    let _guard = self.request_lock.lock().await;
    // ... existing logic unchanged ...
}
```

The seq filter remains as a defensive backstop against stale
multicast / cross-process frames, but no longer load-bears
against multi-task races.

**Trade-off vs. dispatcher-task design.** nlink keeps a
simpler single-socket-single-flight model. Concurrent dumps
on a shared `Arc<Connection>` complete in sequence (not in
parallel) — the kernel processes a single netlink socket
FIFO anyway, so the wall-clock cost is only the
RTT-pipelining nlink doesn't do. **For true parallel
throughput, use
[`ConnectionPool<P>`](https://docs.rs/nlink/latest/nlink/netlink/pool/struct.ConnectionPool.html)**
(Plan 159) — each task gets its own connection (and
therefore its own kernel-side socket queue, which the kernel
processes in parallel).

```rust
use std::sync::Arc;
use nlink::{ConnectionPool, Route};

let pool = Arc::new(ConnectionPool::<Route>::new(16)?);
let mut handles = Vec::new();
for _ in 0..16 {
    let pool = pool.clone();
    handles.push(tokio::spawn(async move {
        let conn = pool.acquire().await?;
        conn.get_links().await
    }));
}
```

**What this fix does NOT cover.** Multicast `events()` streams
running concurrently with requests still race the recv side —
event streams tap the socket directly via `poll_recv` and
don't acquire the request lock. The full per-seq dispatcher
design (NlRouter-style) is still queued as the 0.20 follow-on
but at a much lower priority now that the common case works.

## Plan 199 — WireGuard polling watcher

### What changed

`nlink::netlink::genl::wireguard::watch` ships a typed
poll-and-diff primitive for WireGuard peer-state changes.

### Why polling, not multicast

The kernel `wireguard` GENL family declares zero multicast
groups (`n_mcgrps = 0` in
`drivers/net/wireguard/netlink.c`, verified 2026-05-31).
The UAPI exposes only `WG_CMD_GET_DEVICE` +
`WG_CMD_SET_DEVICE` — request/response, no notifications.

Every WG monitoring tool (wg-tools, systemd-networkd,
cunicu) polls `GET_DEVICE` and diffs client-side. This
module shipping that primitive saves consumers from
reimplementing it.

Linus Lotz submitted [a kernel patch adding multicast
events to the WG family][patch] in January 2021. Status as
of mainline today: **"Awaiting Upstream"** — never merged.
If the kernel grows multicast support, this watcher will
become a compatibility shim; the [`WireguardEvent`] enum
shape stays the same.

[patch]: https://lkml.kernel.org/netdev/20210115195353.11483-1-linus@lotz.li/T/

### Usage

```rust
use std::time::Duration;
use nlink::netlink::{Connection, Wireguard};
use nlink::netlink::genl::wireguard::{
    WireguardWatcher, WireguardWatchOptions,
};

let conn = Connection::<Wireguard>::new_async().await?;
let mut watcher = WireguardWatcher::new(
    conn,
    WireguardWatchOptions::default()
        .interval(Duration::from_secs(5))
        .interface("wg0"),
)?;

loop {
    let events = watcher.next_events().await?;
    for ev in events {
        println!("{ev:?}");
    }
}
```

## Plan 189 — `serde` feature flag (opt-in JSON serialization)

### What changed

The new top-level `serde` feature derives `Serialize` on every
public diff, result, and report type. JSON conventions:

- Structs are `rename_all = "kebab-case"`: a `ConfigDiff`'s
  `links_to_add` field serializes as `"links-to-add"`.
- Enums are `rename_all = "snake_case"` so unit variants emit
  bare strings: `Family::Inet` → `"inet"` (not `{"Inet": null}`).

```toml
nlink = { version = "0.19", features = ["serde"] }
```

```rust
use nlink::netlink::config::ConfigDiff;
let diff: ConfigDiff = cfg.diff(&conn).await?;
let json = serde_json::to_string_pretty(&diff)?;
```

### Migration steps

**Strictly additive.** Existing call sites work unchanged.

The `serde` feature is **opt-in** and not in `default`; it's
included in `full` for convenience. `Deserialize` is **not**
derived — the diff types are products of `compute_diff()`,
not user-constructible from JSON. Round-trip would add no
direct value today.

### Two intentional shape choices to note

- `DeclaredRule`'s `body: Rule` field is `#[serde(skip)]`'d —
  the wire-format expression tree is too varied for stable
  JSON. Consumers wanting the expression details should call
  `body()` programmatically.
- `ApplyError`'s `error: Error` field serializes as its
  `Display` string, not the structural variant — the `Error`
  type isn't shape-stable.

### Future expansions

- `Deserialize` derives (round-trip support) — deferred until
  a consumer asks.
- `schemars` JSON Schema export (Plan 189 §8 expansion) —
  deferred for the same reason; the kebab-case shape pins the

## Plan 186 — VLAN parent ordering in `NetworkConfig::apply`

### What changed

`NetworkConfig::apply` now topologically sorts the new-links
list so a parent declared LATER than its child (or any
ordering produced by a `HashMap`-shaped source) still works.
The sort is stable — independent links keep their declared
order.

```rust
// Either ordering now produces a successful apply.
let cfg = NetworkConfig::new()
    .link("eth0.42", |l| l.vlan("eth0", 42))   // child first
    .link("eth0",    |l| l.dummy());           // parent second
cfg.apply(&conn).await?;                       // both created
```

The `NetworkConfig::link` docstring carries the
order-independence guarantee. Recognized child→parent
dependencies today are `Vlan { parent }` and `Macvlan
{ parent }`; other link types are order-independent at the
type level and remain in declared position.

The fix complements the root-gated integration repro
(`tests/integration/network_config_apply.rs`) shipped earlier
in the cycle — that repro is now permanent regression coverage
for the `vlan_parent_dummy_declared_in_either_order` case.

### Migration steps

**No consumer action required.** Existing configs that
already declared parents first keep their ordering
deterministically.

# Second-batch migration sections (Plans 203-215)

After the original 0.19 plan suite landed, a deep audit
(`docs/AUDIT_REPORT_2026_05_31.md`) surfaced ~96 distinct bugs
across angles the cycle's first pass didn't reach. Plan 203 (the
second-batch master) orchestrated 13 sub-plans covering wire-
format CRITICAL fixes, declarative-config correctness,

shipped pre-cut; Plan 210 (examples cleanup) is in flight at

## Plan 204 — wire-format CRITICAL fixes

Four byte-level corrections against kernel UAPI.

### C1 — nftables verdict constants (BREAKING)

`NFT_JUMP` and `NFT_GOTO` constant values change to match kernel
UAPI:

```rust
// Pre-0.19:
pub const NFT_JUMP: i32 = -2;
pub const NFT_GOTO: i32 = -3;

// 0.19:
pub const NFT_BREAK: i32 = -2;  // NEW
pub const NFT_JUMP: i32 = -3;
pub const NFT_GOTO: i32 = -4;
```

`Verdict::Jump(chain)` previously wrote `-2` (= NFT_BREAK in the
kernel), causing rule evaluation to terminate instead of jumping.
Every subroutine rule was silently broken. Source-level no-op for
`Verdict::Jump` / `Verdict::Goto` users — only callers reading
the raw `NFT_*` constants need updates.

### C2/C3/C4 — XFRM struct sizes + devlink mcast name

Wire-format-only fixes. `XfrmUserpolicyInfo` body 164→168 bytes
(kernel-required pad added), `XfrmUserpolicyId` 68→64 (over-pad
trimmed), `XfrmUserTmpl` 62→64 (Plan 213 sizeof test surfaced
this sibling bug). Pre-0.19 `add_sp` was rejected with EINVAL on
every kernel; `del_sp`/`get_sp` brittle on strict-checking
kernels.

`DEVLINK_MCGRP_NAME` changed from `"devlink"` (which the kernel
doesn't register) to `"config"`. Every
`Connection::<Devlink>::subscribe()` call previously returned
`Error::FamilyNotFound`; now resolves correctly.

No source-level change for users.

## Plan 205 — `ApplyOptions::with_purge` removed (BREAKING)

The purge knob was silently non-functional in 0.18 — the diff
phase never populated the `*_to_remove` collections, so the
apply-side branches were dead code. Pre-0.19 `.with_purge(true)`
was a silent lie.

```rust
// Pre-0.19 (silently no-op):
ApplyOptions::default().with_purge(true)

// 0.19 (does not compile):
ApplyOptions::default().with_purge(true)
//                      ^^^^^^^^^^ method does not exist
```

Also removed: `ConfigDiff::{links,addresses,routes,qdiscs}_to_remove`
fields.

**Migration**: for the "remove undeclared resources" use case,
use the imperative API:

```rust
conn.del_link("eth0").await?;
conn.del_address("eth0", "10.0.0.1".parse()?, 24).await?;
conn.del_route(/* ... */).await?;
```

A correctly-wired purge with a kernel-managed-resource exclusion
list is queued for 0.20.

## Plan 206 — DPLL `phase_offset` widened to i64 (BREAKING)

`DpllPin::phase_offset` field type changes from `Option<i32>` to
`Option<i64>`.

```rust
// Pre-0.19:
let p: Option<i32> = pin.phase_offset;

// 0.19:
let p: Option<i64> = pin.phase_offset;
```

Kernel UAPI declares the attribute as `s64` (attoseconds × 1000);
pre-0.19 silently truncated the high 4 bytes. Telco/PTP/SyncE
values routinely exceed `i32::MAX` (1 ns offset = 1e9 in those
units), producing nonsense readings.

If you used `.phase_offset_ns()`: no change. Same return type
(`Option<i64>`), just no longer truncating internally.

The new `nlink::macros::__rt::parse_i64_attr` /
`emit_i64_attr` helpers and the `GenlMessage` derive's `i64`
recognition are available for downstream macro consumers
defining their own `s64` GENL attributes.

## Plan 207 — NetworkConfig correctness pass

Behavior changes (no source migration needed):

- Link `master` change between two existing bridges now produces
  a `set_master` diff (was silently no-op).
- Route gateway/dev/metric change on the same `(dst, prefix,
  table)` now produces a `routes_to_add` entry that the apply
  path commits via `NLM_F_REPLACE` (was silently no-op).
- `apply_reconcile` recomputes the diff per retry attempt, so
  partial-progress + EBUSY no longer masks the original error
  with EEXIST.
- `LinkState::Down` declared on no-carrier admin-up interfaces
  now correctly fires `set_down` (was silently no-op because the
  comparison read `OperState` instead of `IFF_UP`).
- `replace_qdisc` uses atomic `NLM_F_REPLACE` instead of del+add
  (eliminates the transient `pfifo_fast` window).
- Topo-sort now handles VXLAN `underlay_dev` and `master` refs
  (Plan 186 §3c gap closed for the remaining link types).
- `remove_route` forwards the table identity so non-default-
  table routes can actually be deleted.

## Plan 208 — recv-loop completion

14 of 15 protocol recv-loops now wrap in `with_timeout` + seq
filter + `NLM_F_DUMP_INTR` detection. Pre-0.19 each could hang
indefinitely on a dropped response and silently use an
interrupted-dump snapshot. Now surface as `Error::Timeout` /
`Error::DumpInterrupted`. No source-level change for callers.

## Plan 211 — `Hook::Ingress` split (BREAKING)

Pre-0.19 the single `Hook::Ingress` variant always encoded `0`,
correct only for `Family::Netdev`/`Bridge`. On `Family::Inet`,
ingress is `NF_INET_INGRESS = 5`, so the old encoding silently
installed the chain on `Prerouting`.

```rust
// Pre-0.19:
Chain::new("t", "c").family(Family::Netdev).hook(Hook::Ingress)
Chain::new("t", "c").family(Family::Inet).hook(Hook::Ingress)
// (second one was silently broken — installed on Prerouting)

// 0.19:
Chain::new("t", "c").family(Family::Netdev).hook(Hook::NetdevIngress)
Chain::new("t", "c").family(Family::Inet).hook(Hook::InetIngress)
```

`Hook::NetdevEgress` is also new (`NF_NETDEV_EGRESS = 1`).
`Hook::is_valid_for_family(Family)` validates combinations at
build time.

`bins/nft` accepts both `netdev:ingress` and `inet:ingress`
prefixes; bare `ingress` defaults to the netdev variant for CLI
back-compat.
