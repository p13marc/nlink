# Upgrading from 0.23.0 to 0.24.0

**A depth release around sockdiag, events, and nftables — almost all
additive, with a small mechanical breaking surface and two
behaviour-visible bug fixes you should read about.**

0.24.0 works the 2026 roadmap pass (#170): socket→process/cgroup
attribution, per-socket TCP goodput tracking, a full
`INET_DIAG_BC` bytecode compiler (ss-style expressions filter
kernel-side), typed congestion-control internals, typed nftables
rule-expression decoding, Rule/Nexthop/NsId/MDB events, a WireGuard
device-bootstrap path, and an ergonomics batch (`del_*_if_exists`,
`RateLimiter::reconcile`, `NamespaceSpec` facade variants).

The compile-breaking changes are all mechanical: three diff types and
`NetworkEvent`-adjacent structs became `#[non_exhaustive]`, and two
sockdiag structs gained fields. If you construct those with
`..Default::default()` and match non-exhaustively already, **bump the
version and nothing else changes.**

For the full list of what's new, see
[`CHANGELOG.md ## [0.24.0]`](../../CHANGELOG.md). Upgrading from an
older release? Read the intervening guides — start at
[`README.md`](README.md).

## Breaking changes

### 1. `ConfigDiff`, `StackDiff`, `WireguardConfigDiff` are `#[non_exhaustive]`

Literal construction and functional-update syntax outside nlink no
longer compile (#165). Construct them the only way that ever made
sense — via the diff computations:

```rust
// Before (0.23): possible, never useful
let diff = ConfigDiff { links_to_add: vec![], /* … */ };

// After (0.24): obtain from the API
let diff = cfg.diff(&conn).await?;          // NetworkConfig → ConfigDiff
let diff = stack.diff().await?;             // Stack → StackDiff
let diff = wg_cfg.diff(&wg_conn).await?;    // WireguardConfig → WireguardConfigDiff
```

Reading fields is unchanged. This is what lets 0.24's own
`WireguardConfigDiff::devices_to_add` (and future categories) land
additively.

### 2. Sockdiag struct literals: `InetFilter.expr`, `InetSocket.cc_info`

Two sockdiag structs gained fields (neither is `#[non_exhaustive]`):

- `InetFilter` gained `expr: Option<FilterExpr>` (#163) — the
  kernel-side expression filter.
- `InetSocket` gained `cc_info: Option<CcInfo>` (#163) — typed
  BBR/DCTCP/vegas congestion-control state.

Exhaustive struct literals need the new field or a
`..Default::default()` tail:

```rust
// Before (0.23):
let filter = InetFilter { protocol: Protocol::Tcp, states, /* every field */ };

// After (0.24) — either:
let filter = InetFilter { protocol: Protocol::Tcp, states, ..Default::default() };
// or use the builder:
let filter = SocketFilter::tcp().states(&[TcpState::Listen]).build();
```

### 3. `subscribe_all()` joins more groups; new `NetworkEvent` variants arrive

`Connection::<Route>::subscribe_all()` now also joins `NsId`,
`Ipv4Rule`, `Ipv6Rule`, and the new `Nexthop` and `Mdb` groups
(#165). `NetworkEvent` gained `NewRule`/`DelRule`,
`NewNexthop`/`DelNexthop`, `NewNsId`/`DelNsId`, `NewMdb`/`DelMdb`.

`NetworkEvent` was already `#[non_exhaustive]`, so matches with a
`_ => {}` arm compile unchanged — but under `subscribe_all()` the new
event kinds now **arrive at runtime**. If your consumer counts or
logs every event, expect policy-rule / nexthop / namespace-ID /
bridge-MDB traffic it didn't see before. Subscribe narrowly with
`subscribe(&[RtnetlinkGroup::…])` if you want the old set.

## Behaviour changes (bug fixes you might have worked around)

### 4. `InetExtension::mask()` was off by one — now fixed

The kernel checks `ext & (1 << (attr − 1))`; `mask()` returned
`1 << attr`. In practice every extension request was shifted one
over: `with_mem_info()` actually requested `INET_DIAG_INFO`,
`with_tcp_info()` requested `VEGASINFO`, and so on (#163).

After upgrading, the `with_*` builders request what they say. If you
compensated by picking the "wrong" builder to get the right data, or
passed raw bits into `InetFilter.extensions` based on the old
`mask()` output, re-check those call sites — raw literal bits
(`1 << 0` for meminfo etc.) were always correct and are unaffected.

### 5. `TcpInfo` tail fields are now populated

`parse_tcp_info` stopped at byte 168, so `bytes_sent`,
`bytes_retrans`, `busy_time`, `rwnd_limited`, `sndbuf_limited`,
`delivered`, `delivered_ce`, `dsack_dups`, `reord_seen`,
`rcv_ooopack` and `snd_wnd` always read **0** (#171). They now carry
real kernel values (on kernels new enough to emit them — 4.10+/4.19+
blocks). Dashboards or assertions built on the always-zero behaviour
will see real numbers.

### 6. `WireguardConfig::diff` no longer errors on an absent device

A declared-but-missing WireGuard link used to fail `diff()` with the
kernel's not-found error. It now lands in the new
`WireguardConfigDiff::devices_to_add` and `diff()` succeeds (#169).
`apply()` still refuses (with an error naming the devices) until the
links exist — create them with the new
`WireguardConfig::ensure_devices(&route_conn)`, or use
`facade::apply::wireguard*`, which bootstraps automatically. If you
relied on the `diff()` error to detect absent devices, check
`diff.devices_to_add` instead.

### 7. XFRM dumps stopped spamming the kernel log

`get_security_associations()` / `get_security_policies()` (and the
streaming variants) sent a zeroed header struct as the dump body,
producing the ratelimited kernel warning `netlink: 224 bytes leftover
after parsing attributes` on every poll (#160). Dumps are now
header-only. Results were always correct; only the log noise is gone.

## Headline additions (no action needed)

- **sockdiag** (#162/#163/#171): `SocketOwnerMap` / `CgroupPathMap`
  attribution, `SocketRateTracker` TCP goodput, kernel-side
  `FilterExpr` compilation (`InetFilter::expr` /
  `compile_filter`), `CcInfo` structs, `with_cc_info()`.
  Recipe: [`per-process-bandwidth`](../recipes/per-process-bandwidth.md).
- **nftables** (#164): `RuleInfo::expressions()` → typed `RuleExpr`,
  `RuleInfo::counter()` for per-rule hit/byte counters.
- **Ergonomics** (#169): `del_link_if_exists` +
  route/address/qdisc/filter siblings; `RateLimiter::reconcile{,_dry_run,_with_options}`;
  `Stack::{apply,diff}_in(NamespaceSpec)` + facade `_in` variants;
  `NamespaceSpec::connection_async`; `WireguardConfigDiff` serde
  (secrets redacted); `change_count()` on `StackDiff`/`StackApplyReport`;
  `NETNS_RUN_DIR` at the crate root.
