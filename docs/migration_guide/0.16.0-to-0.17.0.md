# Upgrading from nlink 0.16.0 to 0.17.0

0.17.0 is a **small, mostly internal-quality** release. The cycle
theme was closing the "hidden hang" class of bugs that the 0.16
cut's CI evidence surfaced, plus sweeping up the ergonomic +
tooling papercuts the cycle revealed. Most code that compiled
against 0.16.0 will compile against 0.17.0 unchanged.

There are two breaking changes, both in the `nftables` subsystem
and both stemming from the same root cause (`NftablesConfig::diff`
flagged unchanged rules as needing replacement on every
idempotent reapply — Plan 178). One behavior change applies
globally (default operation timeout on `Connection<P>`).

If you're using `#[deny(warnings)]`, there are no new warnings to
clean up — 0.17 doesn't add deprecations.

## Breaking changes

### `Register` discriminant values changed (Plan 178)

The `nlink::netlink::nftables::types::Register` enum's
discriminants switched from `NFT_REG32_xx` (`8..=11`, the 4-byte
register aliases) to the canonical `NFT_REG_x` form (`1..=4`,
the 16-byte registers):

```rust
// 0.16.0
pub enum Register {
    Verdict = 0,
    R0 = 8,   // NFT_REG32_00
    R1 = 9,
    R2 = 10,
    R3 = 11,
}

// 0.17.0
#[repr(u32)]
pub enum Register {
    Verdict = 0,
    R0 = 1,   // NFT_REG_1
    R1 = 2,
    R2 = 3,
    R3 = 4,
}
```

The enum also gained `#[repr(u32)]`, locking the layout so the
wire-format `as u32` cast is well-defined.

**What's wire-different**: nothing the kernel can observe. The
kernel canonicalizes any 4-byte transfer through `NFT_REG32_00`
to `NFT_REG_1` internally and dumps the canonical form, so a
rule submitted with `R0 = 8` and a rule submitted with `R0 = 1`
are byte-identical to other readers — but the *dumped form*
always uses `NFT_REG_1`. 0.16's submission form differed from
what the kernel echoed back, which broke `NftablesConfig::diff`'s
body-bytes comparison (Plan 178 — see below).

**Who breaks**: code that did `Register::R0 as u32` and stored
or compared the literal `8` (now `1`). If you're matching on
the enum variant directly (`match reg { Register::R0 => ... }`),
nothing changes — the variant name is the same.

```rust
// 0.16.0 — compiled and stored 8
let raw: u32 = Register::R0 as u32;
assert_eq!(raw, 8);  // FAILS in 0.17.0 (would be 1)

// 0.17.0 — explicit match, no breakage
let raw: u32 = match reg {
    Register::Verdict => 0,
    Register::R0 => 1,
    Register::R1 => 2,
    Register::R2 => 3,
    Register::R3 => 4,
    _ => 0,
};
```

If you have downstream tests that pinned the numeric value (e.g.
asserting against captured wire bytes), update the fixtures.

### `NftablesDiff::rules_to_delete` tuple shape changed (Plan 178)

```rust
// 0.16.0
pub rules_to_delete: Vec<(String, Family, RuleHandle)>,
                    //   (table,  family, handle)

// 0.17.0
pub rules_to_delete: Vec<(String, Family, String, RuleHandle)>,
                    //   (table,  family, chain,  handle)
```

The chain name is now carried explicitly. The kernel rejects a
`NFT_MSG_DELRULE` with an empty `NFTA_RULE_CHAIN` (returns
`ENOENT`) even when `NFTA_RULE_HANDLE` pins the rule, contrary
to an earlier assumption in `apply.rs`. Plan 178's reconcile-
test surge surfaced this: `cfg.diff(&conn).apply(&conn)` on a
config with one rule removed used to fail at the delete step.

If you inspect `rules_to_delete` directly (rare — most callers
just call `.apply()` and let nlink dispatch the deletes), update
your destructure pattern:

```rust
// 0.16.0
for (table, family, handle) in &diff.rules_to_delete {
    println!("delete {table} handle={}", handle.0);
}

// 0.17.0
for (table, family, chain, handle) in &diff.rules_to_delete {
    println!("delete {table}/{chain} handle={}", handle.0);
}
```

The display output of `NftablesDiff::summary()` now also includes
the chain name (`- rule Inet filter/input (handle=42)` instead
of `- rule Inet filter (handle=42)`); update any log parsers
that scrape it.

## Behavior changes

### `Connection<P>` operations now time out after 30 s by default (Plan 171)

Every `Connection<P>` method that performs a netlink round-trip
(every getter, setter, dump, batch commit, …) now wraps the
underlying `recv` loop in a 30-second `tokio::time::timeout`.
Before 0.17.0, the timeout was opt-in via
`Connection::timeout(Duration)`, and the default was
`None` — a kernel that never responded would hang the call
indefinitely.

This change was driven by the 0.16 cut's CI evidence: a missing
`NLMSG_DONE` marker in `Connection<Nftables>::send_batch` caused
a 22-minute GHA hang that should have been a clean
`Error::Timeout`. Plan 172 then wrapped the same shape across
all 9 recv-loops in the lib so future kernel anomalies surface
as timeouts, not silent indefinite blocks.

The 30 s default matches the integration test suite's existing
explicit cap. Override per-Connection:

```rust
// Tighter for a known-fast operation
let conn = Connection::<Route>::new()?.timeout(Duration::from_secs(5))?;

// Loosen for a known-slow operation (large dump on a busy router)
let conn = Connection::<Route>::new()?.timeout(Duration::from_secs(120))?;

// Opt out entirely (rarely useful — surface the kernel hang via
// timeout rather than your own monitoring)
let conn = Connection::<Route>::new()?.no_timeout();
```

**Who breaks**: callers whose ops legitimately take > 30 s
without a kernel anomaly. The two patterns we know of in
practice: streaming dumps on multi-million-route BGP tables, and
ctnetlink dumps on hosts with > 1 M conntrack entries. Both
should bump the timeout explicitly via `.timeout(...)` or stream
the dump in chunks via the `dump_stream*` APIs (which apply the
timeout per-chunk, not over the whole dump).

If your code relied on the old "block forever" semantics for
debug/exploration (uncommon), wire `.no_timeout()` to restore
the prior behavior.

## What's new (no migration needed; additive)

### `nlink::lab::init_test_tracing()` + auto-wired in `require_root!()` (Plan 174)

Lab-feature only. Initializes a `tracing-subscriber` that honors
`RUST_LOG`, routes through libtest's capture. Auto-invoked by
`nlink::require_root!()` / `require_root_void!()` so integration
tests pick up the lib's `#[tracing::instrument]` spans without
per-test boilerplate. The `lab` feature now pulls in
`tracing-subscriber` as an optional dep.

### `From<AddressParseError>` + `From<RouteParseError>` for `nlink::Error` (Plan 173)

Removes `.map_err(|e| nlink::Error::InvalidMessage(e.to_string()))?`
ceremony in `NetworkConfig` caller chains. The two parse-error
types now implement `From<_>` so `?` propagates them cleanly:

```rust
// 0.16.0
let addr: Address = "10.0.0.1/24"
    .parse()
    .map_err(|e: AddressParseError| nlink::Error::InvalidMessage(e.to_string()))?;

// 0.17.0
let addr: Address = "10.0.0.1/24".parse()?;
```

### `Bottleneck::score: f64` normalized severity (Plan 169 Phase 3)

`Diagnostics::find_bottleneck()` now returns a `Bottleneck` with
a `score: f64` field (range 0.0..=1.0) computed from
`drop_rate × 0.6 + backlog_pressure × 0.3 + error_rate × 0.1`.
Backlog and error components are gated on the bottleneck's
`BottleneckType` (only counted when applicable), so a pure
hardware-error bottleneck scores on the error component alone.
Useful for sorting multiple bottlenecks by severity in a
controller dashboard.

### `scripts/cut-release.sh` (Plan 175)

One-shot release-cut orchestrator. Walks the full cut sequence
(pre-flight checks → CHANGELOG promotion → CI green-gate →
publish dry-runs → merge → tag → publish → GitHub release →
next-cycle branch) with confirmation prompts at every
irreversible step. Bakes in the three friction points from the
0.16 cut (`cargo publish -p nlink --dry-run` skip, CHANGELOG
promotion sed, GitHub-release-body length truncation). See
[CLAUDE.md "Publishing"](../../CLAUDE.md#publishing).

### `docs/release-validation-manual.md` (Plan 176)

Pre-cut hardware-validation checklist for the lib paths no CI
runner can exercise (XFRM IPsec offload, Devlink rate, Devlink
port function state, `net_shaper` caps). The cut script
(`scripts/cut-release.sh`) prompts to walk the checklist before
the irreversible publish step. Introduces a
`> ⚠ No CI coverage — manually validated YYYY-MM-DD against
> <hardware>` CHANGELOG annotation convention.

## Fixed

### `Connection::<Nftables>::send_batch` no longer hangs on a missing batch-end ACK (Plan 170)

The recv-loop didn't filter by `nlmsg_seq` and terminated on
the first per-op ACK rather than the BATCH_END's ACK. A
sequence-skew on the GHA `rust:bookworm` container surfaced
this as a 22-minute hang on the 0.16 cut CI. Fix: seq-filter
+ end-seq termination + `NLM_F_ACK` on `NFNL_MSG_BATCH_END`.
The canonical recv-loop shape is now documented in
[CLAUDE.md "Recv-loop shape"](../../CLAUDE.md#recv-loop-shape-canonical)
and audited across all 9 lib recv-loops by Plan 172.

### `NftablesConfig::diff` body-bytes false-positive (Plan 178)

Keyed rules were flagged as `to_replace` on every idempotent
re-diff, churning kernel state on every reapply for any caller
of the declarative-config reconcile loop. Three coordinated
fixes (the Register and `rules_to_delete` ones in **Breaking
changes** above; the third is internal):

The diff comparison now passes both sides through a
`normalize_tlv` walker that strips `NLA_F_NESTED` (`0x8000`) and
`NLA_F_NET_BYTEORDER` (`0x4000`) hint bits and sorts sibling
attributes by type at every depth. This closes the writer-vs-
kernel divergence on the NESTED bit (the writer set it on every
nest; the kernel doesn't on its outgoing serialization) and
intra-nest attribute ordering (writer in source order, kernel
in canonical numeric order).

## CI / observability

### Integration tests now log spans

Per Plan 174, the integration workflow sets
`RUST_LOG=nlink=debug,nlink::netlink::nftables=trace` and the
test harness initializes a libtest-friendly subscriber via
`nlink::lab::init_test_tracing()` (auto-invoked by
`require_root!()`). The next hidden hang takes 1 CI iteration
to diagnose instead of 3.

### `crates/nlink/tests/integration/IGNORED.md` catalogs every `#[ignore]`

13 currently-ignored tests, each with reason category + tracking
plan. A new CI gate (`scripts/audit-ignored-tests.sh`, wired
into `rust.yml`) fails on any new `#[ignore]` missing a catalog
entry, so the inventory doesn't drift.

### `nf_flow_table` + `nf_tables` modprobed explicitly

CI was relying on kernel auto-load; the explicit modprobe
documents intent + survives locked-down containers.

## What did NOT change

- The `nlink-macros` proc-macro derives (`GenlCommand`,
  `GenlAttribute`, `GenlEnum`, `GenlMessage`, `NetlinkAttrs`,
  `#[genl_family]`) — no API change.
- The 8 GENL families from 0.16 (WireGuard, MACsec, MPTCP,
  Ethtool, nl80211, Devlink, DPLL, NetShaper) — `Connection<P>`
  APIs unchanged; the Plan 172 timeout-wrap is invisible.
- `NetworkConfig` (links/addresses/routes/qdiscs) — no API
  change; the new `From` impls for parse errors are additive.
- Every typed TC config + `Rate` / `Bytes` / `Percent` /
  `TcHandle` / `FilterPriority` newtypes — no API change.
- Error predicates (`is_busy()`, `is_not_found()`, …) — no API
  change.
- `Connection<P>` constructors + `enable_strict_checking` /
  `set_ext_ack` toggles — no API change.
